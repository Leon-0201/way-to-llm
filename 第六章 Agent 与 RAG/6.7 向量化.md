# 6.7 向量化

> **核心理解**：向量化是 RAG 系统的核心环节，将文本转换为向量表示，使得语义相似度计算成为可能。向量化质量直接决定检索效果的上限。选择合适的嵌入模型、理解向量维度与相似度计算方法，是构建高效 RAG 系统的关键。

---

## 1. 嵌入模型选型

### 1.1 嵌入模型工作原理

**什么是嵌入模型**：
嵌入模型（Embedding Model）将文本映射为固定维度的向量。核心思想是：语义相似的文本，在向量空间中距离更近。

**工作流程**：
```
文本 → 嵌入模型 → 向量（如 [0.1, -0.5, 0.8, ...]）

示例：
"猫" → [0.2, -0.3, 0.9, ...]
"狗" → [0.3, -0.2, 0.8, ...]  # 与"猫"相似
"汽车" → [-0.8, 0.6, -0.1, ...]  # 与"猫"不相似
```

**向量相似度计算**：
最常用的是余弦相似度（Cosine Similarity），范围从 -1 到 1：
- 1：完全相同
- 0：无关
- -1：完全相反

## 2. 多模态向量化

### 2.1 多模态嵌入模型

**什么是多模态向量化**：
传统向量化是文本→文本向量，而多模态向量化可以将文本和图片映射到同一向量空间，实现跨模态检索。

**示例**：
```
"一只猫的照片" → [0.2, -0.3, 0.9, ...]
文本"猫"      → [0.2, -0.3, 0.8, ...]

两者向量相似，可以跨模态检索
```

## 3. 向量维度与相似度计算

### 3.1 向量维度选择

**维度影响**：

**高维度（1024-3072）**：
- 优点：表达能力强、语义细节丰富、检索精度高
- 缺点：存储空间大、计算慢、容易过拟合

**低维度（256-768）**：
- 优点：存储效率高、计算速度快、泛化能力强
- 缺点：表达能力有限、可能丢失细节

**推荐维度**：
- **通用 RAG**：768-1024 维（平衡性能和效率）
- **高精度检索**：1024-3072 维（追求最佳性能）
- **大规模部署**：256-512 维（存储和速度优先）
- **实时检索**：512-768 维（低延迟要求）

---

### 3.2 维度优化技术

**技术 1：降维（PCA）**
使用主成分分析（PCA）将高维向量降维，可减少 50% 存储空间，提升 40% 检索速度，精度损失<2%。

**适用场景**：大规模部署（百万级向量）、存储成本敏感、实时性要求高

---

**技术 2：量化（Quantization）**
将 float32 转为 int8，可减少 75% 存储空间，精度损失仅 1%。

**适用场景**：超大规模部署、移动端部署、存储成本极高

---

### 3.3 相似度计算方法

**方法 1：余弦相似度（Cosine Similarity）**
- **公式**：cos(θ) = (A·B) / (||A|| × ||B||)
- **范围**：[-1, 1]
- **特点**：最常用，效果好；不受向量长度影响；向量数据库默认支持
- **适用场景**：大多数 RAG 场景（推荐）、文本检索、语义匹配

---

**方法 2：点积相似度（Dot Product）**
- **公式**：A·B = Σ(Ai × Bi)
- **范围**：(-∞, +∞)
- **特点**：计算最快；向量归一化后等价余弦相似度
- **适用场景**：需要极致性能、向量已归一化、大规模检索

---

**方法 3：欧氏距离（Euclidean Distance）**
- **公式**：d(A,B) = √Σ(Ai - Bi)²
- **范围**：[0, +∞)
- **特点**：直观（几何距离）；高维空间效果差
- **适用场景**：聚类任务、空间分布分析

---

### 3.4 相似度计算优化

**优化 1：向量归一化**
归一化后，点积计算等价于余弦相似度，计算速度快 2-3 倍。

**优化 2：批量计算**
一次计算所有相似度，比循环计算快 200 倍。

**优化 3：近似最近邻（ANN）**
使用 HNSW、IVF、LSH 等算法，将 O(n) 复杂度降为 O(log n)。
- **HNSW**：精度高（>95%），适用 Milvus、Chroma
- **IVF**：速度快，适用 FAISS
- **LSH**：极快，精度低

## 4. 向量化实战

### 4.1 批量向量化

**场景：处理 10 万文档**

**最佳实践**：
1. 过滤空文档
2. 排序（相似长度放一起，提升效率）
3. 批量处理（batch_size=64）
4. 多进程加速（num_workers=4）
5. 向量归一化

**性能对比**：
- 逐条处理：约 1000 秒
- 批量处理：约 100 秒
- 提升：10 倍

---

### 4.2 向量化性能优化

**优化 1：模型量化**
使用 int8 量化或 ONNX 加速，推理速度提升 2-3 倍，显存占用减少 50%，精度损失<1%。

**优化 2：模型蒸馏**
教师模型（large）→ 学生模型（base），速度提升 3 倍，存储减少 50%，精度保持 95%+。

**优化 3：缓存机制**
缓存常见查询的向量结果，避免重复计算，可提升 10-100 倍。

---

### 4.3 向量化质量评估

**评估方法 1：语义相似度测试**
测试同义词、反义词、类比推理等，评估语义理解能力。

**评估方法 2：检索测试**
使用标准测试集，计算检索命中率（Hit Rate）、MRR、NDCG 等指标。

## 6. 面试高频问答

### 6.1 基础概念类

**Q1: 什么是向量化？为什么 RAG 需要向量化？**

**参考答案**：
向量化是将文本转换为固定维度向量表示的过程。RAG 需要向量化的原因：

1. **语义相似度计算**
   - 传统关键词匹配只能字面匹配
   - 向量相似度可以语义匹配
   - 示例：查询"如何学习编程"可以匹配到"编程入门教程"

2. **高效检索**
   - 向量空间中的距离等于语义距离
   - 可以用数学方法快速检索
   - 支持近似最近邻（ANN）加速

3. **跨语言检索**
   - 不同语言映射到同一向量空间
   - 英文查询可以检索中文文档

4. **多模态融合**
   - 文本、图像、音频统一为向量
   - 支持跨模态检索

**性能对比**：关键词检索准确率 50-60%，向量检索准确率 70-85%。

---

**Q2: 如何选择嵌入模型？考虑哪些因素？**

**参考答案**：
选型决策框架：

1. **预算和部署方式**

   - 平衡方案：BGE-base-zh, text-embedding-3-small

- 模型蒸馏：large→base，速度提升 3 倍，精度保持 95%+

**Q5: 余弦相似度和点积相似度有什么区别？**

**参考答案**：

**数学定义**：
- 余弦相似度：cos(θ) = (A·B) / (||A|| × ||B||)，范围 [-1, 1]
- 点积：A·B = Σ(Ai × Bi)，范围 (-∞, +∞)

**核心区别**：

1. **归一化**
   - 余弦相似度：自动归一化，不受向量长度影响
   - 点积：未归一化，受向量长度影响

2. **计算速度**
   - 余弦相似度：需计算模长，较慢（10ms）
   - 点积：直接相乘求和，最快（3ms）

3. **等价关系**
   - 向量归一化后：点积 = 余弦相似度

**实际应用**：
- 开发阶段：用余弦相似度（方便）
- 生产环境：向量归一化 + 点积（快速）
- 大规模部署：归一化 + 点积 + ANN 索引

**Q6: 如何处理多语言和跨语言检索？**

**参考答案**：

**解决方案**：

1. **多语言嵌入模型**
   - E5-large-v2：支持 100+ 语言
   - text-embedding-3-large：支持 100+ 语言
   - BGE-m3：支持 100+ 语言

2. **跨语言检索实现**

   - 查询向量化（同一模型）
   - 计算相似度

3. **语言检测 + 翻译**

   - 翻译为统一语言（如英文）
   - 向量化 + 检索
   - 返回结果（可翻译回原语言）

**模型对比**：
- E5-large-v2：100+ 语言，中文 64.8，多语言 64.8
- text-embedding-3-large：100+ 语言，中文 67.8，多语言 68.5
- BGE-m3：100+ 语言，中文 66.2，多语言 67.5

**Q8: 如何处理长文本的向量化？**

**参考答案**：

**挑战**：模型有长度限制（如 512 tokens），长文本超出限制无法直接处理。

**解决方案**：

1. **分段向量化 + 平均池化**
   - 将长文本分段
   - 分别向量化
   - 平均池化得到最终向量
   - 优点：简单直接，保留整体语义

2. **分段向量化 + 加权池化**
   - 考虑位置重要性（首尾更重要）
   - 加权平均
   - 优点：首尾信息保留好

3. **层次化向量化**
   - 段落级向量化
   - 文档级池化
   - 优点：保留层次结构

4. **使用长文本模型**
   - BGE-m3：8192 tokens
   - text-embedding-3-large：8191 tokens
   - E5-mistral-7b：8192 tokens
   - 优点：无需分段，整体语义完整

**方案选择**：
- 文本长度 < 512 tokens：直接用标准模型
- 文本长度 512-2000 tokens：分段 + 平均池化
- 文本长度 2000-8000 tokens：长文本模型（BGE-m3）
- 文本长度 > 8000 tokens：层次化向量化 + 长文本模型

### 6.4 实战应用类

**Q9: 向量化服务上线后效果不好，如何排查？**

**参考答案**：

**排查框架（由下到上）**：

1. **模型层面检查**
   - 模型选择不当（语言不匹配）
   - 模型加载错误
   - 模型版本过旧
   - 解决：换合适的模型，更新到最新版

2. **向量化质量检查**
   - 向量归一化问题
   - 维度不匹配
   - 数据类型错误
   - 解决：添加归一化，统一维度

3. **相似度计算检查**
   - 相似度公式错误
   - 未归一化用点积
   - 距离度量不匹配
   - 解决：修正公式，统一度量

4. **检索效果检查**
   - 构建测试集（100 个问题）
   - 查看检索结果
   - 分析失败案例
   - 解决：调整 Top-K，调整阈值

5. **性能问题检查**
   - 向量化速度慢
   - 内存占用高
   - GPU 利用率低
   - 解决：批量处理，模型量化

**排查流程**：
1. 语义相似度测试：相似对>0.7，不相似对<0.3
2. 检索测试：100 个典型问题，命中率>80%
3. 性能测试：P99<100ms
4. A/B 测试：对比不同模型和参数

**关键指标**：
- 质量指标：语义相似度>0.75，检索命中率>80%，NDCG@10>0.7
- 性能指标：P50<50ms，P99<100ms，吞吐>1000 文档/秒

**Q11: 向量化过程中遇到过哪些坑？**

**参考答案**：

**常见陷阱**：

1. **未归一化**
   - 问题：余弦相似度结果异常
   - 解决：添加 normalize_embeddings=True

2. **模型语言不匹配**
   - 问题：中文用英文模型
   - 解决：换 BGE-large-zh

3. **维度不一致**
   - 问题：查询和文档维度不同
   - 解决：统一模型和参数

4. **批量大小不当**
   - 问题：batch_size 太大 OOM
   - 解决：调整为 32-64

5. **缓存穿透**
   - 问题：大量未命中查询打爆数据库
   - 解决：布隆过滤器过滤

6. **模型更新不一致**
   - 问题：线上模型和测试模型版本不同
   - 解决：模型版本管理

7. **相似度阈值设置不当**
   - 问题：阈值过高漏检，过低误检
   - 解决：基于测试集调优

8. **多语言混用**
   - 问题：不同语言用不同模型
   - 解决：统一用多语言模型

9. **长文本直接截断**
   - 问题：丢失关键信息
   - 解决：分段向量化 + 池化

10. **忽略冷启动**
    - 问题：新文档未向量化
    - 解决：增量向量化流程

**实战建议**：
- 建立标准化测试集
- 自动化评估流程
- 持续监控线上效果
- 定期 A/B 测试
