# 6.4 上下文管理

> **核心理解**：上下文管理是 Agent 的"记忆中枢"，决定了模型能记住多少信息、如何组织信息。优秀的上下文管理策略能在有限的上下文窗口内，最大化信息利用率，提升模型表现。

---

## 1. 三层记忆机制

### 1.1 为什么需要分层记忆

**问题背景**：
```
1. 上下文窗口限制
   - GPT-4: 128K tokens
   - Claude: 200K tokens
   - 但实际应用中仍可能不够用

2. 成本考虑
   - 长上下文 = 更高的 token 成本
   - 更长的响应延迟

3. 信息过载
   - 过多信息会干扰模型判断
   - "注意力稀释"问题

4. 信息类型不同
   - 临时对话 vs 持久事实
   - 通用知识 vs 个性化特征
   - 需要不同的管理策略
```

**三层记忆架构**：
```
┌─────────────────────────────────────┐
│     短期记忆 (Short-term Memory)     │
│  - 当前对话上下文                    │
│  - 快速访问、易变                    │
│  - 容量有限                          │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│     长期记忆 (Long-term Memory)      │
│  - 持久化存储的事实和经历            │
│  - 需要时检索                        │
│  - 容量理论上无限                    │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│   Profile 记忆 (Profile Memory)      │
│  - Agent 性格设定                    │
│  - 用户特征画像                      │
│  - 相对稳定、不常变更                │
└─────────────────────────────────────┘
```

---

## 2. 短期记忆（Short-term Memory）

### 2.1 定义与特点

**定义**：当前对话窗口内的信息，直接放入 prompt 中。

**特点**：
```
- 访问速度：极快（直接在上下文中）
- 容量：受上下文窗口限制（通常 4K-8K tokens）
- 持久性：会话结束即消失
- 成本：每次调用都需要携带
- 更新频率：每轮对话都更新
```

**适用场景**：
```
- 当前对话的轮次记录
- 临时的任务状态
- 即时的用户反馈
- 短期内的指代消解
```

---

### 2.2 实现策略

#### 2.2.1 策略 1：滑动窗口（Sliding Window）

**原理**：保留最近的 N 轮对话，超出则丢弃最早的消息。

**优点**：
- ✅ 实现简单
- ✅ 保证最近的对话始终可用
- ✅ 性能稳定

**缺点**：
- ❌ 可能丢失重要但较早的信息
- ❌ 不区分信息重要性
- ❌ "一刀切"的丢弃策略

---

#### 2.2.2 策略 2：重要性评分（Importance-based）

**原理**：为每条消息评分，保留重要性高的消息。

**优点**：
- ✅ 智能保留重要信息
- ✅ 适应不同对话场景
- ✅ 更好的长期对话质量

**缺点**：
- ❌ 需要额外的 LLM 调用（成本）
- ❌ 实现复杂度较高
- ❌ 重要性评估可能不准确

---

#### 2.2.3 策略 3：混合策略（推荐）

**原理**：结合滑动窗口和重要性评分，平衡性能和成本。

**优点**：
- ✅ 平衡性能和成本
- ✅ 关键信息不丢失
- ✅ 实现相对简单

**缺点**：
- ❌ 需要手动标记关键信息（或自动识别）

---

### 2.3 短期记忆优化技巧

#### 2.3.1 技巧 1：系统提示词优化

**核心思路**：在系统提示词中预先注入用户信息、偏好和规则，让模型始终"记住"这些背景。

**关键内容**：
- **用户特征**：姓名、职业、专业水平等基本信息
- **用户偏好**：回答风格、内容形式、交互方式
- **当前任务**：本次对话的核心目标
- **行为规则**：3-5 条具体的对话准则

**示例**：
```
【用户特征】软件工程师，3 年经验，喜欢简洁回答
【用户偏好】代码优先，实例驱动，不需要基础概念解释
【当前任务】学习 Python 异步编程
【对话规则】
1. 提供可运行的代码示例
2. 避免冗长理论，先给方案再解释
3. 承认不确定，不编造答案
```

**优点**：无需额外成本，信息始终可用，一致性好  
**缺点**：占用固定上下文空间，更新需重新生成  
**适用场景**：用户特征稳定、需要保持一致风格的对话

---

#### 2.3.2 技巧 2：对话摘要注入

**核心思路**：定期用 LLM 生成对话摘要，用摘要替代原始历史，大幅压缩 token。

**关键步骤**：
1. **设置间隔**：每 10-20 轮生成一次摘要
2. **生成摘要**：包含用户问题、已解决内容、待办事项
3. **清空历史**：只保留摘要，释放上下文空间
4. **混合使用**：摘要 + 最近 3-5 轮对话

**示例格式**：
```
【对话摘要】
用户目标：学习 Python 异步编程
已讨论：async/await 概念、gather 用法、协程与任务区别
待解决：实际项目应用、完整示例代码
用户偏好：实例驱动学习

【最近对话】
（保留最近 3-5 轮）
```

**优点**：token 压缩 10 倍以上，保留关键信息，支持超长对话  
**缺点**：需定期调用 LLM，可能丢失细节  
**适用场景**：长对话（超过 20 轮）、成本敏感应用

## 3. 长期记忆（Long-term Memory）

### 3.1 定义与特点

**定义**：持久化存储的事实、经历、知识等信息，需要时检索使用。

**特点**：
- 访问速度：较慢（需要检索）
- 容量：理论上无限
- 持久性：永久保存
- 成本：存储成本低，检索有成本
- 更新频率：中等（按需添加/更新）

**存储内容**：
- **事实性知识**：用户明确告知的信息、对话中确认的事实、领域知识
- **经历记忆**：过去的对话主题、已完成的任务、解决的问题
- **程序性知识**：用户的操作习惯、常用工作流程、偏好工具

---

### 3.2 基于向量数据库的实现

#### 3.2.1 核心流程

**技术选型**：
- **向量数据库**：Chroma（轻量）、Milvus（企业级）、Pinecone（托管）
- **嵌入模型**：text-embedding-3-small、bge-large-zh（中文优化）
- **相似度度量**：余弦相似度（cosine）、欧氏距离（L2）

**工作流程**：

1. **添加记忆**
   - 将文本内容通过嵌入模型转为向量
   - 生成唯一 ID 和元数据（类型、用户 ID、时间戳）
   - 存储到向量数据库

2. **检索记忆**
   - 将查询文本转为向量
   - 按相似度搜索 Top-K 条记忆
   - 可按用户 ID、记忆类型过滤
   - 更新访问计数

3. **删除记忆**
   - 按 ID 删除单条记忆
   - 支持批量清理过期记忆

**记忆类型**：
- `preference`：用户偏好（如"喜欢喝拿铁"）
- `fact`：客观事实（如"在北京工作"）
- `progress`：学习进展（如"已掌握 Python 基础"）
- `history`：历史互动（如"上周讨论过异步编程"）

**示例**：
```
添加记忆：
- "用户喜欢喝拿铁咖啡，不喜欢加糖"（类型：preference）
- "用户在北京工作，从事软件工程师职业"（类型：fact）
- "用户正在学习 Python，已掌握基础语法"（类型：progress）

检索记忆：
查询："用户的饮食偏好"
结果：
  - 用户喜欢喝拿铁咖啡，不喜欢加糖（距离：0.12）
  - 用户对花生过敏（距离：0.25）
```

---

### 3.3 记忆更新与遗忘

#### 3.3.1 记忆更新策略

**问题**：避免存储重复或过时的信息

**更新流程**：
1. **检索相似记忆**：用新信息检索最相似的已有记忆
2. **相似度判断**：
   - 距离 < 0.1：非常相似，可能需要更新
   - 距离 > 0.3：不同信息，直接添加
3. **LLM 判断**：让 LLM 判断新信息是否更准确
4. **执行更新**：删除旧记忆，添加新记忆

**示例**：
```
首次添加："用户喜欢喝咖啡"
→ 输出：已添加新记忆

更新信息："用户喜欢喝拿铁咖啡，不喜欢加糖"
→ 检索到相似记忆（距离 0.08）
→ LLM 判断：新信息更具体，需要更新
→ 输出：已更新记忆
```

#### 3.3.2 记忆遗忘策略

**核心思路**：基于艾宾浩斯遗忘曲线 + 访问频率

**遗忘概率计算**：
- **基础衰减**：随时间指数衰减（半衰期 30 天）
- **频率增强**：访问次数越多，遗忘越慢
- **类型权重**：
  - 偏好：0.8（不容易遗忘）
  - 事实：0.5（中等）
  - 进展：0.3（容易过时）
  - 历史：0.2（快速遗忘）

**清理机制**：
- 定期执行（如每周一次）
- 计算每条记忆的召回概率
- 低于阈值（如 0.1）则删除

**示例**：
```
记忆 A：30 天前添加，访问 0 次，类型 history
→ 召回概率：0.08 → 删除

记忆 B：30 天前添加，访问 5 次，类型 preference
→ 召回概率：0.65 → 保留
```

---

### 3.4 长期记忆使用模式

#### 3.4.1 对话中自动检索

**工作流程**：
1. 用户输入 → 检索相关长期记忆（Top-5）
2. 将记忆注入上下文（"相关记忆：..."）
3. 调用 LLM 生成回复
4. 更新短期记忆
5. 从对话中提取新记忆（可选）

**上下文构建**：
```
【系统提示词】你是一个智能助手

【相关记忆】
- 用户喜欢喝拿铁咖啡，不喜欢加糖
- 用户在北京工作，从事软件工程师职业
- 用户正在学习 Python 异步编程

【对话历史】
（最近 10 轮对话）

【当前问题】
用户：我之前说过我在哪里工作吗？
```

**效果**：
- LLM 能"记住"用户的历史信息
- 无需在短期记忆中携带所有内容
- 支持超长对话和跨会话记忆

## 4. Profile 记忆（Profile Memory）

### 4.1 定义与特点

**定义**：存储 Agent 的性格设定、用户特征等相对稳定的信息。

**特点**：
```
- 访问速度：快（通常预加载到上下文）
- 容量：中等（通常 1K-3K tokens）
- 持久性：长期保存，手动更新
- 成本：一次性加载，复用多次
- 更新频率：低（按需更新）
```

**包含内容**：
```
1. Agent 侧（Agent Profile）
   - 性格设定（活泼、严肃、幽默等）
   - 角色定位（助手、导师、朋友等）
   - 行为准则（不做的事情、必须做的事情）
   - 语言风格（正式、随意、专业等）
   - 知识边界（专业领域、不擅长的领域）

2. 用户侧（User Profile）
   - 基本信息（姓名、职业、地区等）
   - 偏好（回答风格、内容深度等）
   - 目标（学习目标、工作目标等）
   - 禁忌（不喜欢的话题、敏感点等）
   - 历史互动模式
```

---

### 4.2 Agent Profile（性格设定）

**结构示例**：

```markdown
# Agent Profile

## 基本信息
- 名称：小智
- 角色：AI 学习助手
- 版本：1.0

## 性格特征
- 热情友好：用温暖的语言鼓励用户
- 耐心细致：不厌其烦地解答问题
- 幽默风趣：适当使用幽默缓解压力
- 积极向上：传递正能量

## 行为准则
### 必须做
- 提供准确、可验证的信息
- 鼓励用户独立思考和实践
- 承认不知道的事情
- 尊重用户的观点和选择

### 不能做
- 不提供未经证实的医疗/法律建议
- 不参与政治敏感话题讨论
- 不代替用户做决策
- 不使用贬低或歧视性语言

## 语言风格
- 语气：友好、鼓励
- 正式程度：中等（介于正式和随意之间）
- 专业术语：根据用户水平调整
- 表情符号：适度使用（每轮 1-2 个）

## 知识边界
### 擅长领域
- 编程语言（Python、JavaScript）
- 人工智能基础概念
- 学习方法和技巧

### 不擅长领域
- 深度专业技术问题
- 实时新闻和事件
- 个人情感咨询
```

### 4.3 User Profile（用户特征）

**结构示例**：

```markdown
# User Profile

## 基本信息
- 用户 ID：user_123
- 姓名：张三
- 职业：软件工程师
- 地区：北京
- 语言：中文（简体）

## 偏好设置
### 回答风格
- 简洁直接：不喜欢冗长解释
- 代码优先：希望看到可运行的代码
- 实例驱动：通过例子理解概念

### 内容深度
- 编程水平：中级（3 年经验）
- 不需要基础概念解释
- 可以讨论进阶话题

### 交互方式
- 喜欢主动提问
- 希望得到引导而非答案
- 接受建设性批评

## 学习目标
1. 掌握 Python 高级特性
2. 学习机器学习基础
3. 提升系统设计能力

## 禁忌与敏感点
- 不喜欢被说教
- 避免过于基础的解释
- 不接受"这个很简单"之类的表述

## 历史互动统计
- 总对话轮次：1250
- 主要话题：Python 编程、机器学习
- 平均满意度：4.5/5
- 活跃时间段：工作日晚上 8-11 点
```

### 4.4 Profile 管理最佳实践

#### 4.4.1 Profile 分离存储

**核心思路**：将 Agent Profile 和 User Profile 分开管理，通过 ProfileManager 统一加载和组合。

**架构设计**：
- **Agent Profile**：全局唯一，所有用户共享同一个人设
- **User Profile**：每个用户独立，存储个性化信息
- **ProfileManager**：负责加载、组合多个 Profile 为系统提示词

**工作流程**：
1. 加载 Agent Profile（一次加载，全局复用）
2. 根据用户 ID 加载对应的 User Profile
3. 将两个 Profile 组合为完整的系统提示词
4. 在对话开始时注入到上下文

**组合方式**：
```
【系统提示词】= Agent Profile + User Profile

Agent Profile 部分：
- 你是名为 {名称} 的 {角色}
- 性格特征：{性格列表}
- 行为准则：{必须做/不能做}
- 语言风格：{语气、正式程度}

User Profile 部分：
- 用户信息：{姓名、职业、地区}
- 用户偏好：{回答风格、内容深度}
- 学习目标：{目标列表}
```

**优点**：
- ✅ 职责清晰，易于维护
- ✅ 支持多用户场景
- ✅ Profile 可独立更新

---

#### 4.4.2 Profile 动态更新

**核心思路**：使用 LLM 从对话中自动提取用户信息，动态更新 User Profile。

**提取流程**：
1. **收集对话**：截取最近 20 轮对话
2. **LLM 提取**：让 LLM 识别并结构化用户信息
3. **信息合并**：将新信息合并到现有 Profile
4. **持久保存**：更新 Profile 文件

**提取内容**：
- **基本信息**：姓名、职业、地区等明确提及的信息
- **偏好设置**：对回答风格、内容形式的要求
- **学习目标**：用户表达的学习意愿或目标
- **禁忌敏感**：用户不喜欢的话题或表述方式

**示例**：
```
用户说："我是软件工程师，想学习 Python 数据分析"

LLM 提取结果：
{
  "基本信息": {"职业": "软件工程师"},
  "偏好": [],
  "目标": ["学习 Python 数据分析"],
  "禁忌": []
}

合并到 Profile 后：
- 职业：软件工程师（新增）
- 学习目标：Python 数据分析（新增）
```

**更新策略**：
- **定期执行**：每 10-20 轮对话分析一次
- **增量更新**：只添加新信息，不覆盖旧信息
- **冲突处理**：新信息与旧信息冲突时，优先保留新信息
- **人工审核**：重要更新可设置人工确认环节

**优点**：
- ✅ 自动积累用户信息，无需手动录入
- ✅ Profile 随使用逐渐完善
- ✅ 提升个性化体验

**注意事项**：
- ⚠️ 避免过度提取（只提取明确信息，不推测）
- ⚠️ 保护用户隐私（敏感信息需加密存储）
- ⚠️ 提供删除接口（用户可清除 Profile）

## 5. 三层记忆的协同工作

### 5.1 完整架构

#### 5.1.1 三层记忆协同流程

**核心架构**：整合短期记忆、Profile 记忆、长期记忆，形成完整的对话系统。

**组件构成**：
- **短期记忆模块**：SlidingWindowMemory，管理当前对话历史
- **长期记忆模块**：LongTermMemory，基于向量数据库的持久化存储
- **Profile 记忆模块**：ProfileManager，加载 Agent 和 User Profile
- **LLM 模块**：OpenAI GPT-4，生成对话回复

**完整对话流程**：

**第 1 步：加载 Profile**
- 从配置文件加载 Agent Profile（性格、角色、行为准则）
- 根据用户 ID 加载 User Profile（用户特征、偏好）
- 组合为系统提示词

**第 2 步：检索长期记忆**
- 将用户输入转为向量
- 检索 Top-5 相关记忆
- 按类型过滤（偏好、事实、进展等）

**第 3 步：构建上下文**
```
【系统提示词】
（来自 Profile 记忆）

【相关记忆】
- 记忆 1 内容（类型：preference）
- 记忆 2 内容（类型：fact）

【对话历史】
（来自短期记忆，最近 10 轮）

【当前问题】
用户输入
```

**第 4 步：调用 LLM**
- 发送完整上下文到 GPT-4
- 获取模型回复

**第 5 步：更新短期记忆**
- 将用户输入添加到对话历史
- 将模型回复添加到对话历史
- 如超出 token 限制，修剪最早的消息

**第 6 步：提取新记忆**
- 使用 LLM 分析对话，识别需要长期存储的信息
- 提取规则：
  - 只提取事实性信息（偏好、事实、目标）
  - 不提取临时性对话内容
  - 返回结构化 JSON 格式
- 将新记忆添加到长期记忆库

**配置参数**：
```
短期记忆：
- max_tokens: 4000（对话历史最大 token 数）

Profile 记忆：
- agent_profile_path: "agent_profile.yaml"

长期记忆：
- collection_name: "user_memory"
- persist_directory: "./memory_db"
```

**对话示例**：
```
第 1 轮：
用户："你好，我叫张三"
→ 提取记忆："用户姓名是张三"（类型：fact）

第 2 轮：

→ 提取记忆："用户在北京工作"（类型：fact）

第 4 轮：

- ✅ 三层记忆各司其职，信息组织清晰

**Q1: 什么是上下文管理？为什么需要分层记忆？**

**参考答案**：
```
上下文管理是 Agent 的"记忆中枢"，决定了模型能记住多少信息、如何组织信息。

为什么需要分层：

1. 上下文窗口限制
   - 即使是 128K tokens 也可能不够用
   - 长上下文 = 更高的 token 成本 + 更长的延迟

2. 信息类型不同
   - 临时对话 vs 持久事实
   - 通用知识 vs 个性化特征
   - 需要不同的管理策略

3. 注意力稀释问题
   - 过多信息会干扰模型判断
   - 分层可以聚焦关键信息

三层记忆架构：
- 短期记忆：当前对话，快速访问，容量有限
- Profile 记忆：性格设定和用户特征，稳定不变
- 长期记忆：持久化存储，需要时检索，容量无限

典型应用：个性化助手、长期对话 Agent、多用户系统。
```

### 7.2 技术实践类

**Q4: 如何实现短期记忆的滑动窗口策略？**

**参考答案**：
```
实现思路：

1. 基本原理
   - 保留最近的 N 轮对话
   - 超出限制则丢弃最早的消息
   - 类似队列的 FIFO 机制

2. 关键参数
   - max_tokens：最大 token 数（如 4000）
   - max_messages：最大消息数（如 20 轮）
   - 保留 system message（系统提示词）

3. 实现步骤
   - 添加新消息到列表末尾
   - 计算当前 token 总数
   - 如超出限制，移除最早的用户消息
   - 重复直到满足限制

4. 优化措施
   - 关键信息永久保留（标记 is_critical）
   - 重要性评分辅助决策
   - 定期生成摘要压缩历史

优缺点：
- ✅ 实现简单，性能稳定
- ❌ 可能丢失重要但较早的信息
- ❌ 不区分信息重要性

适用场景：短对话、成本敏感的应用。
```

**Q6: 如何实现记忆的更新和遗忘机制？**

**参考答案**：
```
记忆更新策略：

1. 检测重复
   - 用新信息检索相似记忆
   - 距离 < 0.1：非常相似，可能需要更新
   - 距离 > 0.3：不同信息，直接添加

2. LLM 判断
   - 让 LLM 比较新旧信息
   - 判断新信息是否更准确
   - 避免盲目更新

3. 执行更新
   - 删除旧记忆
   - 添加新记忆
   - 记录更新日志

记忆遗忘策略（基于艾宾浩斯曲线）：

1. 基础衰减
   - 随时间指数衰减
   - 半衰期设置（如 30 天）

2. 频率增强
   - 访问次数越多，遗忘越慢
   - 每次访问提升 recall 概率

3. 类型权重
   - 偏好：0.8（不容易遗忘）
   - 事实：0.5（中等）
   - 进展：0.3（容易过时）
   - 历史：0.2（快速遗忘）

4. 定期清理
   - 每周执行一次
   - 计算每条记忆的召回概率
   - 低于阈值（如 0.1）则删除

效果：
- 避免记忆无限增长
- 保留重要信息
- 自动清理过期内容
```

**Q9: 如何优化长对话场景下的上下文管理？**

**参考答案**：
```
优化策略组合：

1. 对话摘要注入
   - 每 10-20 轮生成一次摘要
   - 用摘要替代原始历史
   - token 压缩 10 倍以上
   - 摘要内容：用户问题、已解决、待办

2. 重要性评分
   - LLM 为每条消息评分（1-10 分）
   - 保留重要性高的消息
   - 优先丢弃闲聊内容
   - 关键信息永久保留

3. 混合策略（推荐）
   - 滑动窗口 + 重要性评分
   - 关键信息单独存储（critical_slots）
   - 普通消息进入滑动窗口
   - 平衡性能和成本

4. 分层摘要
   - 小摘要：每 10 轮，保留最近 100 轮细节
   - 大摘要：每 50 轮，保留整体脉络
   - 主题摘要：每个话题单独摘要

5. 长期记忆辅助
   - 将早期对话存入长期记忆
   - 需要时检索而非全部携带
   - 减少短期记忆压力

实际效果对比：
- 无优化：50 轮对话 = 10K tokens
- 摘要优化：50 轮对话 = 2K tokens
- 成本降低：80%
```

**Q11: 上下文管理在实际应用中遇到过哪些坑？**

**参考答案**：
```
常见陷阱：

1. 信息过载
   问题：携带太多历史信息，模型表现反而下降
   解决：设置合理的 token 限制，使用摘要压缩

2. 记忆混淆
   问题：多用户场景下记忆串台
   解决：严格的 user_id 隔离，检索时加上过滤

3. 过度提取
   问题：从对话中提取太多琐碎信息
   解决：只提取明确的事实，不推测

4. 遗忘过快
   问题：重要信息被过早清理
   解决：调整遗忘曲线参数，关键信息永久保留

5. Profile 僵化
   问题：Profile 长期不更新，与实际情况脱节
   解决：定期分析对话，动态更新 Profile

6. 检索不准
   问题：长期记忆检索结果不相关
   解决：优化嵌入模型，调整 top_k，添加关键词过滤

7. 成本失控
   问题：长期记忆检索频繁，API 成本高
   解决：缓存检索结果，设置调用频率限制

8. 隐私泄露
   问题：敏感信息未加密存储
   解决：敏感字段加密，提供数据删除接口

实战建议：
- 上线前充分测试边界情况
- 设置完善的监控和告警
- 定期审查记忆质量
- 建立用户反馈机制
```

   - 长期记忆：提供个性化

2. 协同工作是关键
   - 三层记忆各司其职
   - 信息在不同层之间流转
   - 形成完整的记忆系统

3. 优化策略不可少
   - 摘要压缩、重要性评分
   - 记忆更新、遗忘机制
   - 平衡性能和成本

4. 安全隐私是底线
   - 用户数据隔离
   - 敏感信息加密
   - 提供删除接口
```
