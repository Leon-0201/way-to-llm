# 6.6 RAG 数据处理

> **核心理解**：数据处理是 RAG 系统的基石，决定了检索质量的上限。高质量的数据处理流程包括：数据收集与清洗、文档切片策略、元数据管理三个核心环节。记住：Garbage In, Garbage Out（垃圾进，垃圾出），数据质量比算法复杂度更重要。

---

## 1. 数据收集与清洗

### 1.1 数据源类型

**常见数据源**：
```
1. 文档文件
   - PDF：技术文档、论文、报告
   - Word：制度文件、合同、方案
   - Excel：数据表格、清单
   - PPT：演示文稿、培训材料

2. 文本内容
   - Markdown：技术博客、文档
   - HTML：网页内容
   - TXT：纯文本记录

3. 数据库
   - 关系型数据库：MySQL、PostgreSQL
   - NoSQL：MongoDB、Elasticsearch
   - API 接口：第三方服务

4. 即时通讯
   - 钉钉/企微聊天记录
   - Slack 频道讨论
   - 邮件往来
```

---

### 1.2 数据收集策略

#### 策略 1：批量导入

**适用场景**：
```
- 系统初始化阶段
- 定期更新知识库
- 历史数据迁移
```

**实施步骤**：
```
第 1 步：确定数据范围
- 文档类型和格式
- 时间范围（如近 3 年）
- 部门和业务线

第 2 步：收集文档
- 从文件服务器下载
- 从 Git 仓库拉取
- 从数据库导出

第 3 步：建立清单
- 文档名称、来源、大小
- 创建时间、修改时间
- 负责人和审核状态

第 4 步：批量处理
- 自动化脚本解析
- 并行向量化
- 批量入库
```

**工具推荐**：
```
- Python 脚本：自定义收集逻辑
- Apache Nifi：数据流自动化
- Airflow：定时任务调度
```

---

#### 策略 2：增量更新

**适用场景**：
```
- 日常文档更新
- 新增文档处理
- 实时性要求高
```

**实施方式**：
```
1. 文件监控
   - 监控指定文件夹
   - 检测新增/修改/删除
   - 自动触发处理流程

2. 定时任务
   - 每天/每周定时检查
   - 对比修改时间
   - 处理变更文档

3. Webhook 触发
   - 文档系统推送通知

```

### 1.3 数据清洗流程

#### 步骤 1：格式转换

**目标**：统一转为纯文本格式

**PDF 处理**：
```
挑战：
- 扫描版 PDF（图片）无法直接提取
- 复杂排版（多栏、表格）难以还原
- 公式、图表丢失

解决方案：
1. 文本型 PDF
   - PyMuPDF（fitz）：速度快，保留结构
   - PyPDF2：简单提取
   
2. 扫描版 PDF
   - OCR 识别：Tesseract、PaddleOCR
   - 云端 OCR：百度、阿里、腾讯
   
3. 复杂排版
   - LayoutLM：深度学习布局理解
   - 商业工具：Mathpix（公式）
```

**Word 处理**：
```python
# 使用 python-docx
from docx import Document

doc = Document('file.docx')
text = '\n'.join([para.text for para in doc.paragraphs])
```

**HTML 处理**：
```python
# 使用 BeautifulSoup
from bs4 import BeautifulSoup

soup = BeautifulSoup(html_content, 'html.parser')
# 移除脚本和样式
for script in soup(['script', 'style']):
    script.decompose()
text = soup.get_text(separator=' ', strip=True)
```

---

#### 步骤 2：噪声去除

**常见噪声**：
```
1. 格式噪声
   - 多余空行和空格
   - 特殊字符和乱码
   - HTML 标签残留

2. 内容噪声
   - 页眉页脚
   - 页码
   - 版权声明
   - 广告推广

3. 结构噪声
   - 目录（无实质内容）
   - 索引
   - 空白页
```

---

#### 步骤 3：质量检查

**检查维度**：
```
1. 完整性检查
   - 文本长度是否合理（>100 字）
   - 是否有内容缺失
   - 关键章节是否完整

2. 可读性检查
   - 乱码检测（特殊字符比例）
   - 语言一致性（中英混排比例）
   - 句子完整性（断句检测）

3. 重复性检查
   - 文档内重复段落
   - 跨文档重复内容
   - 版本重复（新旧版本）
```

**自动化检查**：
```
1. 长度检查
2. 乱码检查
3. 重复检查
```

**人工抽检**：
```
抽检比例：
- 首批文档：30%（建立基准）
- 常规文档：5-10%
- 关键文档：100%

检查清单：
□ 内容是否完整
□ 格式是否正确
□ 表格是否还原
□ 公式是否准确
□ 引用是否清晰
```

## 2. 文档切片策略

### 2.1 切片的核心目标

**为什么要切片**：
```
1. 适配向量数据库
   - 向量检索适合短文本（500-1000 字）
   - 长文档检索效果差
   - 需要精确到片段级别

2. 提升检索精度
   - 用户问题通常针对具体知识点
   - 小片段更容易精确匹配
   - 避免大段无关信息干扰

3. 节省 token 成本
   - 只检索相关片段
   - 减少上下文长度
   - 降低 LLM 调用成本
```

**切片的权衡**：
```
切片太小：
✅ 精确匹配
✅ 节省 token
❌ 语义不完整
❌ 检索片段过多

切片太大：
✅ 语义完整
✅ 上下文充分
❌ 匹配不精确
❌ 包含噪声多

目标：找到最佳平衡点
```

---

### 2.2 固定长度切片

#### 方法 1：字符数切片
**参数建议**：
```
中文文档：
- 小片段：300-500 字（精确检索）
- 中片段：500-800 字（平衡方案）
- 大片段：800-1500 字（语义完整）

英文文档：
- 小片段：200-400 tokens
- 中片段：400-600 tokens
- 大片段：600-1000 tokens

重叠度（overlap）：
- 建议：10-20%
- 作用：避免切断语义
- 示例：500 字切片，重叠 100 字
```

**优点**：
```
✅ 实现简单，易于理解
✅ 切片均匀，便于管理
✅ 计算高效，适合批量处理
✅ 适合结构松散的文档
```

**缺点**：
```
❌ 可能切断句子和段落
❌ 忽略语义边界
❌ 上下文可能不完整
❌ 需要后处理拼接
```

**适用场景**：
```
- 博客文章、新闻稿
- 论坛帖子、聊天记录
- 结构松散的文档
- 快速原型验证
```

---

#### 方法 2：段落感知切片

**核心思想**：
```
在固定长度基础上，尽量保持段落完整性：
1. 以段落为单位
2. 累积到目标大小
3. 在段落边界切分
```

**优点**：
```
✅ 保持段落完整性
✅ 语义更连贯
✅ 阅读体验更好
✅ 实现复杂度适中
```

**缺点**：
```
❌ 段落过长时仍会切断
❌ 切片大小不均匀
❌ 需要识别段落边界
```

**适用场景**：
```
- 技术文档
- 制度文件
- 论文报告
- 大多数结构化文档（推荐作为默认方案）
```

---

### 2.3 语义切片

#### 方法 1：基于句子相似度

**核心思想**：
```
使用句子嵌入模型，在语义边界处切分：
1. 将文档分为句子
2. 计算相邻句子相似度
3. 在相似度最低处切分
```

**参数建议**：
```
相似度阈值：
- 0.5-0.6：较松，切片较大
- 0.6-0.7：适中（推荐）
- 0.7-0.8：较紧，切片较小

模型选择：
- 中文：bge-base-zh, m3e-base
- 英文：all-MiniLM-L6-v2
- 多语言：e5-base-v2
```

**优点**：
```
✅ 语义连贯性最好
✅ 自动识别主题边界
✅ 切片质量高
```

**缺点**：
```
❌ 计算成本高（需嵌入所有句子）
❌ 处理速度慢
❌ 依赖嵌入模型质量
❌ 阈值选择需要调优
```

**适用场景**：
```
- 高质量要求的场景
- 文档语义结构复杂
- 预算充足
- 离线批处理
```

---

#### 方法 2：基于层次化结构

**核心思想**：
```
利用文档的天然层次结构（章节 - 小节 - 段落）：
1. 识别标题层级
2. 按章节切分
3. 大章节再细分为小片段
```

**优点**：
```
✅ 保持文档结构
✅ 语义完整
✅ 便于理解上下文
✅ 可添加结构元数据
```

**缺点**：
```
❌ 依赖文档格式规范
❌ 实现复杂度高
❌ 不适用于无结构文档
```

**适用场景**：
```
- Markdown 文档
- 结构化的 Word/PDF
- 技术文档、书籍
- 制度文件
```

---

### 2.4 切片策略选择

#### 选择决策树

```
第 1 问：文档是否有清晰结构？
├─ 是（Markdown、带标题的 Word）
│   └─ 选择：层次化切片（优先）
│
└─ 否（纯文本、扫描件）
    └─ 继续第 2 问

第 2 问：对检索质量要求？
├─ 高（专业领域、关键应用）
│   └─ 选择：语义切片
│
└─ 中（一般应用）
    └─ 继续第 3 问

第 3 问：计算资源和时间？
├─ 充足
│   └─ 选择：段落感知切片
│
└─ 有限
    └─ 选择：固定长度切片（快速验证）
```

#### 推荐方案

**默认方案（适合 80% 场景）**：
```
段落感知切片 + 固定长度兜底
- 尽量保持段落完整
- 过长段落用固定长度切分
- 平衡质量和性能
```

**高质量方案**：
```
层次化切片 + 语义切片
- 优先按章节切分
- 大章节用语义切片细分
- 适合专业领域
```

**快速验证方案**：
```
固定长度切片
- 500 字一片，重叠 100 字
- 快速实现和验证
- 后续再优化
```

#### 技巧 3：父子索引

**目的**：兼顾检索精度和上下文完整

**实现**：
```
父文档（大块，2000 字）：
- 用于检索定位
- 包含完整上下文
- 数量较少

子文档（小块，500 字）：
- 用于精确匹配
- 语义聚焦
- 数量较多

检索流程：
1. 检索子文档，找到 Top-K 匹配
2. 通过子文档找到对应的父文档
3. 返回父文档给 LLM（完整上下文）
```

**优势**：
```
✅ 检索精确（小子文档）
✅ 上下文完整（大父文档）
✅ 减少片段数量
✅ 提升回答质量
```

## 3. 元数据管理

### 3.1 元数据的作用

**为什么需要元数据**：
```
1. 精细化检索
   - 按时间范围过滤（"查 2024 年的文档"）
   - 按部门过滤（"只看技术部的"）
   - 按类型过滤（"只要制度文件"）

2. 提升相关性
   - 优先检索最新版本
   - 优先检索权威来源
   - 优先检索核心文档

3. 权限控制
   - 按用户角色过滤
   - 按部门隔离
   - 按密级管理

4. 溯源和审计
   - 追踪文档来源
   - 记录处理时间
   - 版本管理
```

---

### 3.2 元数据类型

#### 类型 1：文档级元数据

**描述文档整体属性**：
```json
{
  "doc_id": "doc_12345",
```

```
基础信息：

统计信息：
- page_count：页数
- word_count：字数
- chunk_count：切片数量
```

### 3.3 元数据提取

#### 方法 1：规则提取

**适用场景**：格式规范的文档

- 从文件名提取
- 从文档结构提取

---

#### 方法 2：LLM 提取

**适用场景**：非结构化文档、需要语义理解

**优点**：
```
✅ 理解语义，提取准确
✅ 适应不同格式
✅ 可提取抽象概念（如目标读者）
✅ 无需手工编写规则
```

**缺点**：
```
❌ 成本较高（调用 LLM）
❌ 速度较慢
❌ 结果有一定随机性
❌ 需要人工校验
```

**优化建议**：
```
- 只对新文档使用 LLM 提取
- 提取后人工抽检（10%）
- 积累规则，逐步替代 LLM
- 使用小模型（gpt-4o-mini）控制成本
```

---

#### 方法 3：混合提取

**最佳实践**：规则 + LLM 结合

**流程**：
```
第 1 步：规则提取（快速、免费）
- 从文件名提取基础信息
- 从文档属性提取时间信息
- 从固定位置提取部门信息

第 2 步：LLM 补充（需要语义理解）
- 提取摘要和关键词
- 识别文档类型
- 判断目标读者

第 3 步：合并和校验
- 合并规则提取和 LLM 提取结果
- 检查必填字段
- 标记置信度
```

## 6. 面试高频问答

### 6.1 基础概念类

**Q1: 为什么 RAG 系统中数据处理这么重要？**

**参考答案**：
```
核心原因：Garbage In, Garbage Out（垃圾进，垃圾出）

数据质量决定上限：
1. 检索质量依赖切片质量
   - 切片语义不完整 → 检索结果不相关
   - 切片包含噪声 → 干扰 LLM 判断
   - 切片大小不合理 → 匹配精度下降

2. 元数据决定检索灵活性
   - 无元数据 → 只能向量检索
   - 有元数据 → 可过滤、加权、权限控制
   - 元数据质量 → 决定过滤准确性

3. 清洗质量影响最终效果
   - 乱码和噪声 → 向量化失真
   - 格式混乱 → 切片边界错误
   - 内容缺失 → 回答不完整

最佳实践：
- 投入 60% 精力在数据处理
- 投入 30% 精力在检索优化
- 投入 10% 精力在生成优化
```

**Q2: 文档切片大小应该如何选择？**

**参考答案**：
```
切片大小的权衡：

小切片（300-500 字）：
✅ 精确匹配，定位准确
✅ 节省 token，成本低
✅ 噪声少，聚焦核心
❌ 语义可能不完整
❌ 上下文不足
❌ 检索片段过多

大切片（800-1500 字）：
✅ 语义完整，连贯性好
✅ 上下文充分
✅ 检索片段少
❌ 匹配不够精确
❌ 包含无关信息
❌ token 成本高

推荐方案（分场景）：

1. FAQ/问答场景
   - 推荐：300-500 字
   - 原因：问题聚焦，精确匹配

2. 制度文档
   - 推荐：500-800 字
   - 原因：条款独立，适度上下文

3. 技术文档/教程
   - 推荐：800-1000 字
   - 原因：需要完整概念讲解

4. 论文/报告
   - 推荐：1000-1500 字
   - 原因：论证需要上下文

关键参数：
- 中文：500-800 字（默认）
- 英文：400-600 tokens
- 重叠度：10-20%（防止切断语义）

验证方法：
1. 抽样 100 个典型问题
2. 测试不同切片大小
3. 看检索准确率和回答质量
4. 选择最优值
```

**Q3: 元数据提取应该用规则还是 LLM？**

**参考答案**：
```
两种方法对比：

规则提取：
✅ 速度快（毫秒级）
✅ 成本低（无 API 费用）
✅ 稳定可靠（确定性结果）
✅ 易调试和维护
❌ 需要手工编写规则
❌ 适应性差（格式变化就失效）
❌ 无法提取抽象概念

LLM 提取：
✅ 理解语义，提取准确
✅ 适应不同格式
✅ 可提取抽象概念（如目标读者）
✅ 无需编写规则
❌ 成本高（每文档几分钱）
❌ 速度慢（秒级）
❌ 结果有随机性
❌ 需要人工校验

推荐方案：混合提取（最佳实践）

第 1 步：规则提取（快速、免费）
- 从文件名提取：部门、类型、日期
- 从文档属性提取：页数、作者
- 从固定位置提取：版本号

第 2 步：LLM 补充（需要语义理解）
- 提取摘要和关键词
- 识别文档类型
- 判断目标读者

第 3 步：合并和校验
- 合并两者结果
- 检查必填字段
- 标记置信度

成本优化：
- 首批文档：LLM 提取（建立基准）
- 新增文档：规则优先，失败再用 LLM
- 关键文档：100% LLM 提取
- 普通文档：规则提取 + 人工抽检

实际效果：
- 规则覆盖：70% 文档
- LLM 补充：30% 文档
- 成本降低：60-70%
- 准确率：>90%
```

**Q4: 如何处理扫描版 PDF 文档？**

```
挑战：

   - 百度 OCR：中文准确率高

   - 腾讯 OCR：性价比高
   
   选择建议：
   - 中文文档：PaddleOCR 或百度 OCR
   - 多语言：EasyOCR
   - 预算充足：商业 OCR

2. 版面分析
   目标：识别文档结构
   - 标题、段落、表格、图片

   - LayoutLM（微软）：深度学习

3. 表格处理

4. 公式处理
   方案：
   - Mathpix：公式识别最强
   - LaTeX OCR：开源替代

完整流程：
    扫描 PDF
    ↓

    ↓
    版面分析（LayoutLM）
    ↓
    结构化提取（标题、段落、表格）
    ↓
    质量检查（人工抽检）
    ↓
    转为 Markdown/文本

成本对比：
- 自研 OCR：开发成本高，边际成本低
- 商业 OCR：按量付费，0.01-0.05 元/页
- 建议：小批量用商业，大批量自研
```

**Q5: 如何设计文档切片的Overlap（重叠）策略？**

**参考答案**：
```
为什么需要 Overlap：

问题：切片可能切断语义，导致两个切片都无法完整理解
解决：添加重叠部分

Overlap 大小选择：

小重叠（5-10%）：
✅ 节省存储空间
✅ 减少冗余
❌ 可能仍会切断语义

中重叠（10-20%，推荐）：
✅ 平衡冗余和完整性
✅ 大多数场景适用
❌ 少量语义丢失

大重叠（20-30%）：
✅ 语义完整性最好
✅ 几乎不会切断
❌ 存储冗余大
❌ 检索结果重复

推荐方案：

1. 按文档类型
   - 制度文档：15%（条款独立）
   - 技术文档：20%（概念连贯）
   - 论文报告：20%（论证连续）
   - FAQ 问答：10%（问题独立）

2. 按切片大小
   - 小切片（300 字）：15%（45 字重叠）
   - 中切片（500 字）：20%（100 字重叠）
   - 大切片（1000 字）：15%（150 字重叠）

3. 智能 Overlap
   基于句子边界：
    不是机械地在固定位置切断
    而是在句子/段落边界处切分
    然后添加前后各 1-2 个句子

验证方法：
1. 抽样检查切片边界
2. 看是否有句子被切断
3. 调整 overlap_ratio 直到满意
```

**Q6: 如何保证切片后的语义完整性？**

**参考答案**：
```
语义完整性的挑战：

问题 1：切断句子
问题 2：切断上下文（指代不明）
问题 3：丢失结构信息（不知道章节信息）

解决方案：

1. 段落感知切片：在段落边界切分，避免切断段落。
2. 句子边界检测：在句子边界处切分。
3. 添加结构元数据补充结构信息。

4. 智能重叠
   - 不只是机械重叠
   - 而是添加前后语境
   - 前 1-2 句 + 后 1-2 句

5. 父子索引（最佳实践）
   父文档（2000 字）：完整章节
     ↓ 包含
   子文档 1-4（各 500 字）：具体片段
   
   检索：检索子文档
   返回：父文档（完整上下文）

6. 后处理拼接
   检索到多个相邻切片
   → 自动拼接
   → 恢复完整语义

质量检查：
人工抽检清单：
□ 是否有句子被切断
□ 是否有代词指代不明
□ 是否有"如下/上述"但内容缺失
□ 是否有结构信息丢失
□ 阅读是否流畅

自动化检查：
- 检测切片末尾是否在不合理位置
- 检测是否包含不完整句子
- 检测关键词密度是否异常

效果对比：
无优化：语义完整率 60-70%
+ 段落感知：75-85%
+ 父子索引：85-95%
```

```
  - 5% 问题在向量化
回归测试：优化后重新跑测试集，对比优化前后效果
```

**Q11: 数据处理中遇到过哪些坑？**

**参考答案**：
```
常见陷阱：

1. PDF 解析坑
   问题：
   - 扫描版当文本版处理（提取空白）
   - 多栏排版顺序错乱
   - 表格提取成纯文本（丢失结构）
   
   解决：
   - 先检测 PDF 类型（文本/扫描）
   - 版面分析后再提取
   - 表格专用解析器

2. 编码问题
   问题：
   - 文件编码不一致（UTF-8、GBK）
   - 特殊字符乱码
   - Emoji 处理不当
   
   解决：
   - 自动检测编码（chardet 库）
   - 统一转 UTF-8
   - 过滤或转义特殊字符

3. 切片切断语义

4. 元数据缺失
   问题：
   - 文档无标题
   - 部门信息缺失
   - 日期格式混乱
   
   解决：
   - LLM 补充提取
   - 建立默认值
   - 标准化日期格式

5. 重复文档
   问题：
   - 同一文档多个版本
   - 不同格式重复（PDF+Word）
   - 跨部门文档重复
   
   解决：
   - 文档去重（SimHash）
   - 版本管理
   - 建立文档关联

6. 隐私泄露
   问题：
   - 敏感信息未脱敏
   - 个人隐私泄露
   - 商业机密暴露
   
   解决：
   - 敏感信息识别（正则）
   - 自动脱敏
   - 权限控制

7. 过度清洗
   问题：
   - 把重要内容当噪声删除
   - 过度标准化丢失语义
   - 表格公式被清理
   
   解决：
   - 保守清洗策略
   - 白名单机制
   - 人工抽检

8. 性能瓶颈
   问题：
   - 大批量处理太慢
   - 向量化耗时
   - 内存溢出
   
   解决：
   - 并行处理
   - 批量向量化
   - 流式处理

9. 断点续传缺失
   问题：
   - 处理中断从头开始
   - 重复处理浪费资源
   - 无法追踪进度
   
   解决：
   - 检查点机制
   - 进度持久化
   - 失败重试

10. 质量监控缺失
    问题：
    - 处理质量无法量化
    - 问题发现滞后
    - 无法追踪趋势
    
    解决：
    - 建立质量指标
    - 自动化监控
    - 定期报告

实战建议：
- 上线前充分测试各种边界情况
- 建立完善的监控和告警
- 保留原始文档和处理日志
- 定期审查和优化流程
- 文档化所有规则和决策
```
