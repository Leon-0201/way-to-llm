# 6.3 工具集成

> **核心理解**：工具集成是 Agent 将大模型的推理能力转化为实际行动的关键桥梁。通过 Function Call、MCP、Skills 等机制，Agent 能够调用外部工具、API 和服务，实现从"思考"到"行动"的闭环。

---

## 1. Function Call / Tool Use

### 1.1 什么是 Function Call

**定义**：
Function Call（函数调用）是 LLM 将自然语言请求转化为结构化函数调用的能力。模型理解用户意图后，输出符合预定义 schema 的参数，触发外部函数执行。

**核心价值**：
```
1. 结构化输出：将开放式的文本生成转为结构化数据
2. 可执行性：直接触发代码执行和外部操作
3. 可靠性：参数经过 schema 验证，减少错误
4. 可扩展性：通过添加工具无限扩展能力边界
```

---

### 1.2 Function Call 工作流程

```
┌─────────────┐
│  用户请求    │ "帮我查一下北京的天气"
└──────┬──────┘
       ↓
┌─────────────────────────────┐
│  LLM 理解意图 + 选择工具     │
│  - 识别需要调用 weather_api │
│  - 提取参数：location=北京   │
└──────┬──────────────────────┘
       ↓
┌─────────────────────────────┐
│  输出结构化调用请求          │
│  {                           │
│    "name": "weather_api",   │
│    "arguments": {           │
│      "location": "北京"      │
│    }                        │
│  }                          │
└──────┬──────────────────────┘
       ↓
┌─────────────────────────────┐
│  执行函数并获取结果          │
│  → 调用外部 API              │
│  → 返回：{"temp": 25, ...}  │
└──────┬──────────────────────┘
       ↓
┌─────────────────────────────┐
│  LLM 生成自然语言响应        │
│  "北京当前温度 25°C..."      │
└─────────────────────────────┘
```

---

### 1.3 工具定义规范

**OpenAI 风格工具定义**：
```json
{
  "name": "search_weather",
  "description": "查询指定城市的天气信息",
  "parameters": {
    "type": "object",
    "properties": {
      "location": {
        "type": "string",
        "description": "城市名称，如'北京'、'上海'"
      },
      "date": {
        "type": "string",
        "description": "日期，格式 YYYY-MM-DD，默认为今天"
      }
    },
    "required": ["location"]
  }
}
```

**关键要素**：
```
1. name：工具的唯一标识符
2. description：清晰描述工具用途，帮助 LLM 理解何时使用
3. parameters：JSON Schema 定义参数结构
4. required：必需参数列表
```

---

### 1.4 工具选择策略

#### 策略 1：显式指定
```python
# 用户明确指定工具
用户："用计算器算一下 123 * 456"
→ 直接调用 calculator 工具
```

#### 策略 2：自动选择
```python
# LLM 根据意图自动选择
用户："3 乘以 7 等于多少？"
→ LLM 识别需要计算 → 自动选择 calculator
```

#### 策略 3：多工具协同
```python
# 复杂任务需要多个工具配合
用户："帮我找一家评分高的日料店并预订"
→ search_restaurants(菜系="日料")
→ filter_by_rating(评分>=4.5)
→ book_restaurant(店名，人数，时间)
```

### 1.5 实现示例

**定义工具**：
```python
from pydantic import BaseModel, Field

class WeatherTool(BaseModel):
    name: str = "get_weather"
    description: str = "查询城市天气"
    
    class Parameters(BaseModel):
        location: str = Field(..., description="城市名称")
        unit: str = Field("celsius", description="温度单位：celsius 或 fahrenheit")
    
    def execute(self, params: Parameters) -> dict:
        # 调用天气 API
        response = weather_api.get_weather(params.location)
        return {
            "temperature": response.temp,
            "condition": response.condition,
            "humidity": response.humidity
        }
```

**调用流程**：
```python
from openai import OpenAI

client = OpenAI()

# 定义工具
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "查询城市天气",
        "parameters": {
            "type": "object",
            "properties": {
                "location": {"type": "string", "description": "城市名"}
            },
            "required": ["location"]
        }
    }
}]

# 发送请求
response = client.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "北京天气怎么样？"}],
    tools=tools
)

# 处理工具调用
if response.choices[0].message.tool_calls:
    tool_call = response.choices[0].message.tool_calls[0]
    result = execute_tool(tool_call.function.name, 
                         tool_call.function.arguments)
    
    # 将结果返回给 LLM 生成最终响应
    final_response = client.chat.completions.create(
        model="gpt-4",
        messages=[
            {"role": "user", "content": "北京天气怎么样？"},
            response.choices[0].message,
            {"role": "tool", "content": json.dumps(result)}
        ]
    )
```

## 2. MCP（Model Context Protocol）

### 2.1 什么是 MCP

**定义**：
MCP（Model Context Protocol）是一个开放的协议标准，用于统一大模型与外部数据源、工具的连接方式。它提供了一种通用的"USB-C 接口"，让 AI 应用能够即插即用各种服务。

**核心目标**：
```
1. 标准化：统一的连接协议，避免碎片化
2. 互操作性：不同模型、不同工具之间的无缝对接
3. 安全性：内置权限控制和数据隔离
4. 可扩展性：轻松添加新的数据源和工具
```

---

### 2.2 MCP 架构

```
┌─────────────────────────────────────────┐
│         AI 应用（Host）                  │
│  （Claude Desktop、IDE 插件等）          │
└──────────────┬──────────────────────────┘
               │ MCP Protocol
               │
┌──────────────▼──────────────────────────┐
│         MCP Server                       │
│  ┌─────────────────────────────────┐    │
│  │  Resources（资源）               │    │
│  │  - 文件系统                      │    │
│  │  - 数据库                        │    │
│  │  - API 服务                       │    │
│  └─────────────────────────────────┘    │
│  ┌─────────────────────────────────┐    │
│  │  Tools（工具）                   │    │
│  │  - 代码执行                      │    │
│  │  - 网络搜索                      │    │
│  │  - 自定义功能                    │    │
│  └─────────────────────────────────┘    │
│  ┌─────────────────────────────────┐    │
│  │  Prompts（提示词模板）           │    │
│  │  - 预定义工作流                  │    │
│  │  - 最佳实践                      │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│         外部服务                         │
│  （GitHub、Slack、Postgres 等）          │
└─────────────────────────────────────────┘
```

---

### 2.3 MCP 核心概念

#### 1. Resources（资源）
**定义**：MCP Server 暴露的数据源，AI 可以读取（有时可写入）。

**示例**：
```json
{
  "uri": "file:///Users/project/src/main.py",
  "name": "main.py",
  "description": "项目主入口文件",
  "mimeType": "text/x-python"
}
```

**使用场景**：
- 读取本地文件
- 访问数据库记录
- 获取 API 数据

---

#### 2. Tools（工具）
**定义**：MCP Server 提供的可执行函数，AI 可以调用。

**示例**：
```json
{
  "name": "execute_sql",
  "description": "在数据库上执行 SQL 查询",
  "inputSchema": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "SQL 查询语句"
      }
    },
    "required": ["query"]
  }
}
```

---

#### 3. Prompts（提示词模板）
**定义**：预定义的提示词模板，帮助用户快速启动常见任务。

**示例**：
```json
{
  "name": "code_review",
  "description": "对代码进行审查",
  "arguments": [
    {
      "name": "file_path",
      "description": "要审查的文件路径",
      "required": true
    }
  ]
}
```

### 2.5 开发 MCP Server

**基础示例（Python）**：
```python
from mcp.server import Server
from mcp.server.stdio import stdio_server
from mcp.types import Resource, Tool

app = Server("my-server")

@app.list_resources()
async def list_resources():
    return [
        Resource(
            uri="file:///data/report.txt",
            name="报告文件",
            mimeType="text/plain"
        )
    ]

@app.read_resource()
async def read_resource(uri):
    if uri == "file:///data/report.txt":
        return open("report.txt").read()
    raise ValueError("Resource not found")

@app.list_tools()
async def list_tools():
    return [
        Tool(
            name="process_data",
            description="处理数据文件",
            inputSchema={
                "type": "object",
                "properties": {
                    "file": {"type": "string", "description": "文件路径"}
                }
            }
        )
    ]

@app.call_tool()
async def call_tool(name, arguments):
    if name == "process_data":
        result = process_file(arguments["file"])
        return result
    raise ValueError(f"Unknown tool: {name}")

async def main():
    async with stdio_server() as streams:
        await app.run(
            streams[0],
            streams[1],
            app.create_initialization_options()
        )

if __name__ == "__main__":
    import asyncio
    asyncio.run(main())
```

## 3. Skills/Plugins

### 3.1 什么是 Skills/Plugins

**定义**：
Skills（技能）或 Plugins（插件）是预定义的工具集合或能力模块，为 Agent 提供特定领域的专业能力。它们通常是经过优化和测试的可靠实现。

**与 Function Call 的区别**：
```
| 维度 | Function Call | Skills/Plugins |
|------|---------------|----------------|
| 粒度 | 单个函数 | 能力模块/集合 |
| 复杂度 | 简单调用 | 可能包含多步骤 |
| 复用性 | 通用 | 领域特定 |
| 封装性 | 低 | 高 |
```

---

### 3.2 Skill 的结构

#### 文档驱动结构

**典型组成**：
```
skill-name/
├── SKILL.md            # 技能描述（人类可读）
├── scripts/            # 执行脚本
├── examples/           # 使用示例
├── utils/              # 工具函数
└── README.md           # 使用说明
```

**SKILL.md 示例**：
```markdown
# 数据分析技能

## 能力描述
提供 CSV/Excel 文件分析、数据可视化、趋势预测等功能。

## 可用脚本

### analyze_csv
分析 CSV 文件，生成统计报告

**用法**：
```bash
python scripts/analyze_csv.py --input data.csv --output report.json
```

**示例**：
```bash
# 基础分析
python scripts/analyze_csv.py --input sales.csv

# 按地区分组
python scripts/analyze_csv.py --input sales.csv --group-by region
```

### plot_chart
生成数据可视化图表

**用法**：
```bash
python scripts/plot_chart.py --data data.json --type line
```

## 使用示例

### 示例 1：销售数据分析
用户："分析这个销售数据"
→ 自动调用 scripts/analyze_csv.py

### 示例 2：生成趋势图
用户："画出上个月的销售趋势"
→ 调用 scripts/plot_chart.py --type line --period last_month
```

**特点**：
- 更加轻量和实用
- 适合文档驱动的 Skill 系统
- 强调可读性、易用性、快速上手
- 更适合开源项目或开发者社区
- LLM 可以直接阅读 Markdown 理解技能

---

#### 结构对比

| 维度 | 规范化结构 | 文档驱动结构 |
|------|-----------|-------------|
| 描述文件 | manifest.json（机器可读） | SKILL.md（人类可读） |
| 实现方式 | handlers/（代码逻辑） | scripts/（脚本集合） |
| 学习成本 | 需要理解 API 规范 | 直接看 examples/ |
| 适用场景 | 平台化、自动化系统 | 文档驱动、社区分享 |
| 灵活性 | 低（需要遵循规范） | 高（自由组织） |
| LLM 友好度 | 中（需要解析 JSON） | 高（直接阅读 Markdown） |

**实践建议**：
- 个人项目/开源项目：推荐文档驱动结构，快速迭代
- 企业级应用：推荐规范化结构，便于管理和自动化
- 混合方案：SKILL.md（给人和 LLM 看）+ manifest.json（给系统看）

## 4. API 集成

### 4.1 API 集成模式

#### 模式 1：直接调用
```python
import requests

def call_weather_api(city: str) -> dict:
    """直接调用天气 API"""
    response = requests.get(
        f"https://api.weather.com/v1/current",
        params={"city": city, "key": API_KEY}
    )
    return response.json()
```

**适用场景**：
- 简单的 RESTful API
- 无状态调用
- 一次性集成

---

#### 模式 2：SDK 封装
```python
from stripe import StripeClient

class PaymentTool:
    def __init__(self, api_key: str):
        self.client = StripeClient(api_key)
    
    def create_payment(self, amount: int, currency: str) -> dict:
        """创建支付"""
        return self.client.checkout.sessions.create({
            "amount": amount,
            "currency": currency,
            # ...
        })
```

**适用场景**：
- 复杂的 API（多端点、多操作）
- 需要认证管理
- 频繁调用

---

#### 模式 3：API Gateway
```python
class APIGateway:
    """统一的 API 网关"""
    
    def __init__(self):
        self.services = {
            "weather": WeatherService(),
            "payment": PaymentService(),
            "search": SearchService()
        }
    
    def call(self, service: str, action: str, params: dict):
        """统一调用接口"""
        if service not in self.services:
            raise ValueError(f"未知服务：{service}")
        return getattr(self.services[service], action)(**params)
```

**适用场景**：
- 多个 API 需要统一管理
- 需要限流、缓存等中间件
- 需要统一的错误处理

---

### 4.2 API 认证与安全

#### 认证方式

**1. API Key**：
```python
headers = {"Authorization": f"Bearer {API_KEY}"}
response = requests.get(url, headers=headers)
```

**2. OAuth 2.0**：
```python
from requests_oauthlib import OAuth2Session

oauth = OAuth2Session(client_id, token=token)
response = oauth.get(protected_url)
```

**3. JWT**：
```python
import jwt

token = jwt.encode(payload, SECRET_KEY, algorithm="HS256")
headers = {"Authorization": f"Bearer {token}"}
```

---

#### 安全最佳实践

```
1. 密钥管理
   - 使用环境变量或密钥管理服务
   - 绝不硬编码在代码中
   - 定期轮换密钥

2. 最小权限原则
   - 只申请必要的权限
   - 使用 scoped tokens

3. 速率限制
   - 实现客户端限流
   - 避免触发 API 限流

4. 错误处理
   - 不泄露敏感信息
   - 记录详细日志用于调试

5. 数据加密
   - 使用 HTTPS
   - 敏感数据加密存储
```

### 4.3 实现示例：集成 Slack API

**定义工具**：
```python
from slack_sdk import WebClient

class SlackTool:
    def __init__(self, token: str):
        self.client = WebClient(token=token)
    
    def send_message(self, channel: str, text: str) -> dict:
        """发送消息到 Slack 频道"""
        response = self.client.chat_postMessage(
            channel=channel,
            text=text
        )
        return {"success": True, "ts": response["ts"]}
    
    def get_channel_info(self, channel_id: str) -> dict:
        """获取频道信息"""
        response = self.client.conversations_info(channel=channel_id)
        return response["channel"]
    
    def list_channels(self) -> list:
        """列出所有频道"""
        response = self.client.conversations_list()
        return [
            {"id": ch["id"], "name": ch["name"]}
            for ch in response["channels"]
        ]
```

**注册为工具**：

## 5. 工具选择与最佳实践

### 5.1 何时使用哪种方式

| 场景 | 推荐方案 | 理由 |
|------|----------|------|
| 简单函数调用 | Function Call | 轻量、直接 |
| 标准化集成 | MCP | 开放标准、互操作性 |
| 领域专业能力 | Skills/Plugins | 封装好、可复用 |
| 第三方服务 | API 集成 | 直接对接、灵活 |

---

### 5.2 工具设计原则

```
1. 单一职责
   - 每个工具只做一件事
   - 避免工具过于复杂

2. 清晰命名
   - 名称准确描述功能
   - 帮助 LLM 正确选择

3. 完善描述
   - description 清晰明确
   - 包含使用场景和限制

4. 参数验证
   - 使用 schema 验证输入
   - 提供友好的错误信息

5. 幂等性
   - 可重复调用不产生副作用
   - 便于 LLM 重试

6. 可观测性
   - 记录调用日志
   - 监控性能和错误
```

---

### 5.3 常见陷阱与解决方案

**陷阱 1：工具过多导致选择困难**
```
问题：提供几十个工具，LLM 难以选择
解决：
- 按场景分组工具
- 使用 hierarchical tools
- 提供工具使用指南
```

**陷阱 2：参数复杂导致调用失败**
```
问题：参数太多或太复杂，LLM 经常填错
解决：
- 简化参数结构
- 提供默认值
- 增加参数验证和提示
```

**陷阱 3：工具执行时间长**
```
问题：某些工具执行很慢，影响体验
解决：
- 异步执行
- 提供进度反馈
- 设置超时和重试
```

**陷阱 4：错误处理不当**
```
问题：工具失败后 LLM 不知道如何处理
解决：
- 返回结构化错误信息
- 提供错误恢复建议
- 实现降级方案
```

## 7. 面试高频问答

### 7.1 基础概念类

**Q1: 什么是 Function Call？它解决了什么问题？**

**参考答案**：
```
Function Call（函数调用）是 LLM 将自然语言请求转化为结构化函数调用的能力。

核心解决的问题：
1. 结构化输出：将开放式文本生成转为结构化数据
2. 可执行性：直接触发代码执行和外部操作
3. 可靠性：参数经过 schema 验证，减少错误
4. 可扩展性：通过添加工具无限扩展能力边界

工作流程：
用户请求 → LLM 理解意图 → 输出结构化调用 → 执行函数 → 返回结果 → LLM 生成响应

典型应用：天气查询、计算器、数据库查询等需要精确执行的场景。
```

---

**Q2: Function Call 和普通 JSON Mode 有什么区别？**

**参考答案**：
```
区别点：
1. 目的不同
   - Function Call：触发外部函数执行
   - JSON Mode：仅保证输出格式为 JSON

2. 语义不同
   - Function Call：包含工具名称、参数，有明确意图
   - JSON Mode：只是结构化数据，无语义

3. 执行性不同
   - Function Call：可直接执行对应函数
   - JSON Mode：需要额外解析和处理

4. 验证不同
   - Function Call：有预定义的 schema 验证
   - JSON Mode：只验证 JSON 格式

示例：
Function Call: {"name": "get_weather", "arguments": {"location": "北京"}}
JSON Mode: {"location": "北京", "temperature": 25}  # 无语义，需额外解析
```

---

**Q3: 什么是 MCP？它解决了什么问题？**

**参考答案**：
```
MCP（Model Context Protocol）是一个开放的协议标准，用于统一大模型与外部数据源、工具的连接方式。

解决的核心问题：
1. 碎片化：不同模型、不同工具之间缺乏统一接口
2. 互操作性差：每次集成新工具都需要定制开发
3. 安全性弱：缺乏统一的权限控制和数据隔离
4. 扩展困难：添加新服务成本高

MCP 的核心组件：
- Resources（资源）：文件系统、数据库、API 服务等
- Tools（工具）：可执行函数，如代码执行、网络搜索
- Prompts（提示词模板）：预定义工作流和最佳实践

优势：
- 标准化：统一的"USB-C 接口"
- 即插即用：轻松添加新的数据源和工具
- 安全性：内置权限控制
- 跨平台：支持不同模型和服务
```

### 7.2 技术实践类

**Q4: 如何设计一个工具选择机制？**

**参考答案**：
```
我的设计框架：

1. 工具表示
   - 名称、描述、参数定义（JSON Schema）
   - 向量化存储（用于语义搜索）
   - 注册表管理

2. 选择策略（三种方式结合）
   - 基于规则：关键词匹配，快速但不够灵活
   - 基于语义：向量相似度，灵活但需要预计算
   - 基于 LLM：理解能力强，但成本高

3. 执行流程
   - 参数验证 → 执行前检查 → 工具调用 → 结果验证 → 错误处理

4. 优化措施
   - 缓存常用工具选择
   - 批量调用减少延迟
   - 监控工具使用情况

实践建议：
- 小系统：基于规则即可
- 中等系统：规则 + 语义
- 复杂系统：三种策略结合
- 关键：完善的错误处理和监控
```

---

**Q5: 如何保证工具调用的安全性？**

**参考答案**：
```
安全设计框架：

1. 权限控制
   - 工具调用权限验证（白名单机制）
   - 敏感操作需要确认（如删除、支付）
   - 资源使用限制（如文件大小、调用次数）

2. 输入验证
   - 参数合法性检查（类型、范围、格式）
   - 注入攻击防护（SQL 注入、命令注入）
   - 大小限制（防止 DoS 攻击）

3. 沙盒隔离
   - 代码执行在沙盒环境中
   - 限制网络访问（禁止访问内网）
   - 文件系统隔离（限制访问路径）

4. 审计与监控
   - 记录所有工具调用日志
   - 异常行为检测
   - 实时告警和熔断

5. 密钥管理
   - 使用环境变量或密钥管理服务
   - 绝不硬编码在代码中
   - 定期轮换密钥

示例（OpenClaw 的 exec 工具）：
- 权限检查 → 审批检查 → 沙盒决策 → 执行命令 → 超时处理 → 输出截断 → 审计记录
```

**Q6: 如何处理工具调用失败的情况？**

**参考答案**：
```
错误处理策略：

1. 错误分类
   - 参数错误：验证失败、缺少必需参数
   - 执行错误：API 失败、超时、资源不足
   - 权限错误：未授权、权限不足
   - 业务错误：业务逻辑失败（如余额不足）

2. 错误返回格式
   {
     "success": false,
     "error": {
       "code": "TIMEOUT",
       "message": "工具执行超时",
       "suggestion": "请重试或减小数据量"
     }
   }

3. LLM 友好的错误处理
   - 返回结构化错误信息（而非堆栈跟踪）
   - 提供错误恢复建议
   - 包含可操作的下一步指导

4. 重试策略
   - 指数退避重试（1s, 2s, 4s, 8s）
   - 最大重试次数限制（如 3 次）
   - 区分可重试和不可重试错误

5. 降级方案
   - 提供备用工具
   - 返回部分结果
   - 转人工处理

6. 让 LLM 参与错误恢复
   - 将错误信息返回给 LLM
   - LLM 根据错误调整策略
   - 支持 LLM 选择其他工具
```

---

**Q7: 如何优化多工具调用的性能？**

**参考答案**：
```
优化策略：

1. 并行执行
   - 识别独立任务，并行调用多个工具
   - 使用 asyncio 或线程池
   - 注意：有依赖关系的任务不能并行

2. 缓存策略
   - 缓存常用工具的调用结果
   - 设置合理的 TTL（如 5 分钟）
   - 注意：实时数据不适合缓存

3. 批量处理
   - 合并多个相似调用为一次批量调用
   - 减少网络往返次数
   - 示例：batch_get_weather([北京，上海，广州])

4. 流式输出
   - 长任务返回流式结果
   - 避免一次性加载大量数据
   - 提升用户体验

5. 超时控制
   - 为每个工具设置合理的超时时间
   - 避免单个工具卡住整个流程
   - 实现优雅降级

6. 工具选择优化
   - 缓存工具选择结果
   - 使用向量相似度快速匹配
   - 避免每次都让 LLM 选择

性能对比示例：
- 串行：3 个工具 × 2s = 6s
- 并行：max(2s, 2s, 2s) = 2s
- 提升：3 倍
```

### 7.3 架构设计类

**Q8: Function Call、Skills、MCP 三者如何选择？**

**参考答案**：
```
选择框架：

| 场景 | 推荐方案 | 理由 |
|------|----------|------|
| 简单函数调用 | Function Call | 轻量、直接、原生支持 |
| 标准化集成 | MCP | 开放标准、互操作性强 |
| 领域专业能力 | Skills/Plugins | 封装好、可复用、易分享 |
| 第三方服务 | API 集成 | 直接对接、灵活 |
| 企业内部系统 | Skills + MCP | 封装业务能力 + 标准化接口 |

决策树：
1. 是否需要调用外部函数？
   - 否 → 不需要 Function Call
   - 是 → 继续 2

2. 是否是简单的单次调用？
   - 是 → Function Call
   - 否 → 继续 3

3. 是否需要跨平台、跨模型复用？
   - 是 → MCP
   - 否 → 继续 4

4. 是否需要封装领域知识、便于分享？
   - 是 → Skills
   - 否 → 直接 API 集成

实际案例：
- 天气查询：Function Call（简单、直接）
- 企业 CRM 集成：Skills（封装业务能力）
- 本地文件系统：MCP（标准化、跨平台）
```

### 7.4 开放思考类

**Q11: Function Call 的局限性是什么？未来会如何发展？**

**参考答案**：
```
当前局限性：

1. 依赖预定义 Schema
   - 需要事先定义工具，无法动态创建
   - 复杂场景下 Schema 定义困难

2. 上下文限制
   - 工具数量受上下文窗口限制
   - 大量工具时选择准确率下降

3. 状态管理
   - 无状态调用，难以处理多轮交互
   - 需要外部管理会话状态

4. 复杂任务处理
   - 单步调用，难以表达复杂工作流
   - 需要多次往返调用

5. 错误恢复
   - 依赖 LLM 自行处理错误
   - 缺乏系统性的错误恢复机制

未来发展方向：

1. 动态工具创建
   - LLM 根据需求动态生成工具
   - 代码生成 + 自动执行

2. 层次化工具系统
   - 工具组合成工作流
   - 支持嵌套调用

3. 状态感知
   - 工具调用携带上下文状态
   - 支持多轮对话和长程依赖

4. 自主学习和优化
   - 根据使用反馈优化工具选择
   - 自动发现更优工具组合

5. 标准化和互操作
   - MCP 等标准普及
   - 跨平台工具复用

6. 多模态工具
   - 支持图像、音频、视频处理
   - 支持物理世界交互（机器人）
```
