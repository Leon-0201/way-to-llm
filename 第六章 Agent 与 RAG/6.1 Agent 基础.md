# 6.1 Agent 基础

> **核心理解**：Agent 是能够感知环境、进行推理、执行行动以实现目标的智能系统。核心在于将大模型的推理能力与外部工具和记忆系统结合，形成闭环的自主执行能力。

---

## 1. 什么是 Agent

### 1.1 定义与核心特征

**定义**：
Agent 是一个能够感知环境、进行推理、执行行动以实现特定目标的自主或半自主智能系统。

**核心特征**：
```
1. 自主性（Autonomy）
   - 能够在没有人工干预的情况下做出决策
   - 根据目标自主规划执行步骤
   - 适应环境变化调整策略

2. 感知能力（Perception）
   - 接收和理解用户输入
   - 获取外部环境信息
   - 监控执行状态和结果

3. 推理能力（Reasoning）
   - 分析问题和情境
   - 制定计划和策略
   - 评估和选择最优方案

4. 行动能力（Action）
   - 调用工具和 API
   - 与环境交互
   - 执行具体任务

5. 学习能力（Learning）
   - 从经验中学习
   - 持续优化行为
   - 适应新场景
```

---

### 1.2 Agent 与 传统程序的区别

| 维度 | 传统程序 | Agent |
|------|----------|-------|
| **决策方式** | 预定义规则 | 自主推理 |
| **灵活性** | 固定流程 | 动态调整 |
| **适应性** | 有限 | 强 |
| **交互方式** | 命令行/API | 自然语言 |
| **错误处理** | 异常捕获 | 自我修正 |
| **学习能力** | 无 | 持续学习 |

**示例对比**：
```
传统程序（预订系统）：
用户 → 选择日期 → 选择餐厅 → 确认 → 完成
（固定流程，无法处理流程外请求）

Agent 系统：
用户："我想找个适合约会的餐厅"
Agent → 理解意图 → 询问偏好 → 搜索推荐 → 预订 → 完成
（灵活交互，可处理模糊需求）
```

## 2. Agent 的设计范式

### 2.1 ReAct（Reasoning + Acting）

**核心思想**：将推理和行动交替进行，边思考边执行。

**工作流程**：
```
Thought（思考）→ Action（行动）→ Observation（观察）→ 循环

具体步骤：
1. Thought: 分析当前情况，决定下一步
2. Action: 执行具体行动（调用工具）
3. Observation: 观察行动结果
4. 重复 1-3 直到任务完成
```

**示例**：
```
用户：今天北京天气怎么样？适合出门吗？

Agent 执行过程：
Thought: 我需要查询北京今天的天气信息
Action: 调用天气 API(location="北京")
Observation: {"temperature": 25, "condition": "晴", "aqi": 80}

Thought: 现在我有天气信息了，需要评估是否适合出门
Action: 分析天气数据（温度适宜、晴天、空气质量良好）
Observation: 综合评估为"适合出门"

Thought: 我已经有了完整信息，可以回答用户了
Action: 生成最终回答
最终输出："北京今天晴朗，气温 25 度，空气质量良好，非常适合出门活动！"
```

**优势**：
```
✅ 透明可解释：思考过程清晰可见
✅ 灵活适应：可根据情况调整策略
✅ 错误可修正：发现错误可以及时调整
✅ 工具集成：易于调用外部工具
```

**劣势**：
```
❌ 延迟较高：多轮推理增加响应时间
❌ token 消耗：思考过程消耗更多 token
❌ 可能过度思考：简单问题也走完整流程
```

---

### 2.2 Plan-and-Solve（计划与执行）

**核心思想**：先制定完整计划，再逐步执行。

**工作流程**：
```
1. 理解任务：分析用户需求和目标
2. 制定计划：分解为可执行的子任务
3. 执行计划：按顺序或并行执行子任务
4. 整合结果：汇总各子任务结果
5. 验证输出：检查是否满足原始目标
```

**示例**：
```
用户：帮我规划一个 3 天的北京旅行行程

Agent 执行过程：

步骤 1：理解任务
- 目标：3 天北京旅行规划
- 约束：时间 3 天，地点北京
- 隐含需求：景点、餐饮、住宿、交通

步骤 2：制定计划
Day 1: 市区经典景点
  - 上午：天安门广场、故宫
  - 下午：王府井购物
  - 晚上：北京烤鸭

Day 2: 历史文化之旅
  - 上午：天坛公园
  - 下午：颐和园
  - 晚上：胡同体验

Day 3: 现代北京
  - 上午：798 艺术区
  - 下午：三里屯
  - 晚上：返程准备

步骤 3：执行计划
- 查询各景点开放时间和门票
- 查询餐厅推荐和预订
- 查询酒店和交通信息

步骤 4：整合结果
- 生成详细行程单
- 添加实用建议
- 提供预算估算

步骤 5：验证输出
- 检查时间安排是否合理
- 确认信息准确性
- 确保覆盖主要需求
```

**优势**：
```
✅ 全局优化：考虑整体而非局部
✅ 条理清晰：结构化输出
✅ 易于并行：独立子任务可并行执行
✅ 结果可控：可预先评估计划可行性
```

**劣势**：
```
❌ 灵活性差：计划制定后不易调整
❌ 前期成本高：需要完整规划
❌ 环境适应弱：难以应对突发变化
```

---

### 2.3 ReAct vs Plan-and-Solve 对比

| 维度 | ReAct | Plan-and-Solve |
|------|-------|----------------|
| **决策方式** | 逐步推理 | 预先规划 |
| **灵活性** | 高（动态调整） | 中（按部就班） |
| **透明度** | 高（思考可见） | 中（计划可见） |
| **效率** | 中（可能过度思考） | 高（按计划执行） |
| **适用场景** | 探索性任务、需要工具调用 | 结构化任务、多步骤规划 |
| **错误处理** | 即时修正 | 计划调整 |

**选择建议**：
```
使用 ReAct：
- 需要频繁调用工具
- 环境变化快
- 任务不明确需要探索

使用 Plan-and-Solve：
- 任务结构清晰
- 需要全局优化
- 子任务相对独立
```

### 2.4 其他设计范式

#### Reflexion（反思）
```
核心思想：通过自我反思改进行为

流程：
执行 → 评估 → 反思 → 调整 → 再执行

应用：
- 代码生成和调试
- 复杂问题求解
- 持续优化场景
```

#### Tree of Thoughts（ToT）
```
核心思想：树状探索多个推理路径

流程：
问题 → 多分支思考 → 评估 → 选择最优路径

应用：
- 创意写作
- 策略规划
- 复杂决策
```

## 3. 常见 Agent 框架

### 3.1 LangChain

**核心特点**：
```
- 组件化设计：Chain、Agent、Tool 等组件
- 丰富的集成：支持多种 LLM 和工具
- 生态系统完善：大量预构建组件
- 易于上手：文档丰富，社区活跃
```

**核心概念**：
```python
from langchain.agents import AgentExecutor, create_react_agent
from langchain.tools import Tool

# 1. 定义工具
tools = [
    Tool(
        name="Search",
        func=search_function,
        description="搜索信息"
    ),
    Tool(
        name="Calculator",
        func=calculate_function,
        description="计算数学表达式"
    )
]

# 2. 创建 Agent
agent = create_react_agent(llm, tools, prompt)

# 3. 执行
agent_executor = AgentExecutor(agent=agent, tools=tools)
response = agent_executor.invoke({"input": "用户问题"})
```

**适用场景**：
- ✅ 快速原型开发
- ✅ 需要丰富工具集成
- ✅ 团队协作项目

---

### 3.2 LangGraph

**核心特点**：
```
- 图结构：基于有向图的工作流
- 状态管理：显式的状态定义和传递
- 循环支持：支持循环和条件分支
- 可观测性：内置追踪和调试
```

**核心概念**：
```python
from langgraph.graph import StateGraph, END

# 1. 定义状态
class AgentState(TypedDict):
    messages: list
    current_step: str
    results: dict

# 2. 定义节点
def node_a(state):
    # 处理逻辑
    return {"messages": [...]}

def node_b(state):
    # 处理逻辑
    return {"results": {...}}

# 3. 构建图
workflow = StateGraph(AgentState)
workflow.add_node("node_a", node_a)
workflow.add_node("node_b", node_b)
workflow.add_edge("node_a", "node_b")
workflow.add_edge("node_b", END)

# 4. 编译和运行
app = workflow.compile()
result = app.invoke(initial_state)
```

**适用场景**：
- ✅ 复杂工作流
- ✅ 需要状态管理
- ✅ 多 Agent 协作

---

### 3.3 AutoGen

**核心特点**：
```
- 多 Agent 协作：多个 Agent 协同完成任务
- 对话驱动：通过对话实现协作
- 灵活角色：可定义不同角色的 Agent
- 人类参与：支持人机协作
```

**核心概念**：
```python
from autogen import ConversableAgent, UserProxyAgent

# 1. 创建 Agent
assistant = ConversableAgent(
    name="Assistant",
    llm_config={"config_list": [...]}
)

user_proxy = UserProxyAgent(
    name="User",
    human_input_mode="ALWAYS"
)

# 2. 定义任务
user_proxy.initiate_chat(
    assistant,
    message="帮我分析这个数据集"
)

# 3. 多 Agent 协作
group_chat = GroupChat(
    agents=[agent1, agent2, agent3],
    messages=[]
)
```

**适用场景**：
- ✅ 多 Agent 协作
- ✅ 需要人类参与
- ✅ 复杂任务分解

---

### 3.4 框架对比

| 框架 | 核心优势 | 学习曲线 | 适用场景 |
|------|---------|---------|---------|
| **LangChain** | 生态丰富、易上手 | 低 | 快速开发、原型验证 |
| **LangGraph** | 状态管理、图结构 | 中 | 复杂工作流、循环任务 |
| **AutoGen** | 多 Agent 协作 | 中 | 协作场景、人机交互 |

**选择建议**：
```
- 快速原型：LangChain
- 复杂工作流：LangGraph
- 多 Agent 协作：AutoGen
- 生产环境：根据具体需求选择或组合使用
```

## 4. Agent 的核心能力

### 4.1 任务规划与分解

**能力描述**：将复杂任务分解为可执行的子任务。

**实现方式**：
```
1. 目标分析
   - 理解最终目标
   - 识别约束条件
   - 确定成功标准

2. 任务分解
   - 分解为子任务
   - 确定依赖关系
   - 安排执行顺序

3. 资源分配
   - 选择合适的工具
   - 分配时间和资源
   - 考虑并行执行

4. 动态调整
   - 监控执行进度
   - 处理异常情况
   - 调整计划
```

**示例**：
```
任务：组织一场线上技术分享会

分解：
├── 1. 确定主题和讲师
│   ├── 调研热门话题
│   ├── 联系潜在讲师
│   └── 确认时间和主题
├── 2. 宣传和报名
│   ├── 制作宣传材料
│   ├── 发布活动信息
│   └── 管理报名名单
├── 3. 技术准备
│   ├── 选择直播平台
│   ├── 测试设备
│   └── 准备 backup 方案
└── 4. 活动执行
    ├── 主持开场
    ├── 技术支持
    └── 互动问答
```

```
1. 信息获取类

2. 计算分析类
```

**选择策略**：

```
   - 错误处理机制

```

### 4.3 自我反思与修正

**能力描述**：评估自身输出，发现并修正错误。

**实现方式**：
```
1. 输出评估
   - 检查格式是否正确
   - 验证内容准确性
   - 评估完整性

2. 错误识别
   - 逻辑错误
   - 事实错误
   - 格式错误

3. 修正策略
   - 重新生成
   - 局部修改
   - 补充信息

4. 学习改进
   - 记录错误类型
   - 更新提示词
   - 优化决策策略
```

**示例代码**：
```python
def self_reflect(output, criteria):
    # 1. 定义评估标准
    checks = {
        "format": check_format(output),
        "accuracy": check_accuracy(output),
        "completeness": check_completeness(output)
    }
    
    # 2. 识别问题
    issues = []
    for criterion, passed in checks.items():
        if not passed:
            issues.append(f"{criterion} check failed")
    
    # 3. 决定是否需要修正
    if issues:
        reflection = f"Found issues: {', '.join(issues)}"
        return False, reflection
    else:
        return True, "Output passed all checks"
```

### 4.4 多 Agent 协作

**能力描述**：多个 Agent 协同完成复杂任务。

**协作模式**：
```
1. 分工协作
   - 不同 Agent 负责不同子任务
   - 并行执行提高效率
   - 最后整合结果

2. 主从模式
   - 一个主 Agent 协调
   - 多个从 Agent 执行
   - 主 Agent 负责决策

3. 对等协作
   - Agent 之间平等对话
   - 共同讨论决策
   - 集体智慧

4. 竞争协作
   - 多个 Agent 独立解决
   - 比较结果选择最优
   - 提高可靠性
```

**示例场景**：
```
任务：开发一个 Web 应用

Agent 团队：
- PM Agent：需求分析和项目管理
- Architect Agent：系统架构设计
- Frontend Agent：前端开发
- Backend Agent：后端开发
- QA Agent：测试和质量保证

协作流程：
1. PM Agent 分析需求，制定计划
2. Architect Agent 设计架构
3. Frontend 和 Backend Agent 并行开发
4. QA Agent 测试并提供反馈
5. 迭代改进直到完成
```

**Q2: ReAct 和 Plan-and-Solve 有什么区别？**

```

我的选择：
- 需要频繁调用工具：ReAct

```

## 6. 总结

### 6.1 核心要点
```
1. Agent 是自主智能系统，具备感知、推理、行动能力
2. 核心架构：感知层 → 认知层 → 行动层 + 记忆层
3. 主要范式：ReAct（边想边做）、Plan-and-Solve（先计划后执行）
4. 主流框架：LangChain、LangGraph、AutoGen
5. 核心能力：任务规划、工具使用、自我反思、多 Agent 协作
```

### 6.2 实践建议
```
1. 根据场景选择合适的设计范式
2. 从简单开始，逐步增加复杂度
3. 重视错误处理和可观测性
4. 建立完善的评估体系
5. 关注安全性和可靠性
6. 持续学习和优化
```

### 6.3 前沿方向
```
1. 自主 Agent（更高程度的自主性）
2. 多 Agent 协作系统
3. Agent 长期记忆和持续学习
4. 具身智能（Embodied AI）
5. Agent 标准化和互操作性
6. 人机协作新范式
```
