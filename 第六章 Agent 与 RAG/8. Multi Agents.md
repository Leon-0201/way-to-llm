# 8. 生产级多Agent系统架构设计指南
## 一、多Agent系统核心基础
### 1. 什么是多Agent系统
多Agent系统是由多个目标驱动的智能体通过协作完成复杂任务的系统，核心是突破单Agent的能力边界，通过分工协作完成单Agent难以处理的复杂任务。
#### 核心优势（基于Anthropic生产实践数据）
- 任务完成率大幅提升：多Agent系统比单Agent在复杂研究任务上性能提升90.2%
- 并行处理能力：多个Agent同时执行独立任务，复杂任务耗时降低90%
- 能力专精化：不同Agent专注不同领域，输出质量更高
- 突破上下文限制：分布式处理，总有效上下文容量是单Agent的数倍
#### 适用场景
✅ 开放式研究、多维度信息检索、跨系统流程自动化、多角色协作场景
✅ 任务超出单Agent上下文窗口、需要并行处理的场景
❌ 强实时协作、高度依赖上下文一致性、低价值高token消耗的场景

> 「面试重点」多Agent vs 单Agent选型决策树
> ```
> 任务复杂度低？ → 单Agent
> 任务可拆分为多个独立子任务？ → 多Agent
> 对成本敏感且任务价值低？ → 单Agent
> 需要多领域专业知识？ → 多Agent
> ```

### 2. 核心设计理念
- **单一职责原则**：每个Agent只负责一个明确的领域/任务，保持能力边界清晰
- **组合优于继承**：通过Agent组合而非继承实现复杂能力，提升复用性
- **声明式目标驱动**：定义目标而非执行步骤，让Agent自主推理执行路径
- **状态与逻辑分离**：Agent本身无状态，上下文在运行时通过InvocationContext注入

## 二、主流架构模式
### 🔹 模式1：Google ADK 层次化组合架构
Google ADK采用代码优先的设计哲学，通过`sub_agents`参数构建树状层次化架构：
- 核心机制：父Agent不需要显式调用子Agent，LLM通过读取子Agent的`description`字段自主推理任务应该委派给谁
- 典型架构分层：
  ```
  Root Agent（全局协调层）
  ├─ 业务领域Agent（业务逻辑层）
  │  ├─ 执行工具Agent（能力实现层）
  │  └─ 质量审查Agent（校验层）
  └─ 公共能力Agent（通用能力层）
  ```
- 优势：结构清晰，职责明确，适合企业内部工作流、多角色协作系统等场景
- 最佳实践：协作层级不超过3层，避免调试成本指数级上升，禁止循环依赖

### 🔹 模式2：Anthropic 编排器-工人（Orchestrator-Worker）架构
Anthropic的Research系统采用的生产级架构，适合开放式探索类任务：
- 核心角色：
  - Lead Agent（编排器）：负责分析任务、制定计划、分解子任务、分配给Subagent、汇总结果、迭代判断是否需要补充信息
  - Subagent（工人）：专注执行具体的专项任务（如特定方向的搜索、数据分析），独立完成后返回结果给编排器
- 核心流程：
  ```
  A[用户查询] --> B[Lead Agent制定研究计划]
  B --> C[并行创建多个专项Subagent]
  C --> D[Subagent独立执行搜索/分析]
  D --> E[结果汇总到Lead Agent]
  E --> F{是否需要补充信息?}
  F -->|是| C
  F -->|否| G[CitationAgent校验引用]
  G --> H[返回最终结果]
  ```
- 优势：并行处理效率高，适合广度优先探索、多源数据整合、复杂问题分析等场景，能够突破单Agent的上下文窗口限制
- 生产数据：该架构在S&P 500公司董事会成员查询这类复杂任务上，完胜单Agent系统

### 🔹 混合架构模式
两种架构可以自由组合使用：层次化结构中的每个节点内部可以采用编排器-工人模式并行处理子任务，适合超复杂大型系统、跨领域多任务平台。

## 三、单Agent设计规范
### 1. Agent核心要素（Google ADK标准）
一个完整的Agent通过以下参数定义其能力边界：
| 要素 | 作用 | 设计规范 |
|------|------|----------|
| model | 推理使用的大模型 | 根据任务复杂度选择，简单任务用小模型降低成本，复杂任务用大模型保证效果 |
| name | 唯一标识符 | kebab-case格式，语义清晰，用于日志追踪和调用 |
| instruction | 内部使命 | 明确Agent的核心职责、工作流程、输出要求，直接注入system prompt |
| description | 外部职责说明 | 给其他Agent看的能力描述，必须具体清晰，是动态委派的核心依据 |
| tools | 可调用工具 | 仅提供完成任务所需的最小工具集，避免能力冗余 |
| sub_agents | 下属子Agent | 仅包含当前Agent需要协调的子Agent，保持层级清晰 |

### 2. Agent设计最佳实践
- 单一职责：每个Agent只负责一个明确的任务，避免"超级Agent"
- 能力收敛：仅提供完成任务必需的工具和权限，减少错误决策概率
- 描述精准：`description`要包含明确的触发场景，方便父Agent准确委派
- 无状态设计：Agent不保存状态，所有状态通过上下文注入，支持多会话复用

## 四、多Agent协作模式（面试重点）
### 🔹 模式1：动态委派模式（Google ADK默认）
- 执行方式：父Agent通过LLM推理，自主决定将任务委派给哪个子Agent，无需显式调用代码
- 核心依赖：子Agent`description`字段的精准度
- 适用场景：任务边界模糊、需要动态决策分配的场景
- 示例：代码审查协调Agent根据任务内容自动委派给质量检查Agent或安全审计Agent

### 🔹 模式2：顺序流水线模式（SequentialAgent）
- 执行方式：Agent按固定顺序依次执行，前一个Agent的输出自动作为下一个的输入
- 适用场景：步骤有明确依赖关系、前序输出是后序输入的场景（如需求分析→方案设计→评审→开发）
- 设计要点：明确每个步骤的输入输出格式，设置阶段Checkpoint，不满足条件则不进入下一个阶段

### 🔹 模式3：并行执行模式（ParallelAgent）
- 执行方式：多个无依赖的Agent同时执行，所有结果返回后再继续处理
- 适用场景：任务相互独立、需要缩短总耗时的场景（如CI多检查项并行、多维度信息检索）
- 优化收益：Anthropic实践证明，并行化可将复杂研究任务的耗时降低90%
- 注意事项：避免多个Agent同时写入同一个状态key导致的竞争条件，子Agent应仅读取父Agent写入的状态，不直接修改全局状态

### 🔹 模式组合方案
模式支持嵌套组合，典型组合示例：
> 产品研发全流程 = SequentialAgent（需求分析→方案设计→开发→测试→上线）
> 每个阶段内部 = ParallelAgent（并行执行多个独立检查项）
> 如测试阶段并行执行：单元测试Agent、集成测试Agent、性能测试Agent、安全测试Agent

## 五、状态管理与上下文传递
### 1. 分层状态机制（Google ADK标准）
ADK通过状态前缀机制管理不同作用域的状态，完美解决多Agent之间的状态传递问题：
| 状态前缀 | 生命周期 | 作用域 | 典型用途 |
|----------|----------|--------|----------|
| 无前缀 | 当前会话 | 会话内所有Agent共享 | 任务进度、上传文件列表、临时结果 |
| `user:` | 跨会话 | 同一用户所有会话共享 | 用户偏好、历史设置、权限信息 |
| `app:` | 全局 | 所有用户所有会话共享 | 全局配置、功能开关、API端点 |
| `temp:` | 当前调用 | 仅单次Agent调用可见 | 中间结果、临时缓存、推理链路记录 |

> 示例：
> ```python
> session.state["progress"] = "50%"          # 当前会话可见
> session.state["user:theme"] = "dark"       # 跨会话用户偏好
> session.state["app:version"] = "1.0"       # 全局配置
> session.state["temp:intermediate"] = {...} # 调用结束自动清理
> ```

### 2. 跨Agent状态传递方案
- 子Agent自动继承父Agent的InvocationContext，可以直接读取父Agent写入的状态，无需显式传参
- 最佳实践：父Agent写入状态，子Agent仅读取，避免并行写入导致的竞争条件
- 大体积数据处理：对于文件、图片、生成报告等大内容，使用ArtifactService存储，状态中仅保存引用，避免占用会话状态空间
- 上下文溢出解决方案：Anthropic生产实践采用分段摘要+外部内存方案，超长任务执行过程中定期将完成阶段的内容摘要存入外部记忆，需要时再检索，避免上下文窗口溢出。

## 六、协作规则与Prompt设计
### 1. 编排器Prompt设计要点（基于Anthropic生产经验）
- **明确委派规则**：如何分解任务、如何给子Agent分配目标、输出格式要求、任务边界
- **资源分配规则**：根据任务复杂度动态调整子Agent数量和工具调用配额：
  - 简单事实查询：1个Agent，3~10次工具调用
  - 对比分析任务：2~4个Subagent，10~15次调用每个
  - 复杂研究任务：10+Subagent，明确分工
- **迭代终止条件**：明确什么情况下已经收集到足够信息，可以停止研究，避免无限循环
- **错误处理规则**：子Agent失败时的重试、回退、升级机制

### 2. 子Agent Prompt设计要点
- 明确任务边界：该做什么，不该做什么，不需要处理超出范围的内容
- 输出格式要求：返回结果的统一结构，方便编排器解析和汇总
- 工具使用指南：优先用什么工具，什么情况下需要调整查询策略
- 质量校验规则：结果需要满足什么标准才能返回

### 3. 防止异常行为的Guardrails
- 子Agent数量上限：避免简单任务创建过多Agent导致token浪费
- 工具调用配额：限制单Agent的最大工具调用次数，防止无限循环
- 超时机制：单Agent最大执行时间，避免卡住整个流程
- 去重机制：避免多个子Agent重复执行相同任务，提升资源利用率

## 七、评估与质量保障体系
### 1. 多Agent评估的核心挑战
- **非确定性**：相同输入可能走完全不同的执行路径，无法用固定步骤校验
- **结果开放性**：很多任务没有唯一标准答案，难以用规则自动化评估
- **Emergent行为**：小的Prompt修改可能导致不可预测的协作行为变化

### 2. 三层评估体系（Anthropic生产实践）
#### （1）小样本快速验证
- 使用20~50个典型测试用例快速验证版本效果，适合早期开发阶段
- 可以快速捕捉大的效果波动，验证核心路径是否正常，投入小见效快

#### （2）LLM-as-Judge自动化评估
- 评估维度：事实准确性、引用准确性、完整性、源质量、工具效率
- 实现方式：用大模型按照评分标准自动打分，和人工判断一致性达85%+
- 适用场景：大规模版本迭代的效果量化对比，可以批量处理数百个测试用例

#### （3）人工评估
- 覆盖边缘场景、发现自动化评估遗漏的问题（如源质量偏差、幻觉）
- 定期抽样验证自动化评估的准确性，保证评估结果可靠
- 适用场景：核心功能上线前的最终校验

### 3. 核心评估指标
| 指标类型 | 关键指标 | 参考阈值 |
|----------|----------|----------|
| 效果指标 | 任务完成率、准确率、用户满意度 | 任务完成率>85% |
| 效率指标 | 平均完成时间、token消耗量、工具调用次数 | 比单Agent耗时降低>50% |
| 成本指标 | 单任务平均成本、资源利用率 | 比单Agent成本增加不超过4倍（高价值任务可接受） |

## 八、工程落地最佳实践
### ✅ 开发岗要求
1. **项目结构规范**：按Agent领域拆分目录，每个Agent独立模块，方便维护
2. **调试工具**：使用ADK Web UI可视化查看调用链路、状态变化、工具返回结果，提升调试效率
3. **本地开发**：开启热重载功能，修改代码无需重启即可生效，提升开发速度
4. **部署方案**：容器化部署，支持水平扩缩容，应对流量波动

### 🔬 算法岗要求
1. **效果迭代**：通过A/B测试对比不同架构/提示的效果，数据驱动优化
2. **成本优化**：分级使用模型，简单任务用小模型，复杂任务用大模型，平衡效果和成本
3. **协作规则优化**：持续调整委派和分工规则，减少重复工作和遗漏，提升协作效率
4. **瓶颈分析**：定位系统瓶颈（如某个Agent耗时过长、某个工具调用失败率高），针对性优化

## 九、面试高频考点（分岗位）
### ✅ 开发岗通用题
1. **Q：多Agent系统有哪些主流架构模式？分别适用什么场景？**
   答：主流有两种架构：① Google ADK的层次化组合架构，适合企业内部工作流、多角色协作等职责明确的场景；② Anthropic的编排器-工人架构，适合开放式研究、多源信息检索等需要并行探索的场景。还有混合架构，适合超复杂大型系统。
2. **Q：Google ADK中的Agent核心要素有哪些？`instruction`和`description`有什么区别？**
   答：核心要素包括model、name、instruction、description、tools、sub_agents。区别是：instruction是给Agent自己看的执行指南，详细说明工作流程和要求；description是给其他Agent看的能力说明，清晰描述该Agent能做什么、什么时候用，是父Agent动态委派任务的核心依据。
3. **Q：多Agent协作有哪些模式？动态委派和顺序流水线各有什么优劣势？**
   答：有三种核心模式：动态委派、顺序流水线、并行执行。动态委派的优势是灵活性高，适合任务边界模糊的场景，缺点是可控性稍差；顺序流水线的优势是步骤可控，适合有明确依赖关系的流程，缺点是灵活性不足，并行度低。
4. **Q：多Agent系统中的状态如何传递？不同前缀的状态有什么区别？**
   答：子Agent自动继承父Agent的InvocationContext，可以直接读取父写入的状态。不同前缀的状态作用域不同：无前缀是当前会话可见，`user:`是跨会话用户级，`app:`是全局可见，`temp:`是仅当前调用可见，调用结束自动清理。
5. **Q：多Agent生产落地有哪些核心可靠性保障措施？**
   答：核心措施包括：分级错误处理和重试机制、断点续跑避免从头执行、全链路可观测性、彩虹部署不中断现有任务、成本控制和异常告警。

### 🔬 算法岗专项题
1. **Q：对比层次化组合架构和编排器-工人架构的优劣势，如何选型？**
   答：层次化组合架构的优势是结构清晰、职责明确、可控性强，适合结构化的企业工作流场景，缺点是并行度较低；编排器-工人架构的优势是并行度高、适合开放式探索类任务，缺点是协调复杂度高，适合研究、信息检索等场景。选型主要看任务类型：结构化流程选层次化，开放式探索选编排器-工人。
2. **Q：多Agent评估和单Agent评估有什么区别？有哪些核心挑战？**
   答：区别是多Agent系统是非确定性的，相同输入可能走不同路径，没有固定的执行步骤可以校验，且很多任务没有唯一标准答案。核心挑战包括：非确定性导致的评估难度、结果开放性难以量化、emergent行为不可预测。
3. **Q：如何设计多Agent的协作规则，避免重复工作、死循环和资源浪费？**
   答：核心方法包括：明确子Agent的任务边界避免重叠、设置子Agent数量和工具调用配额上限、定义明确的迭代终止条件、加入去重机制避免重复任务、监控异常行为及时熔断。
4. **Q：多Agent系统的效果和成本如何平衡？有哪些优化方向？**
   答：平衡方法：① 分级使用模型，简单任务用小模型，复杂任务用大模型；② 动态分配资源，根据任务复杂度调整Agent数量和调用配额；③ 并行执行提升效率，降低总耗时；④ 优化Prompt减少无效token消耗。优秀的策略可以在效果损失<5%的前提下，降低30%以上的成本。
5. **Q：多Agent系统有哪些emergent行为？如何应对这些不可预测的行为？**
   答：Emergent行为指没有明确编程但自发出现的行为，比如子Agent之间重复工作、过度创建子Agent、死循环讨论等。应对方法：加入明确的Guardrails限制异常行为、完善可观测性及时发现异常、持续迭代Prompt和协作规则、通过测试覆盖常见的异常场景。
