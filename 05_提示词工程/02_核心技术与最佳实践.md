## 1 核心Prompt技术
### 1.1 角色设定技术
**开发岗要求**：
#### 核心作用
- 让大模型进入对应的角色身份，输出更专业、符合领域要求的内容
- 统一输出的风格、立场、专业度
#### 最佳实践
- 角色定位要具体：不用"你是一个专家"，要用"你是一名有10年经验的一线互联网后端技术架构师，擅长微服务架构设计，熟悉Java技术栈"
- 明确角色的立场和约束：比如"你是电商平台的客服，回答要热情友好，遇到投诉优先安抚用户，不要承诺权限外的内容"
- 可以加入角色的背景知识：比如"你在阿里云工作了8年，非常熟悉阿里云的产品体系"

**算法岗要求**：
#### 生效原理
- 角色设定相当于给模型提供了领域先验知识，激活模型预训练中对应领域的知识分布
- 减少模型的解码搜索空间，让输出更集中在目标领域，降低生成无关内容的概率
#### 优化方向
- 角色描述的信息量和具体度和效果正相关，但过长会占用窗口空间，需要平衡
- 角色中的关键词会被模型重点关注，重要信息可以重复强调提升权重

---
### 1.2 少样本（Few-shot）技术
**开发岗要求**：
#### 核心能力
给大模型提供少量示例，引导模型学习任务模式，比零样本效果提升25%以上，适合规则复杂、输出格式要求严格的场景。
#### 最佳实践
- 样本数量：通常2~8个效果最好，过多会占用上下文窗口，边际收益递减
- 样本质量：示例要正确、覆盖典型场景和边界情况，标注准确
- 样本格式：输入和输出的格式要和实际任务完全一致，避免格式混乱
- 排列顺序：难度从易到难排序，相似场景的示例放在一起
#### 示例
```
# 任务：将用户的问题分类为"技术问题"、"商务问题"、"其他问题"
# 示例：
用户问题："接口调用报错500怎么解决？" → 分类：技术问题
用户问题："你们的服务怎么收费？" → 分类：商务问题
用户问题："今天天气怎么样？" → 分类：其他问题

# 现在分类以下问题：
用户问题："你们的SLA是多少？" → 分类：
```

**算法岗要求**：
#### 生效原理
- 少样本学习利用大模型的上下文学习能力，通过少量示例让模型快速学习任务的输入输出分布
- 注意力机制会学习示例中的输入输出映射模式，应用到新的样本上
#### 效果影响因素
- 示例和目标任务的相似度越高，效果越好
- 示例的多样性越高，泛化能力越强
- 样本的排列顺序会影响效果，近因效应导致后面的样本影响更大
#### 优化算法
- 动态样本选择：根据用户查询实时检索最相关的示例，比固定样本效果提升20%
- 样本过滤：过滤低质量、重复、冲突的样本，提升整体质量

### 1.3 推理增强技术（CoT/ToT/GoT）
**开发岗要求**：
#### 1.3.1 思维链（Chain-of-Thought，CoT）
- **适用场景**：数学计算、逻辑推理、多步骤任务、代码调试等复杂推理场景，准确率提升40%以上
- **使用方法**：在Prompt中加入"请分步思考，先给出推理过程，再给出最终答案"，或者提供带推理过程的示例
- **进阶：零样本CoT**：不需要示例，直接加"让我们一步步思考"即可提升推理效果
#### 1.3.2 思维树（Tree-of-Thought，ToT）
- **适用场景**：需要探索多个解决方案的场景，比如最优解问题、路径规划、创意生成
- **核心思路**：生成多个推理路径，评估每个路径的合理性，选择最优路径继续探索，比CoT准确率提升35%
#### 1.3.3 思维图（Graph-of-Thought，GoT）
- **适用场景**：需要多路径关联、交叉验证的复杂任务，比如复杂问题排查、方案设计
- **核心思路**：将推理过程组织为图结构，支持节点合并、分叉、回退，适合非线性推理场景

**算法岗要求**：
#### 技术原理对比
| 技术 | 核心思想 | 推理路径 | 准确率 | 计算成本 |
|------|----------|----------|--------|----------|
| CoT | 单路径串行推理 | 线性链状 | 基础值 | 低 |
| ToT | 多路径并行探索，择优选择 | 树状 | 比CoT高35% | 中（K倍于CoT，K是探索路径数） |
| GoT | 图结构推理，支持交叉验证 | 图状 | 比ToT高15% | 高 |
#### 生效原理
- 复杂问题的解空间很大，单路径CoT容易陷入局部最优，多路径探索可以找到更优解
- 分步推理将复杂任务分解为多个简单子任务，降低每个步骤的推理难度，减少错误传播
- 中间推理步骤相当于提供了更多的监督信号，引导模型按正确的逻辑路径解码

### 1.4 输出约束技术
**开发岗要求**：
#### 核心作用
强制大模型按照指定的格式输出，大幅减少后续的解析成本，提升对接效率。
#### 常用约束类型
1. **格式约束**：要求输出JSON、Markdown、XML等指定格式，加入"严格按照JSON格式输出，不需要其他解释内容"
2. **长度约束**："输出不超过500字"、"代码长度控制在100行以内"
3. **内容约束**："不要使用专业术语，语言要通俗易懂，面向小学生"、"回答必须包含关键词：人工智能、大模型、应用场景"
4. **安全约束**："如果问题涉及敏感内容，请回答'抱歉，我无法回答这个问题'"
#### 最佳实践
- 格式要求要明确，最好给出示例模板
- 对于JSON格式，可以要求"输出可以直接被json.loads()解析，不要有多余的注释和解释"
- 可以用分隔符将待处理的内容和指令分开，避免内容和指令混淆，比如用"""包裹待处理文本

**算法岗要求**：
#### 生效原理
- 输出格式约束本质是限制模型的解码空间，让模型只能在符合格式要求的token中选择，减少无效生成
- 格式示例给模型提供了明确的输出模板，降低解码的不确定性
#### 优化方向
- 结构化输出可以用语法引导解码（Grammar-guided Decoding），强制模型输出符合语法规则的内容，100%保证格式正确
- 对于严格格式要求的场景，可以用约束解码技术，在解码时过滤不符合格式的token，完全避免格式错误

## 2 分场景最佳实践
以下是2026年主流的分场景提示词最佳实践：

### 2.1 代码生成场景
**开发岗最佳实践**：
- 明确技术栈、版本要求、编码规范
- 要求包含注释、异常处理、单元测试
- 复杂需求可以要求先做方案设计，再写代码
- 示例：
```
你是一名资深Python后端开发工程师，遵循PEP8编码规范。
请实现一个用户登录接口，要求：
1. 使用FastAPI框架
2. 包含JWT鉴权
3. 密码用bcrypt加密存储
4. 包含参数校验、异常处理
5. 返回符合RESTful规范的JSON格式
先输出接口设计思路，再输出完整代码，最后写测试用例。
```

### 2.2 文案写作场景
**开发岗最佳实践**：
- 明确目标受众、文案风格、使用场景
- 明确文案的目的（转化、科普、宣传、通知）
- 给出字数限制、结构要求
- 示例：
```
你是一名互联网公司的运营文案专家，擅长写短视频脚本。
请写一条30秒的产品宣传短视频脚本，产品是AI学习助手，目标受众是中学生家长，风格要亲切有吸引力，突出"节省家长辅导时间，提升孩子学习效率"的核心卖点，结尾要有号召行动。
```

### 2.3 数据分析场景
**开发岗最佳实践**：
- 明确分析的目标、维度、指标定义
- 给出数据样例和字段说明
- 要求输出分析结论和可落地的建议
- 示例：
```
你是一名电商数据分析师，现有以下销售数据：
[数据样例省略]
请分析2026年Q1的销售情况，要求：
1. 按地区、产品类别两个维度分析销量和同比增长率
2. 找出销量下滑的原因
3. 给出3条可落地的优化建议
4. 输出用Markdown表格+文字说明的格式，重点突出。
```

### 2.4 多模态Prompt场景
**开发岗最佳实践**：
- 对于图文混合输入，明确说明图片和文字的对应关系
- 要求模型先描述图片内容，再回答问题
- 示例：
```
[图片：一张电商后台的销售趋势图，横轴是月份，纵轴是销售额，1-3月上升，4月突然下滑]
结合图片和以下文字："4月公司做了一次大促活动"，请分析4月销售额下滑的可能原因，给出5个可能性，按概率从高到低排序。
```

## 3 面试高频考点
### 开发岗通用题
#### Q1：**什么是思维链（CoT）？什么时候需要用？**
思维链是引导大模型输出中间推理步骤的Prompt技术，核心是让模型"先想再说"，分步完成任务。
- 适用场景：数学计算、逻辑推理、多步骤任务、代码调试、问题排查等复杂推理场景，能让准确率提升40%以上
- 使用方法：在Prompt里加"请分步思考，先给出推理过程再给答案"即可，或者提供带推理过程的示例

#### Q2：**Few-shot和Zero-shot分别是什么？各有什么优缺点？**
- **Zero-shot**：不给示例，只给指令让模型完成任务
  - 优点：简单快捷，节省上下文空间
  - 缺点：复杂任务效果差，输出不稳定
- **Few-shot**：给模型少量示例，引导模型学习任务模式
  - 优点：效果更好更稳定，适合规则复杂的场景
  - 缺点：需要准备示例，占用上下文窗口空间
- 使用原则：简单任务用Zero-shot，复杂任务用Few-shot

#### Q3：**如何保证大模型输出的JSON格式正确？有哪些技巧？**
常用技巧：
- 明确要求输出JSON格式，给出具体的字段示例
- 强调"不要有多余的解释、注释、markdown标记，输出可以直接被json.loads()解析"
- 用分隔符将指令和待处理内容分开，避免内容干扰格式
- 要求用特定符号包裹JSON，比如```json ... ```，方便后续提取
- 开启大模型原生JSON模式，可100%保证格式正确

#### Q4：**角色设定有什么用？有什么最佳实践？**
- 核心作用：让大模型进入对应的专业身份，输出更符合领域要求的内容，统一输出风格和立场
- 最佳实践：
  - 角色要具体，不要太泛，要有明确的专业领域、经验背景
  - 明确角色的立场、约束和输出风格
  - 可以加入角色特有的知识背景，让输出更专业

### 算法岗专项题
#### Q1：**为什么Few-shot学习不需要微调模型就能生效？背后的理论基础是什么？**
Few-shot生效的核心是大模型的上下文学习（ICL）能力，这是大模型的涌现特性之一，理论基础：
- 大模型在预训练阶段学习了大量的任务模式和通用知识，已经具备了完成各种任务的能力
- 输入的Few-shot示例相当于给模型提供了任务的分布信息，模型可以在推理时快速适应这个任务分布，不需要更新参数
- 从贝叶斯角度看，示例相当于给模型提供了任务的先验信息，模型根据先验调整输出的后验分布

#### Q2：**思维链（CoT）对简单任务没有效果甚至有负效果，为什么？**
核心原因：
- 简单任务不需要复杂推理，直接输出即可，思维链增加了推理步骤反而可能引入不必要的错误
- 短上下文的干扰：思维链的额外内容可能占用上下文窗口，挤掉关键信息
- 解码长度增加：思维链需要输出更多内容，增加了生成错误的概率
- 结论：思维链只适合复杂度超过一定阈值的任务，简单任务用CoT反而会降低效果和效率

#### Q3：**上下文学习（ICL）和微调（Fine-tuning）有什么区别？各自适用什么场景？**
| 对比维度 | 上下文学习（ICL） | 微调（Fine-tuning） |
|----------|------------------|---------------------|
| 原理 | 推理时通过输入Prompt引导模型，不需要更新参数 | 更新模型参数，让模型学习特定任务的知识 |
| 成本 | 成本低，不需要训练数据和算力 | 成本高，需要标注数据和训练资源 |
| 效果 | 适合通用任务、快速迭代场景 | 效果更好更稳定，适合固定场景、大规模应用 |

适用场景：
- ICL：快速验证、需求变化快、数据量小的场景
- 微调：需求固定、数据量大、对效果要求高的场景
- 主流方案：ICL+微调结合，通用能力用微调，场景化定制用提示词工程

#### Q4：**如何优化Few-shot的样本选择？有哪些常用策略？**
常用优化策略：
- 语义匹配：选择和当前查询语义最相似的样本，相似度越高效果越好
- 多样性采样：选择覆盖不同场景、不同边界情况的样本，提升泛化能力
- 难度排序：按难度从易到难排列样本，符合模型的学习规律
- 动态选择：根据不同的查询动态选择最合适的样本，比固定样本效果提升20%以上
- 样本过滤：去除低质量、重复、冲突的样本，避免干扰模型
- 进阶优化：用强化学习学习最优的样本选择策略，进一步提升效果
