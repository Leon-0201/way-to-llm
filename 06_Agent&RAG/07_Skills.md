## 1 Skill 核心基础
### 1.1 什么是Skill
Skill是可复用的标准化工作流模板，将领域知识、操作流程、最佳实践固化为Agent可以直接调用的能力单元，实现"一次编写，多次复用"，彻底解决重复性任务需要反复输入相同prompt的痛点。
#### 核心价值
- 提升效率：重复性任务不用反复解释规则，直接调用Skill即可完成
- 保证一致性：相同类型的任务输出格式、质量、流程完全统一
- 降低门槛：普通用户不需要掌握专业领域知识，也能通过Skill完成复杂任务
#### 「面试重点」Skill vs Prompt vs MCP 区别与联系
| 类型 | 定位 | 复用性 | 适用场景 | 关系 |
|------|------|--------|----------|------|
| Prompt | 单次任务指令 | 低 | 临时、个性化任务 | Skill是标准化的Prompt集合 |
| Skill | 可复用工作流模板 | 高 | 标准化、重复性任务 | Skill调用MCP工具完成操作 |
| MCP | 工具/服务的统一接口 | 中 | 提供底层能力支持 | MCP是厨具食材，Skill是菜谱，Prompt是单次做菜的临时指令 |

### 1.2 2026行业通用标准结构
```
your-skill-name/               # 文件夹名必须用kebab-case，不能有空格或大写
├── SKILL.md                   # 必须，大小写敏感，严格使用SKILL.md，不能是skill.md/SKILL.MD
├── scripts/                   # 可选，可执行脚本（Python、Bash等）
│   ├── validate_data.py
│   └── deploy.sh
├── references/                # 可选，扩展文档、指南、参考资料
│   ├── api-spec.md
│   └── style-guide.md
└── assets/                    # 可选，模板、Schema、静态资源
    └── report-template.md
```
> 注意：不要放README.md，所有文档统一放进SKILL.md或references/目录

### 1.3 渐进式披露核心设计
Skill的内容不是一次性全部加载，而是三层按需加载，完美平衡能力和token消耗：
| 层级 | 内容 | 加载时机 | 作用 |
|------|------|----------|------|
| L1 | YAML frontmatter元数据 | Agent启动时加载 | 帮助Agent判断"我该不该用这个Skill" |
| L2 | SKILL.md正文 | 确定使用Skill时才加载 | 包含完整的执行指令和流程 |
| L3 | references/、assets/下的资源 | 正文中引用到时才加载 | 按需扩展，不会浪费不必要的token |

#### 1.3.1 核心元数据字段（SKILL.md头部）
```yaml
---
name: project-initialization
description: 初始化新项目工作区，包括代码仓库创建、CI/CD配置、项目模板生成。当用户提到"初始化项目"、"创建新项目"、"项目脚手架"时使用。
author: your-name
version: 1.0.0
tags: [项目管理, 研发效能]
---
```
> 「面试重点」`description`是最关键的字段，必须同时包含**做什么**和**触发时机**：
> 好的示例：`管理项目迭代工作流，包括迭代规划、任务创建和进度追踪。当用户提到"迭代规划"、"创建任务"、"项目排期"或"冲刺计划"时使用。`
> 差的示例：`帮助处理项目。`

## 2 Skill 核心分类与适用场景
### 2.1 三类核心用途
| 类型 | 说明 | 适用场景 | 核心技巧 |
|------|------|----------|----------|
| 文档资产创建类 | 生成格式一致的标准化输出 | 需求文档、代码片段、报表、设计稿生成 | 嵌入风格指南、提供模板、增加质量检查清单 |
| 工作流自动化类 | 编排多步骤流程 | 客户入驻、项目初始化、版本发布流程 | 分步骤设置验证门槛、内置迭代优化循环 |
| MCP增强类 | 叠加领域知识的工具封装 | SDK调用、合规检查、多服务协同 | 按顺序协调多个MCP调用、嵌入领域专业知识、完善错误处理 |

### 2.2 主流Skill生态
- **官方生态**：Anthropic Skill Store、Google ADK Skill Hub、字节Coze插件市场、OpenAI GPTs商店
- **开源生态**：OpenCLAUDE Skill仓库、社区贡献Skill集市
- **企业级**：内部Skill市场、细粒度权限管控、全链路审计能力

## 3 Skill 设计模式（面试重点）
### 3.1 基础模式（Google ADK 5大标准模式）
#### 3.1.1 Tool Wrapper（工具封装）
**解决的问题**：Agent调用SDK/API时容易写出过时用法、参数错误、违反最佳实践

**核心逻辑**：将库的API约定、最佳实践、常见坑点封装成Skill，Agent只在需要时加载，不会浪费上下文

**示例（Stripe SDK封装）**：
```markdown
## 使用规则
- 所有金额单位必须是分（cents），不是元
- 创建Customer时必须先检查email是否已存在
- webhook验证必须用stripe.webhooks.constructEvent()

## API 速查
### 创建支付意图
python
import stripe
stripe.api_key = os.environ["STRIPE_KEY"]
intent = stripe.PaymentIntent.create(
    amount=2000,  # $20.00，单位是分
    currency="usd",
    payment_method_types=["card"],
)
```
**适用场景**：Agent需要调用特定SDK/API/工具的场景

#### 3.1.2 Generator（生成器）
**解决的问题**：生成类任务输出格式不一致，每次都需要重新说明格式要求

**核心逻辑**：模板+风格指南+变量填充，用assets/放模板，references/放风格指南，指令负责流程控制

**目录结构示例**：
```
spec-generator/
├── SKILL.md
├── assets/
│   └── spec-template.md    # 文档模板（带占位符）
└── references/
    └── style-guide.md      # 写作规范
```
**核心指令**：
```markdown
## 流程
1. 读取 assets/spec-template.md 获取文档结构
2. 读取 references/style-guide.md 获取写作规范
3. 向用户确认以下变量：项目名、目标用户、核心功能列表
4. 按模板填充，严格遵循风格指南
5. 输出完整文档
```
**适用场景**：文档生成、代码生成、报表生成等需要标准化输出的场景

#### 3.1.3 Reviewer（审查器）
**解决的问题**：审查类任务标准不统一、漏检关键问题、不同场景审查尺度不一致

**核心逻辑**：结构化Checklist+分级输出，将评审标准从Agent的"感觉"变成可维护的配置

**示例（代码审查Skill）**：
```markdown
## 审查清单
### 安全（严重）
- [ ] 是否有SQL注入风险？
- [ ] 用户输入是否做了转义？
- [ ] 是否有硬编码的密钥或密码？

### 性能（中等）
- [ ] 是否有N+1查询？
- [ ] 大列表是否做了分页？

### 规范（低）
- [ ] 函数命名是否符合项目约定？
- [ ] 是否有未使用的导入？

## 输出格式
对每条不通过的项，输出：
- 严重等级：严重 / 中等 / 低
- 位置：文件名:行号
- 问题描述
- 修复建议
```
**适用场景**：代码审查、内容审核、合规检查、质量校验场景

#### 3.1.4 Inversion（反转型）
**解决的问题**：需求不明确导致Agent瞎猜、生成不符合预期的结果，需要反复返工

**核心逻辑**：翻转控制流，Agent先当采访者，按结构化问题收集完所有必要信息，再开始执行

**示例（API开发Skill）**：
```markdown
## 执行规则（不可跳过）
在编写任何代码之前，你必须完成以下采访：
### 第一阶段：技术栈确认
1. 使用什么语言和框架？
2. 目标运行环境？（本地 / Docker / 云服务）
3. 是否有现有的项目结构？

### 第二阶段：需求细化
4. 核心功能列表（按优先级排序）
5. 认证方式？（JWT / OAuth / API Key）
6. 数据存储方案？

### 第三阶段：约束条件
7. 性能要求？（QPS / 响应时间）
8. 是否需要向后兼容？
9. 截止日期？

在所有问题回答完毕之前，禁止生成任何实现代码。
```
**适用场景**：需求模糊、参数多、约束复杂的场景

#### 3.1.5 Pipeline（流水线）
**解决的问题**：复杂任务容易跳步骤、漏关键环节、质量不可控

**核心逻辑**：将工作流拆成严格的顺序步骤，每步之间设置Checkpoint质量门禁，必须按顺序执行

**示例（竞品分析Skill）**：
```markdown
## 工作流（严格按顺序执行）
### Step 1: 数据收集
- 搜索竞品的5个关键指标
- checkpoint：确认已收集至少3个竞品的数据

### Step 2: 分析对比
- 制作对比矩阵
- checkpoint：表格至少包含5个维度

### Step 3: 草拟方案
- 基于分析结果写推荐方案
- checkpoint：方案必须包含可行性评估和风险点

### Step 4: 自检
- 按照 references/review-checklist.md 自检
- checkpoint：所有"严重"级别项必须通过
```
> 模式支持嵌套：比如上述Pipeline最后一步嵌套了Reviewer模式
**适用场景**：多步骤复杂工作流、需要质量管控的场景

### 3.2 进阶模式（Anthropic 实战模式）
#### 3.2.1 多MCP协调模式
**解决的问题**：跨多个服务的工作流需要手动协调数据传递，用户需要记住不同服务的调用顺序

**核心逻辑**：按阶段划分不同MCP的操作，明确数据依赖和流转，自动完成跨服务协同

**示例（设计稿交付Skill）**：
```markdown
## 工作流
### 阶段一：设计导出（Figma MCP）
1. 从Figma导出设计资产
2. 生成设计规格说明
3. 创建资产清单

### 阶段二：资产存储（Drive MCP）
4. 在Drive创建项目文件夹
5. 上传所有资产
6. 生成可分享链接

### 阶段三：任务创建（Linear MCP）
7. 创建开发任务
8. 将资产链接附加到任务
9. 分配给工程团队

### 阶段四：通知（Slack MCP）
10. 在#engineering频道发布交付摘要
11. 包含资产链接和任务引用
```
**适用场景**：设计稿交付、跨系统数据同步等涉及多个服务的场景

#### 3.2.2 迭代精炼模式
**解决的问题**：输出质量需要多轮迭代才能达标，单次生成无法满足要求

**核心逻辑**：生成→检查→修复的循环，明确终止条件，自动迭代直到符合标准

**适用场景**：报告生成、文案优化、代码重构等需要多次打磨的场景

#### 3.2.3 上下文感知工具选择模式
**解决的问题**：同一目标需要根据不同情况选择不同工具，用户需要自己判断用什么工具

**核心逻辑**：内置决策树，根据上下文自动选择最优工具

**示例（智能文件存储Skill）**：
```markdown
## 决策树
1. 检查文件类型和大小
2. 选择最合适的存储位置：
   - 大文件（>10MB）：用云存储MCP
   - 协作文档：用Notion/Docs MCP
   - 代码文件：用GitHub MCP
   - 临时文件：用本地存储

## 执行存储
根据决策结果：
- 调用相应的MCP工具
- 应用该服务特定的元数据
- 生成访问链接
```
**适用场景**：智能文件存储、请求路由、多模型调度等场景

#### 3.2.4 领域专业智能模式
**解决的问题**：需要专业领域知识才能完成的任务，普通Agent缺乏相关知识

**核心逻辑**：将行业规则、合规要求、专业知识直接编码到Skill中，不需要用户具备专业知识

**适用场景**：财务处理、合规检查、医疗咨询、法律审查等专业领域场景

### 3.3 模式组合方法
模式可以自由嵌套和组合，常见组合示例：
- **技术方案生成**：`Inversion（收集需求）→ Generator（填充方案模板）→ Reviewer（自检质量）`
- **代码开发**：`Tool Wrapper（SDK封装）→ Generator（代码生成）→ Reviewer（代码审查）`
- **发布流程**：`Pipeline（多步骤）+ 多个Tool Wrapper（不同环节的工具封装）`

#### 3.3.1 模式选择决策树
```
开始
├─ 需要领域知识？ → Tool Wrapper
├─ 输出需要结构一致？ → Generator
├─ 需要质量检查？ → Reviewer
├─ 需求不明确？ → Inversion
└─ 多步骤流程？ → Pipeline
```
> 大多数真实场景会组合2-3个模式使用

## 4 Skill 开发最佳实践
### 4.1 开发岗要求
#### 4.1.1 标准开发流程
1. **需求分析**：明确Skill的目标用户、适用场景、核心价值
2. **模式选择**：根据场景选择合适的设计模式或组合
3. **开发实现**：编写SKILL.md、资源文件、脚本
4. **测试验证**：触发测试、功能测试、性能对比
5. **发布分发**：上传到Skill市场或内部仓库

#### 4.1.2 编写规范
- 关键步骤用要点列表突出，避免大段文字淹没重要信息
- SKILL.md正文超过5000字的内容移到`references/`目录按需加载
- 明确错误处理和回滚逻辑，不要让流程卡在错误状态
- 复杂流程尽量拆分小步骤，每个步骤只做一件事

#### 4.1.3 测试验证三要素
1. **触发测试**：准备一批测试查询，验证Skill在该触发时触发，不该触发时不触发
2. **功能测试**：验证输出格式、工具调用、错误处理符合预期
3. **性能对比**：对比使用Skill前后的对话轮次、token消耗、任务完成率，验证优化效果

#### 4.1.4 常见问题排查
| 问题 | 原因 | 解决方案 |
|------|------|----------|
| Skill不触发 | description写得太模糊，缺少关键词 | 在description里加入用户实际会说的触发词，比如"当用户说'帮我规划冲刺'时使用" |
| 触发过度 | 适用范围太广，不相关场景也触发 | 增加负向触发条件，比如"不适用于简单数据探索场景" |
| 不按指令执行 | 指令太长太模糊，关键步骤被淹没 | 简化指令，用要点列表突出关键步骤，拆分过长内容到references/ |
| MCP调用失败 | 工具名或参数错误 | 检查工具名大小写、参数是否和MCP文档完全一致，先单独测试MCP是否正常 |

### 4.2 算法岗要求
#### 4.2.1 架构设计原则
- **单一职责原则**：每个Skill只做一件事，避免职责过多导致逻辑复杂
- **渐进式加载原则**：尽量减少常驻token消耗，非必要内容放在引用资源中
- **可组合性原则**：设计时考虑和其他Skill的组合使用，避免耦合
- **容错性原则**：明确错误处理逻辑，避免单个步骤失败导致整个流程崩溃

#### 4.2.2 优化方向
- **触发准确率优化**：基于用户反馈不断优化description的关键词，减少误触发和漏触发
- **流程效率优化**：减少不必要的步骤，并行执行无依赖的操作，降低平均完成轮次
- **质量控制优化**：增加自动化验证步骤，减少人工干预，提升一次通过率

#### 4.2.3 企业级Skill平台设计核心能力
- **全生命周期管理**：开发、测试、发布、下线全流程管控
- **权限控制**：不同角色的Skill使用、编辑、发布权限
- **统计分析**：Skill的调用量、成功率、用户满意度、平均耗时等指标监控
- **版本管理**：支持多版本并存、灰度发布、快速回滚
- **安全审计**：敏感Skill的使用审计、合规检查

## 5 面试高频考点
### 开发岗通用题
#### Q1：**什么是Skill？它和Prompt、MCP有什么区别和联系？**
Skill是可复用的标准化工作流模板，将领域知识和最佳实践固化为Agent可以直接调用的能力单元。
- 和Prompt的区别：Prompt是单次任务指令，复用性低；Skill是标准化的Prompt集合，复用性高
- 和MCP的区别：MCP是工具/服务的统一接口，提供底层能力；Skill是使用这些能力的最佳实践，两者是配合关系，类似菜谱和厨具食材的关系

#### Q2：**Skill的标准文件结构是什么？渐进式披露设计有什么优势？**
标准结构是SKILL.md（必须）+ scripts/、references/、assets/（可选）。
渐进式披露是三层按需加载：
- L1元数据：Agent启动时加载，用于判断是否需要使用Skill
- L2正文：确定使用Skill时才加载，包含完整执行指令
- L3资源：正文中引用到时才加载，按需扩展
优势是不管启用多少Skill，常驻token消耗始终可控，完美平衡了能力和成本。

#### Q3：**Skill有哪些常见设计模式？分别适用什么场景？**
基础模式有5种：
- Tool Wrapper：适合SDK/API调用场景，封装最佳实践
- Generator：适合标准化生成场景，保证输出格式一致
- Reviewer：适合审查校验场景，统一评审标准
- Inversion：适合需求不明确的场景，先收集信息再执行
- Pipeline：适合多步骤复杂工作流场景，严格控制执行流程
进阶模式有多MCP协调、迭代精炼、上下文感知选择、领域专业智能等，适合更复杂的业务场景。

#### Q4：**开发Skill时如何保证触发准确率？有哪些常见问题？**
核心是写好`description`字段，必须同时包含做什么和触发时机，加入用户实际会说的关键词。
常见问题及解决方案：
- 不触发：优化description，加入用户实际使用的触发词
- 触发过度：增加负向触发条件，明确不适用的场景
- 不按指令执行：简化指令，用要点列表突出关键步骤，拆分过长内容到references/
- MCP调用失败：检查工具名大小写、参数是否和MCP文档完全一致，先单独测试MCP是否正常

#### Q5：**Skill和MCP的关系是什么？两者如何配合使用？**
- MCP是大模型和外部工具的统一连接协议，提供工具访问能力，解决"能做什么"的问题
- Skill是固化的工作流，告诉Agent如何使用这些工具完成任务，解决"怎么做才是对的"的问题
两者是互补关系，配合使用可以让Agent可靠地完成复杂任务，避免错误的工具调用和流程执行。

### 算法岗专项题
#### Q1：**Skill的渐进式披露设计的底层逻辑是什么？如何平衡能力和token消耗？**
底层逻辑是根据使用阶段按需加载内容，避免一次性加载所有内容浪费token。
平衡方法：
- L1元数据：只放判断是否需要使用Skill的最小信息，常驻内存，token消耗极低
- L2正文：只放执行Skill必须的核心指令，触发时才加载，不会占用常驻内存
- L3资源：辅助参考资料放在references/、assets/目录，引用时才加载，非必要不加载
这样既保证了Skill的能力完整性，又将token消耗降到最低，平衡了能力和成本。

#### Q2：**如何设计一个企业级Skill平台？需要考虑哪些核心能力？**
核心能力包括：
- 全生命周期管理：支持Skill的开发、测试、发布、下线全流程管控
- 权限控制：细粒度的角色权限管理，区分使用、编辑、发布等不同权限
- 统计分析：监控Skill的调用量、成功率、用户满意度、平均耗时等核心指标
- 版本管理：支持多版本并存、灰度发布、快速回滚能力
- 安全审计：敏感Skill的使用审计、合规检查，满足企业安全要求
还要考虑和现有系统的集成，比如和企业的MCP平台、Agent调度系统、权限系统打通。

#### Q3：**多Skill组合的工作流调度有哪些核心难点？如何解决？**
核心难点及解决方案：
- 状态管理：多个Skill之间的状态传递和共享问题，采用分布式状态存储，支持跨Skill状态读写
- 错误处理：单个Skill失败时的回滚和重试问题，实现事务性执行、重试策略、降级机制
- 执行顺序：有依赖的Skill的调度顺序问题，通过依赖分析自动生成执行DAG，按顺序调度
- 性能优化：并行执行无依赖的Skill，提升整体执行效率
解决方案是采用流水线调度架构，内置以上能力，支持复杂工作流的可靠执行。

#### Q4：**Skill的效果如何评估？有哪些核心指标？**
从三个维度评估：
- 触发效果：触发准确率、误触发率、漏触发率，衡量Skill被正确调用的能力
- 执行效果：任务完成率、一次通过率、用户满意度，衡量Skill的执行质量
- 性能指标：平均完成轮次、token消耗、平均耗时，衡量Skill的执行效率
可以通过A/B测试对比使用Skill前后的效果变化，验证优化效果，持续迭代提升。

#### Q5：**对比Anthropic和Google的Skill设计模式，各自有什么优劣势？**
- Google的5大基础模式：更通用，覆盖了大多数常见场景，模式之间正交性好，职责清晰，容易组合使用，适合解决通用问题
- Anthropic的进阶模式：更偏向实际业务场景，解决更复杂的多服务协同、领域知识落地、智能决策等问题，适合解决复杂业务问题
两者是互补关系，实际开发中通常结合使用：基础模式解决通用问题，进阶模式解决复杂业务问题，灵活组合适配不同场景需求。
