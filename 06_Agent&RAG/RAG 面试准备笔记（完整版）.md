# RAG 面试准备笔记（完整版）

核心框架：RAG（检索增强生成）全流程 = 文档入库预处理 → 数据清洗 → 文本切片 → 向量化\&amp;向量库存储 → 用户Query处理 → 混合检索 → 重排序 → LLM生成\&amp;幻觉抑制，核心目标是解决大模型幻觉、提升回答精准度，落地性强，面试重点围绕“流程细节\+技术选型\+优化方案\+工程落地”展开。

## 一、文档入库（知识入库）—— 数据输入核心

核心：将不同格式文档转化为可处理的文本，重点解决“解析精度”和“成本平衡”问题

- 结构化文档（Excel、Word、MD）：通过对应代码库直接读取，无需额外处理，保证数据完整性。

- PDF文档：

    - 核心问题：原生解析易出现乱码、排版错乱、页眉页脚、水印、跨页断行，影响读取质量。

    - 解决方案：优先用专业PDF解析器（PyMuPDF、Pdfplumber）精准提取版面和文本，大模型仅用于“排版规整、断句修复、无用信息过滤”。

- 图片数据（图文混合）：两种方案按需选择，兼顾精度与成本

    - 高精度方案：多模态大模型（GPT\-4V、LLaVA、Qwen\-VL），端到端图文理解，可实现OCR文字识别\+版面布局\+表格/公式解析一体化，还能感知图文逻辑。

    - 低成本方案：版面分析模型（布局检测）\+ 通用OCR，拆分任务降低成本，缺点是仅能提取文字，无法理解图文关联，准确度稍逊。

## 二、数据清洗——RAG的生命线

核心：提升数据质量，避免低质数据导致检索偏差和模型幻觉，是RAG落地的基础

- 基础清洗：去除乱码、HTML/CSS标记、页眉页脚、水印等无关冗余内容。

- 核心操作：数据去重（删除重复信息）、低质过滤（删除无法识别、语义混乱的文本）。

- 面试加分点：强调“数据质量是RAG的核心瓶颈”，清洗不到位会直接影响后续检索和生成效果。

## 三、文本切片（数据分片）—— 平衡检索精度与上下文完整性

核心：将长文本拆分为合适的语义单元（chunk），同时留存元数据，避免“检索不准”或“上下文断裂”

- 核心切片方式：

    - 基础切片：按章节、段落、所属部门等元数据解析，适配结构化较强的文档。

    - 层次化切片：以自然段落为基础，兼顾整体与局部。

    - 固定长度切片：采用“固定token滑动窗口\+边界补全”，避免一句话被截断，核心参数为chunk\_size（切片长度）、chunk\_overlap（避免切片边界信息丢失，提升检索连贯性；同时需关注上下文窗口限制，避免切片过长导致超出LLM上下文窗口，需结合chunk\_size合理控制。）。

    - 弹性长度切片：以句号、分号等标点为边界，保证切片内语义完整。

    - 语义切片（成本较高，适合会议纪要等场景）：

        - 让大模型按语义单元切割，精度最高。

        - 每句做Embedding，当相似度突然变低了，证明语义变化可以切割了。

- 父子索引：大父块存储完整上下文，小子块用于向量检索；检索到子块后，拼接对应父块内容，平衡“检索精准度”和“上下文完整性”。

- 必备操作：切片时同步保存元数据（文档来源、时间、部门、类型等），用于后续检索过滤，提升效率。

## 四、向量化与向量库存储——检索的核心底座

- 向量化：使用Embedding模型将切片后的文本转化为向量（核心：将语义转化为可计算的数值）

    - 开源模型：BGE\-M3（核心优势：多语言、稀疏\+稠密\+多粒度统一）、Qwen\-Embedding、Sentence\-BERT。

    - 闭源模型：OpenAI Embedding、智谱Embedding、讯飞Embedding（适合对精度要求高、预算充足的场景）。

- 向量库选型：

    - Milvus：高并发、支持亿级向量存储，多种索引类型，工业最常用，原生支持“向量\+BM25混合检索\+RRF打分”；

    - Qdrant：轻量、易部署，适合中小规模向量存储，支持混合检索。

- 向量索引（核心原理，纠正原错误）：

    - IVF索引：基于K\-Means聚类，先将向量聚类，再进行倒排检索，速度快、召回率略低，适合海量数据场景。

    - HNSW索引：基于层次化近邻图结构，召回率更高、速度适中，是目前工业主流默认索引。

## 五、用户Query处理——提升检索效果的关键一步

核心：将用户提问转化为更易匹配的向量，解决“提问模糊、语义稀疏”导致的检索失效，统称Query Rewrite

- Query优化方式：

    - 问题改写：补充用户提问中缺失的信息、优化表述，提升与知识库切片的匹配成功率。

    - 问题扩写：生成多个类似问题，提升与知识库切片匹配成功率。

    - 问题拆解：当用户提问过于复杂（多意图、多维度），将其拆解为多个简单子问题，分别检索，最后汇总答案。

    - HyDE（假设性答案检索）：生成一段与问题相关的假答案（假设性回答），用假答案的向量进行检索，解决“提问过短、语义不明确”的问题。

## 六、混合检索——提升检索精准度（工业标配）

核心：结合“语义匹配”和“关键词匹配”，叠加元数据过滤，避免单一检索的局限性

- 两种核心检索方式：

    - 稠密检索：基于Embedding向量，使用余弦相似度（最常用）、欧氏距离、点积计算语义匹配度，解决“语义相似但关键词不同”的问题。

    - 稀疏检索：

        - 基于关键词匹配，常用BM25、TF\-IDF，解决“关键词明确但语义表述不同”的问题，搜索指定参数、型号等；

        - 进阶用稀疏向量（如BGE\-M3自带），使用点积相似度，提升精度。语义理解能力和关键词搜索能力介于稠密检索和关键词匹配之间。

- 工业方案：稠密\+稀疏混合检索（Hybrid Search），通过RRF合并排序结果，再叠加元数据筛选（如按时间、文档类型、部门过滤），进一步提升检索准确度。

- 检索策略：Top\-K召回（默认召回几十到上百条），结合相似度阈值过滤，避免低质量切片进入后续环节。

- 补充细节：负样本过滤（Negative Retrieval）：在微调自己的Embedding模型时候，需要删掉太简单的负样本（随机无关）、删掉**假负样本**（看起来不像、实际相关，会训崩模型），只留**难负样本**，模型才学得会区分。然后通过对比学习微调Embedding模型

## 七、重排序（Rerank）—— 从粗召回中筛选最优结果

核心：将粗召回的100条左右信息，精排序筛选出最相关的10条左右，提升后续LLM生成的精准度，降低成本

- 三种主流重排序策略：

    - Cross\-Encoder（交叉编码器）：输入“Query\+文档切片”拼接内容，一对一打分排序，精度天花板，但推理成本高、串行打分速度慢，适合对精度要求极高的场景。（例如BGE\-Rerank、Qwen\-Rerank）

    - ColBERT：将Query和文档都转化为Token级向量，计算两者的最大相似度匹配分数；支持离线预编码文档向量，速度远快于Cross\-Encoder，精度略低，适合工业大规模落地。

    - 轻量级Rerank模型：BGE\-Rerank、Qwen\-Rerank，精度介于两者之间，速度快、成本低，是目前工业主流选择。

## 八、LLM答案生成与幻觉抑制——RAG落地的核心目标

核心：基于检索到的切片生成答案，重点抑制幻觉，保证回答的准确性和合规性，分“规则层\+技术层”双重管控

- 规则层管控：

    - System Prompt强约束：明确要求模型仅依靠检索到的内容回答，禁止编造。

    - 检索阈值过滤：设置相似度阈值，低于阈值的低质量切片不送入LLM。

    - 引用溯源（Citation）：要求模型生成答案时，标注引用的切片来源（如文档名称、章节、切片ID），便于追溯和校验。

    - 兜底策略：发现幻觉问题时，重新检索或转人工处理。

- 技术层管控：

    - 上下文压缩：用LLM或轻量模型压缩检索到的多条切片，去除冗余内容，减少上下文长度，降低幻觉概率和LLM成本。

    - 事实约束与校验：通过专门的事实校验模型，对生成的答案进行校验，发现幻觉及时修正。

    - 知识库管理：定期进行知识库增量更新、版本管理，删除过时、错误数据（过时数据是幻觉的核心来源之一）。

    - Self\-RAG：让模型自主判断“是否需要检索、检索内容是否可用、是否需要重检索”，自主规避幻觉。

## 九、RAG优化方案

### 流程设计优化

- Corrective RAG（CRAG）：在生成回答之前，使用一个质检员评估检索内容是否有用，如果检索内容没用，直接进入降级、兜底流程。

- Self\-RAG：添加四个检索点，包括这个问题是否需要检索、检索到的内容是否相关（同CRAG）、生成的回答是否有文档支撑、生成的回答是否有用。

- Adaptive RAG：收到问题之后进入问题路由，简单问题直接回答、中等难度问题普通RAG回答、困难问题Self\-RAG回答。

- Agentic RAG：让LLM根据问题自行调度，选择最合适的RAG方式。比如是使用混合检索召回密集文本，还是使用GraphRAG进行多跳查询，还是使用Text2SQL RAG检索结构化数据。

- Multi\-Agent RAG：拆分整个流程中的各个角色（分发、质检、推理等），每个环节可以单独迭代优化。

- Speculative RAG：将检索到的文档分成多个子集，使用弱模型并行生成答案后，由强模型进行验证与择优。

### 结构化知识增强

- GraphRAG：解决多跳问题，将文本数据先通过知识三元组抽取，构建知识图谱（Neo4j），查询流程为 问题向量化 → 知识图谱节点搜索 → 根据chunk\_id 查询切片。成本高，需要衡量。

- Text2SQL RAG：将查询转化为SQL语句进行信息获取，需要特别注意安全问题。（如 SQL 注入）

- 多模态RAG：把图片、表格和文本统一到一个向量空间里。

### 工程落地优化

- 增量入库：全量更新（适合小知识库）VS 增量更新（适合大知识库，仅更新新增/修改文档），搭配定时爬取/同步机制。

- 向量库运维：数据过期删除、冷热数据分离（高频访问数据放内存，低频放磁盘）、索引优化。

- 性能优化：Embedding批量推理、向量检索缓存（缓存高频Query的检索结果）、Rerank批量处理。

- 评估体系：RAG三大核心指标——召回率（检索到的相关切片比例）、精准率（检索到的切片中真正相关的比例）、答案合规性（无幻觉、符合要求）；评测方式：自动化评估（用数据集验证）\+ 人工抽检。

## 十、评估框架RAGAS

四个核心指标：

- 忠实度（回答有没有瞎编）

- 答案相关性（答的是不是你问的）

- 上下文精确率（搜到的有多少是有用的）

- 上下文召回率（该搜到的搜到了吗）

从数据库**导出已有原文 chunk **→ LLM 反向生成：`question \+ reference标准答案` → 用**线上真实 RAG 链路**跑出：`contexts \+ answer` → 组成 RAGAS 数据集，自动打分 → 看低分 case → 改检索 / 分块 / 重排 → 复测对比迭代
