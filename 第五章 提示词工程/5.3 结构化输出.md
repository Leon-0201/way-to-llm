# 5.3 结构化输出

> **核心理解**：结构化输出是通过提示词约束模型输出格式，使结果可直接被机器解析和处理的技术。它将非结构化的自然语言转换为结构化数据，降低后续处理成本。

---

## 1. 为什么需要结构化输出

### 1.1 核心问题
```
非结构化输出的痛点：
- 格式不统一，难以自动解析
- 包含冗余信息，需要人工处理
- 无法直接对接下游系统
- 结果不稳定，容易出现格式错误
```

### 1.2 结构化输出的优势
```
✅ 机器可直接解析
✅ 格式统一，易于自动化处理
✅ 对接下游系统（数据库、API、业务逻辑）
✅ 结果可验证，便于调试
✅ 降低后续处理成本
```

---

## 2. 常见结构化输出方法

### 2.1 JSON Schema 约束

**核心思想**：通过JSON Schema定义输出格式，强制模型生成符合规范的JSON数据。

#### 2.1.1 JSON Schema 原理

JSON Schema是一种基于JSON格式的规范，它定义了一套词汇和规则，用来描述JSON数据的结构和内容，以及验证JSON数据是否满足特定的结构和约束。

**核心原理**：
```
1. 结构定义：描述JSON对象的属性、类型、格式等
2. 约束规则：定义必填字段、枚举值、范围限制等
3. 验证机制：通过Schema验证JSON数据的合法性
4. 自动生成：基于Schema自动生成符合格式的JSON
```

#### 2.1.2 Mask 实现输出控制

在大模型中，JSON Schema通过**Constrained Decoding**技术实现输出控制：

```
1. Schema解析：将JSON Schema转换为语法规则
2. Token Masking：在生成每个Token时，mask掉不符合Schema的Token（将其生成概率设为0）
3. 强制合规：模型只能生成符合Schema规则的Token序列
4. 100%格式保证：最终输出一定符合预先定义的Schema
```

**技术细节**：
```
- 基于有限状态机（FSM）跟踪当前位置的合法Token
- 动态更新允许的Token集合
- 拒绝不符合语法规则的Token生成
- 确保输出结构完全匹配Schema
```

#### 2.1.3 示例与技巧

**示例提示词**：
```json
{
  "type": "object",
  "properties": {
    "name": {"type": "string"},
    "age": {"type": "integer"},
    "hobbies": {"type": "array", "items": {"type": "string"}}
  },
  "required": ["name", "age"]
}

请根据上述JSON Schema格式输出以下信息：
小明，25岁，喜欢篮球、阅读和编程
```

**输出结果**：
```json
{
  "name": "小明",
  "age": 25,
  "hobbies": ["篮球", "阅读", "编程"]
}
```

**关键技巧**：
- 使用明确的格式要求
- 提供示例输出
- 增加格式验证提示
- 使用工具调用格式（Tool Call）
- 优先选择支持原生JSON模式的模型（如Qwen3、GPT-4o）

### 2.2 函数调用格式

**核心思想**：将模型输出格式化为函数调用形式，便于直接执行。

**示例提示词**：
```python
请使用以下函数格式输出结果：
def add(a: int, b: int) -> int:
    return a + b

问题：3 + 5 = ?
```

**输出结果**：
```python
add(3, 5)
```

**适用场景**：
- 工具调用
- API调用
- 代码生成
- 自动化执行

### 2.3 XML/Markdown 格式控制

**核心思想**：使用标记语言定义输出结构，便于解析和展示。

**XML示例**：
```xml
<student>
  <name>小明</name>
  <age>25</age>
  <hobbies>
    <hobby>篮球</hobby>
    <hobby>阅读</hobby>
    <hobby>编程</hobby>
  </hobbies>
</student>
```

**Markdown示例**：
```markdown
| 姓名 | 年龄 | 爱好 |
|------|------|------|
| 小明 | 25 | 篮球、阅读、编程 |
```

**适用场景**：
- 文档生成
- 报告输出
- 网页内容
- 表格数据

## 3. 实现技巧

### 3.1 格式约束四步法
```
1. 明确格式要求：告诉模型需要生成的格式
2. 提供示例输出：展示正确格式的例子
3. 增加验证提示：要求模型检查输出格式
4. 处理错误情况：说明格式错误时的处理方式
```

### 3.2 常见问题与解决

**❌ 格式错误**：
```
问题：模型输出不符合要求
解决：
- 增加格式验证提示
- 提供更详细的示例
- 使用工具调用格式（模型对函数调用格式支持更好）
```

**❌ 信息缺失**：
```
问题：模型遗漏必填字段
解决：
- 明确列出所有必填字段
- 使用JSON Schema的required属性
- 增加字段检查提示
```

**❌ 类型错误**：
```
问题：模型输出错误的数据类型
解决：
- 明确指定每个字段的类型
- 使用JSON Schema的type属性
- 增加类型验证提示
```

## 5. 最佳实践

### 5.1 基础原则
```
1. 格式优先：先定义格式，再填充内容
2. 简洁明确：避免模糊的格式要求
3. 示例驱动：提供清晰的示例输出
4. 验证机制：增加格式检查和错误处理
5. 逐步优化：根据实际输出调整提示词
```

### 5.2 高级技巧
```
- 使用工具调用格式（Tool Call）替代普通JSON
- 结合JSON Schema和示例输出
- 使用多轮对话修正格式错误
- 自动化验证输出格式
```

## 6. 面试高频问答

### 6.1 基础概念类

**Q1: 什么是结构化输出？为什么重要？**

**速答模板**：
```
结构化输出是通过提示词约束模型输出格式，使结果可直接被机器解析和处理的技术。

重要性：
1. 机器可直接解析，无需人工处理
2. 格式统一，易于自动化流程
3. 对接下游系统，降低集成成本
4. 结果可验证，便于调试和监控
```

---

**Q2: JSON Schema 约束和普通JSON提示有什么区别？**

**速答模板**：
```
普通JSON提示：仅描述格式，缺乏严格约束
JSON Schema：提供机器可验证的格式定义

区别：
1. 严格性：Schema包含类型、必填项、枚举等约束
2. 可验证性：Schema可被工具自动验证
3. 清晰度：Schema更明确地定义格式要求
4. 稳定性：Schema减少格式错误的可能性
```

---

### 6.2 实践应用类

**Q3: 如何提高结构化输出的成功率？**

**速答模板**：
```
我的方法：
1. 明确格式要求：使用JSON Schema或函数调用格式
2. 提供示例输出：展示正确格式的例子
3. 增加验证提示：要求模型检查输出格式
4. 处理错误情况：说明格式错误时的处理方式
5. 逐步优化：根据实际输出调整提示词

经验：
- 工具调用格式的成功率最高（模型对其优化更好）
- 示例输出比纯文字描述更有效
- 增加"请严格按照格式输出"等提示词
```

**Q4: 结构化输出有哪些常见问题？如何解决？**

