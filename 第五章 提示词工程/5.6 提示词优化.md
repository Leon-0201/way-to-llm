# 提示词优化

> **核心理解**：提示词优化是通过系统化的方法和工具，持续提升提示词效果的过程。核心在于建立可量化的评估体系，结合自动化技术和人工经验，实现提示词的迭代优化。

---

## 1. 为什么需要提示词优化

### 1.1 核心问题
```
1. 效果不稳定
   - 同一提示词，不同模型表现差异大
   - 同一模型，不同版本表现不同
   - 微小改动可能导致效果大幅波动

2. 依赖经验
   - 新手难以写出高效提示词
   - 专家经验难以复制和传承
   - 优化过程耗时耗力

3. 缺乏量化
   - 难以客观评估提示词质量
   - 优化方向不明确
   - 无法比较不同方案
```

### 1.2 优化目标
```
1. 准确性：输出结果的正确率
2. 稳定性：多次运行的一致性
3. 效率：token 消耗和响应时间
4. 泛化性：对不同输入的适应能力
5. 可维护性：易于理解和修改
```

## 2. 提示词优化方法

### 2.1 人工优化（Manual Optimization）

**核心思想**：基于经验和实验，手动调整提示词。

**优化流程**：
```
1. 基线建立
   - 编写初始提示词
   - 在测试集上评估效果
   - 记录基准指标

2. 问题分析
   - 收集错误案例
   - 分析错误类型和原因
   - 识别改进点

3. 假设生成
   - 基于分析提出优化假设
   - "如果添加示例，效果会更好"
   - "如果简化指令，会更清晰"

4. 实验验证
   - A/B 测试不同版本
   - 统计显著性检验
   - 选择最优方案

5. 迭代循环
   - 持续监控效果
   - 发现新问题
   - 重复优化流程
```

**关键技巧**：
```
✅ 单一变量原则：每次只改一个地方
✅ 详细记录：记录每次改动和结果
✅ 多样化测试：覆盖不同场景和边界案例
✅ 用户反馈：收集真实用户的使用体验
```

---

### 2.2 自动优化（Automatic Optimization）

#### 方法 1：AutoPrompt

**核心思想**：使用梯度或搜索算法自动优化提示词。

**工作原理**：
```
1. 初始化：随机或基于模板生成初始提示词
2. 评估：在训练集上评估效果
3. 优化：使用算法生成新候选
   - 基于梯度：计算提示词的梯度方向
   - 基于搜索：遗传算法、贝叶斯优化等
4. 选择：选择效果最好的候选
5. 迭代：重复 2-4 直到收敛
```

**示例流程**：
```python
# 伪代码示例
def autoprompt_optimize(task, training_data):
    # 1. 初始化提示词模板
    prompt_template = "完成以下任务：{task}\n输入：{input}\n输出："
    
    # 2. 定义搜索空间
    search_space = {
        "prefix": ["请", "请帮我", "麻烦"],
        "instruction": ["完成", "处理", "分析"],
        "format": ["直接输出", "逐步说明", "详细解释"]
    }
    
    # 3. 搜索最优组合
    best_prompt = None
    best_score = 0
    
    for combination in generate_combinations(search_space):
        prompt = build_prompt(prompt_template, combination)
        score = evaluate(prompt, training_data)
        
        if score > best_score:
            best_score = score
            best_prompt = prompt
    
    return best_prompt
```

**适用场景**：
- ✅ 有明确的评估指标
- ✅ 有足够的训练数据
- ✅ 提示词结构相对固定
- ❌ 需要大量计算资源
- ❌ 可能陷入局部最优

#### 方法 2：基于大模型的优化（LLM-based Optimization）

**核心思想**：使用大模型本身来优化提示词。

**代表工具**：
- DSPy
- PromptEngine
- LangChain 的提示词优化模块

**工作流程**：
```
1. 提供任务描述和示例
2. LLM 分析当前提示词的问题
3. LLM 生成优化建议
4. LLM 生成新的提示词版本
5. 评估并选择最优版本
```

**示例（DSPy 风格）**：
```python
import dspy

# 1. 定义任务

optimizer = dspy.BootstrapFewShot(

# 3. 优化提示词
optimized_model = optimizer.compile(
    model=base_model,
    trainset=trainset
)
```

**优势**：
```
✅ 自动化程度高
✅ 可以发现人类忽略的优化点
✅ 快速迭代多个版本
✅ 可复现和可比较
```

**劣势**：
```
❌ 依赖评估指标的质量
❌ 可能过度拟合训练数据
❌ 需要一定的技术门槛
```

#### 方法 3：基于反馈的优化（Feedback-based Optimization）

**核心思想**：利用用户反馈持续优化提示词。

**实现方式**：
```
1. 收集反馈
   - 显式反馈：点赞/点踩、评分
   - 隐式反馈：停留时间、修改行为
   - 对话反馈：用户追问、纠正

2. 分析反馈
   - 识别不满意案例
   - 分析不满意原因
   - 归类问题类型

3. 生成优化
   - 针对问题类型调整提示词
   - 添加针对性示例
   - 修改模糊或歧义表述

4. 验证效果
   - A/B 测试新旧版本
   - 监控反馈指标变化
   - 确认改进效果
```

**示例系统**：
```python

## 3. 评估体系

### 3.1 评估指标

#### 定量指标
```
1. 准确性（Accuracy）
   - 正确答案比例
   - 任务完成率
   - 示例：数学题正确率

2. 一致性（Consistency）
   - 多次运行结果相似度
   - 不同输入下的稳定性
   - 示例：同一问题 10 次运行，8 次答案相同 = 80% 一致性

3. 效率（Efficiency）
   - token 消耗量
   - 响应时间
   - 示例：平均每次调用 500 tokens

4. 覆盖率（Coverage）
   - 能处理的场景比例
   - 边界案例处理能力
   - 示例：100 个测试案例，90 个通过 = 90% 覆盖率
```

#### 定性指标
```
1. 可读性：提示词是否清晰易懂
2. 可维护性：是否易于修改和扩展
3. 泛化性：对新场景的适应能力
4. 安全性：是否存在安全风险
```

---

### 3.2 评估方法

#### 方法 1：Golden Set 测试
```
原理：使用人工标注的标准测试集评估

步骤：
1. 构建测试集（100-1000 个案例）
2. 每个案例包含输入和期望输出
3. 运行提示词，对比实际输出和期望输出
4. 计算各项指标

优点：
- 客观可量化
- 可复现
- 便于比较

缺点：
- 构建成本高
- 可能覆盖不全
- 需要定期更新
```

#### 方法 2：A/B 测试
```
原理：同时运行多个版本，对比效果

步骤：
1. 准备多个提示词版本（A、B、C...）
2. 随机分配用户请求到不同版本
3. 收集各版本的反馈数据
4. 统计显著性检验
5. 选择最优版本

优点：
- 真实场景测试
- 直接反映用户满意度
- 发现意外问题

缺点：
- 需要足够流量
- 测试周期长
- 可能影响用户体验
```

#### 方法 3：人工评估
```
原理：由专业人员评估输出质量

评估维度：
1. 准确性：答案是否正确
2. 完整性：是否覆盖所有要点
3. 清晰度：表达是否清晰
4. 有用性：是否解决用户问题
5. 安全性：是否存在风险

评分方式：
- Likert 量表（1-5 分）
- 多维度打分
- 多人评估取平均

优点：
- 全面深入
- 发现细微问题
- 提供改进建议

缺点：
- 成本高
- 主观性强
- 难以规模化
```

#### 方法 4：LLM-as-a-Judge
```
原理：使用大模型评估输出质量

实现：
```python
evaluation_prompt = """
请评估以下 AI 助手的回答质量：

用户问题：{question}
AI 回答：{answer}

请从以下维度打分（1-5 分）：
1. 准确性：答案是否正确
2. 完整性：是否覆盖所有要点
3. 清晰度：表达是否清晰
4. 有用性：是否解决用户问题

请给出具体分数和理由。
"""
```

优点：
- 成本低
- 可规模化
- 标准一致

缺点：
- 依赖评估模型质量
- 可能有偏见
- 难以评估复杂问题
```

## 4. 优化实践

### 4.1 优化 Checklist

**提示词质量检查**：
```
□ 目标明确：任务目标清晰无歧义
□ 格式清晰：结构合理，层次分明
□ 示例充分：提供足够的正面示例
□ 边界清晰：说明限制和约束条件
□ 语气适当：符合场景的语语气风格
□ 安全合规：不包含风险内容
□ 简洁高效：无冗余信息
□ 可维护：易于理解和修改
```

**常见问题检查**：
```
□ 模糊指令：如"处理一下"、"分析一下"
□ 缺少示例：没有提供参考格式
□ 信息过载：一次性要求太多
□ 逻辑混乱：指令顺序不合理
□ 自相矛盾：不同部分要求冲突
□ 忽略边界：没有说明限制条件
□ 过度复杂：不必要的复杂性
```

### 4.2 优化技巧

#### 技巧 1：增量优化
```
原则：小步快跑，快速迭代

做法：
1. 每次只改动一个小地方
2. 立即测试验证效果
3. 记录改动和结果
4. 保留有效改动，回滚无效改动

示例：
版本 1: "请总结这篇文章"
版本 2: "请用 3 句话总结这篇文章"  ← 添加数量限制
版本 3: "请用 3 句话总结这篇文章的核心观点"  ← 明确总结内容
版本 4: "请用 3 句话总结这篇文章的核心观点，每句不超过 30 字"  ← 添加长度限制
```

#### 技巧 2：对比实验
```
原则：控制变量，科学对比

做法：
1. 确定要测试的变量（如示例数量）
2. 设计多个版本（0 个、3 个、5 个示例）
3. 在相同测试集上运行
4. 统计对比结果
5. 选择最优方案

示例：
A 组：无示例
B 组：3 个示例
C 组：5 个示例

结果：B 组准确率最高，且 token 消耗合理 → 选择 B 组
```

#### 技巧 3：错误分析
```
原则：从错误中学习

做法：
1. 收集所有错误案例
2. 分类错误类型
   - 理解错误
   - 格式错误
   - 信息缺失
   - 逻辑错误
3. 针对每类错误设计修复
4. 验证修复效果

示例：
错误类型：格式错误（30%）
原因：没有提供格式示例
修复：添加 2-3 个格式示例
验证：格式错误降至 5%
```

#### 技巧 4：边界测试
```
原则：测试极端情况

做法：
1. 设计边界案例
   - 超长输入
   - 超短输入
   - 模糊输入
   - 矛盾输入
2. 测试提示词表现
3. 添加边界处理说明

示例：
边界案例：
- 输入为空怎么办？
- 输入包含敏感信息怎么办？
- 问题超出能力范围怎么办？

优化：
"如果输入为空或无法理解，请礼貌地请求用户提供更多信息。"
```
