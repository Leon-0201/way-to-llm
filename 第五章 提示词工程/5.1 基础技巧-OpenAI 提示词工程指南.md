# OpenAI 提示词工程指南学习笔记

> **核心理解**：提示词工程的本质是通过精心设计的上下文，引导 LLM 的 next token 概率分布向高质量区域集中。所有技巧都在帮助概率采样采到更好的 next token。

---

## 1. 理解 LLM 的底层工作原理

### 1.1 Next Token Prediction 是核心

**LLM 的工作原理：**
```
P(next_token | context) = softmax(logits)
```

- 模型根据已有上下文（提示词 + 历史对话）预测下一个 token
- 所有提示词技巧都在**影响这个条件概率分布**
- 好的提示词 = 降低熵值 = 概率分布更集中 = 输出质量更高

### 1.2 从概率论理解提示词工程

**信息论视角：**
```
H(X) = -Σ P(x) log P(x)

清晰指令 → 降低 H(next_token) → 采样更集中 → 质量更高
```

**贝叶斯视角：**
```
P(理想输出 | 提示词) ∝ P(提示词 | 理想输出) × P(理想输出)
```

- **角色设定**：调整先验 P(理想输出)
- **示例**：提供 P(输出 | 输入) 的条件分布样本
- **参考文本**：提供强证据，极大提高后验概率

## 2. 写出清晰的指令

### 2.1 在你的查询中包含详尽的信息，以获取更为准确的答案

**核心要点：**
- 提供充足的上下文信息
- 明确说明任务背景和目标
- 避免模糊和歧义的表述

**从概率角度理解：**
- 模糊指令 → 高熵值 → 概率分布分散 → 质量不稳定
- 详细信息 → 低熵值 → 概率分布集中 → 质量稳定

**示例对比：**

❌ 模糊的指令（高熵）：
```
帮我写一封邮件
```
模型的概率分布：
- 商务邮件？私人邮件？投诉信？感谢信？...（各种可能）

✅ 清晰的指令（低熵）：
```
帮我写一封专业的商务邮件给客户张三，主题是关于我们新推出的 SaaS 产品演示安排。
邮件内容应包括：
1. 感谢他上周的咨询
2. 提供三个可选的演示时间（下周二至周四下午 2-4 点）
3. 说明演示时长约 30 分钟
4. 附上会议链接
语气要友好但专业，长度控制在 150 字以内
```
模型的概率分布：
- 商务语气、SaaS 产品、演示安排、具体时间...（高度集中）

**学习笔记：**
- 信息越详细，模型输出越精准
- 包含约束条件（时间、长度、语气等）
- 明确列出需要包含的要点

### 2.2 要求模型采用角色

**核心要点：**
- 通过角色设定让模型进入特定专业领域
- 角色会影响回答的语气、深度和视角

**从概率角度理解：**
```
P("反向传播" | 上下文，角色=算法工程师) >> P("反向传播" | 上下文，角色=厨师)
```

角色设定的作用：
1. **缩小采样空间**：将概率质量重新分配到与角色相关的 token 上
2. **激活特定知识区域**：Attention 机制更多关注训练数据中与该角色相关的内容
3. **改变先验分布**：P(理想输出 | 角色) 更集中

**示例：**

```
你是一位经验丰富的 Python 技术面试官，请帮我准备一份针对中级开发者的面试题库。
问题应涵盖：
- Python 基础语法
- 数据结构与算法
- 实际项目经验
- 代码优化能力
```

**概率解释：**
- 角色 = "Python 技术面试官"
- 激活的知识子空间：Python、面试、算法、代码审查...
- 抑制的知识子空间：烹饪、娱乐、体育...（无关内容概率降低）

**常见角色设定：**
- 技术专家（程序员、数据科学家、架构师）
- 教育工作者（老师、导师、培训师）
- 商业角色（营销专家、产品经理、顾问）
- 创意角色（作家、编辑、设计师）

**学习笔记：**
- 角色设定 = 专业领域 + 经验水平 + 任务视角
- 可以组合多个角色，如"资深 Python 开发者兼技术作家"
- 角色越具体，概率分布越集中

### 2.3 使用分隔符清楚地指示输入的不同部分

**核心要点：**
- 使用分隔符区分指令、输入数据、参考材料
- 常用分隔符：```、"""、---、###、<> 等

**从概率角度理解：**
- 分隔符 = 明确的语义边界标记
- 帮助 Attention 机制正确分配权重
- 避免不同部分的 token 互相干扰

**示例：**

```
请总结以下文章的主要内容：

"""
[文章内容放在这里]
"""

请用 3 个要点进行总结，每个要点不超过 20 字。
```

**Attention 机制视角：**
```
Attention(Q, K, V) = softmax(QK^T / √d)V

分隔符帮助模型：
- 识别哪些 token 属于指令（高权重）
- 哪些 token 属于内容（需要处理）
- 哪些 token 属于约束条件（需要遵循）
```

**复杂场景示例：**

```
任务：分析客户反馈并提取关键问题

指令：
1. 阅读客户反馈
2. 识别主要问题类别
3. 为每个问题提供严重性评级（1-5 分）
4. 给出改进建议

客户反馈：
"""
[客户反馈内容]
"""

输出格式：
| 问题类别 | 描述 | 严重性 | 建议 |
```

**学习笔记：**
- 分隔符让模型清晰理解不同部分的作用
- 对于复杂任务，使用多层分隔结构
- 保持分隔符使用的一致性

### 2.4 指定完成任务所需的步骤

**核心要点：**
- 将复杂任务分解为明确的步骤
- 按顺序列出每个步骤
- 对于多步骤任务，要求模型逐步执行

**从概率角度理解：**
```
复杂任务直接完成：
P(结果 | 复杂任务) —— 需要同时考虑多个因素，熵值高

分步骤执行：
P(步骤 1 | 任务) → P(步骤 2 | 步骤 1) → P(步骤 3 | 步骤 2) → ...
每一步的条件概率都很高，整体质量提升
```

**示例：**

```
请按照以下步骤分析这份销售数据：

步骤 1：计算每个月的总销售额和增长率
步骤 2：识别销售额最高和最低的三个产品类别
步骤 3：分析季度趋势和季节性模式
步骤 4：基于历史数据预测下个季度的销售额
步骤 5：提供 3 条具体的业务改进建议

请按顺序执行每个步骤，并在每个步骤后展示计算结果。
```

**为什么有效：**
- 降低每一步的认知负担
- 中间结果作为后续步骤的上下文
- 避免遗漏关键环节

**学习笔记：**
- 步骤分解降低模型的认知负担
- 明确步骤之间的依赖关系
- 对于关键步骤，可以要求模型展示中间结果

---

### 2.5 举例说明

**核心要点：**
- 通过示例展示期望的输出格式和风格
- 示例比描述更直观有效
- 提供 1-3 个典型示例即可

**从概率角度理解：**
```
Few-Shot Learning 提供：
P(输出 | 输入) 的条件分布样本

模型不需要从零推断任务模式
直接学习 (input, output) 对的映射关系
```

**示例：**

```
请将以下口语化表达转换为正式商务用语：

示例 1：
输入："这个东西太贵了"
输出："该产品的定价超出了我们的预算范围"

示例 2：
输入："我不确定能不能按时完成"
输出："我们需要评估是否能按既定时间表完成交付"

现在请转换：
输入："这个功能做起来太麻烦了"
```

**为什么 Few-Shot 比 Zero-Shot 好：**
- 提供了任务的具体实例
- 减少了模型猜测任务意图的不确定性
- 明确了输出格式和风格的概率分布

**学习笔记：**
- 使用"输入 - 输出"对展示转换模式
- 示例应覆盖常见场景
- 示例风格要与目标任务一致

---

### 2.6 指定所需的输出长度

**核心要点：**
- 明确指定字数、段落数、要点数等
- 可以使用"简短"、"详细"等相对描述
- 最好给出具体数字

**从概率角度理解：**
- 长度约束 = 额外的条件信息
- 减少模型在"写多少"上的不确定性
- 让模型在有限的 token 预算内优化信息密度

**示例：**

```
请用 200-250 字总结这篇文章的核心观点

或

请用 3 个要点总结，每个要点不超过 30 字

或

请提供一个段落（5-7 句话）的摘要
```

**学习笔记：**
- 长度约束帮助模型控制信息密度
- 可以组合使用：要点数 + 每点字数
- 对于表格输出，可以指定行列数

## 3. 提供参考文本

### 3.1 指示模型使用参考文本回答

**核心要点：**
- 提供权威、准确的参考材料
- 明确要求模型基于参考内容回答
- 减少模型幻觉（编造信息）

**从概率角度理解：**
```
P(准确答案 | 参考文本) >> P(准确答案 | 无参考文本)

参考文本提供：
- 强约束条件
- 减少模型依赖内部参数记忆
- 降低幻觉概率
```

**示例：**

```
请根据以下产品文档回答用户的问题：

产品文档：
"""
[产品规格、功能说明等]
"""

用户问题：这个产品支持哪些集成？

要求：
- 仅基于上述文档回答
- 如果文档中没有相关信息，请说明"文档中未提及此信息"
- 不要使用外部知识
```

**为什么有效：**
- 参考文本作为上下文的一部分
- 模型可以直接"引用"而不是"回忆"
- 减少参数记忆中的噪声干扰

**学习笔记：**
- 适用于客服、技术支持等场景
- 确保参考文本的完整性和准确性
- 明确限制模型使用外部知识

---

### 3.2 指示模型使用参考文本中的引用来回答

**核心要点：**
- 要求模型引用原文支持答案
- 提高答案的可信度和可验证性
- 便于用户追溯信息来源

**从概率角度理解：**
- 引用要求 = 额外的验证机制
- 模型必须找到原文支持才能生成答案
- 降低编造信息的概率

**示例：**

```
请根据以下研究论文回答问题，并在回答中引用相关段落：

论文内容：
"""
[论文全文]
"""

问题：这项研究的主要发现是什么？

要求：
- 使用引号标注原文引用
- 标注引用所在的章节或页码
- 先给出总结，再提供支持证据
```

**输出示例：**
```
这项研究的主要发现包括：

1. "实验组的表现比对照组提升了 35%"（第 3 节：实验结果）
2. "该方法在大规模数据集上表现最佳"（第 4 节：讨论）
```

**学习笔记：**
- 引用格式要明确要求
- 适合学术、法律等需要严谨出处的场景
- 可以要求标注段落号、章节名等

## 4. 将复杂的任务拆分为更简单的子任务

### 4.1 使用意向分类来识别与用户查询最相关的说明

**核心要点：**
- 先判断用户意图，再选择对应的处理流程
- 建立意图分类体系
- 不同意图对应不同的提示词模板

**从概率角度理解：**
```
P(回答 | 意图=信息查询) ≠ P(回答 | 意图=操作指导)

先分类再处理：
P(回答 | 问题) = Σ P(回答 | 意图 i) × P(意图 i | 问题)

选择概率最高的意图对应的处理策略
```

**示例：**

```
第一步：判断用户查询的意图类型
- 信息查询：询问事实、定义、解释
- 操作指导：需要步骤说明
- 创意生成：需要创意内容
- 分析任务：需要数据分析
- 代码相关：需要编程帮助

第二步：根据意图类型选择相应的响应策略

用户查询：[用户输入]

请先判断意图类型，然后按照对应策略回答。
```

**学习笔记：**
- 意图分类提高回答的针对性
- 可以构建多层次的分类体系
- 适合构建复杂的对话系统

---

### 4.2 对于需要很长对话的对话应用程序，总结或过滤以前的对话

**核心要点：**
- 长对话会消耗大量 token
- 定期总结历史对话节省上下文
- 保留关键信息，过滤冗余内容

**从概率角度理解：**
```
P(next_token | 完整历史) vs P(next_token | 关键信息摘要)

总结的作用：
- 保留高信息量的内容
- 去除低信息量的冗余
- 在有限的 context window 内最大化信息密度
```

**示例：**

```
请总结之前对话的关键信息：

对话历史：
"""
[之前的对话内容]
"""

总结要求：
1. 提取用户的核心需求和偏好
2. 记录已解决的问题
3. 列出待处理的事项
4. 保留重要的上下文信息
5. 总结控制在 300 字以内
```

**应用场景：**
- 客服对话系统
- 心理咨询助手
- 长期项目协作

**学习笔记：**
- 在对话达到一定长度时触发总结
- 总结作为新的上下文继续对话
- 可以分层总结：最近详细 + 历史概要

---

### 4.3 分段总结长文档，递归构建完整摘要

**核心要点：**
- 长文档超出 token 限制时分段处理
- 先分段总结，再整合为完整摘要
- 递归应用总结策略

**从概率角度理解：**
```
直接总结长文档：
- 超出 context window
- 注意力分散，质量下降

分段总结：
P(摘要 | 文档) ≈ P(摘要 | 分段摘要 1, 分段摘要 2, ...)

每段总结的概率分布都很集中
整体质量更高
```

**示例：**

```
任务：总结这份 50 页的报告

处理步骤：

步骤 1：将文档按章节分为 10 个部分
步骤 2：为每个部分生成 100 字摘要
步骤 3：整合 10 个部分摘要为 500 字总摘要
步骤 4：从总摘要中提取 3 个核心要点

请按步骤执行，并展示每个步骤的结果。
```

**学习笔记：**
- 适合处理书籍、长报告、论文等
- 保持各段摘要的结构一致性
- 最终摘要应反映文档的整体结构

## 5. 给模型时间"思考"

### 5.1 在匆忙下结论之前，指示模型制定自己的解决方案

**核心要点：**
- 要求模型先思考再回答
- 对于复杂问题，先规划解决路径
- 避免仓促的错误答案

**从概率角度理解：**
```
直接回答复杂问题：
P(答案 | 问题) —— 需要考虑太多因素，熵值高

先规划再回答：
P(答案 | 问题，规划) = P(答案 | 规划) × P(规划 | 问题)

规划步骤降低了答案空间的熵
```

**示例：**

```
问题：[复杂问题]

在回答之前，请先：
1. 分析问题涉及的关键因素
2. 列出可能的解决方法
3. 评估每种方法的优缺点
4. 选择最佳方案并说明理由
5. 最后给出完整答案
```

**学习笔记：**
- "思考过程"提高答案质量
- 适合数学、逻辑、策略等问题
- 可以要求模型自我质疑和验证

---

### 5.2 使用内心独白或一系列查询来隐藏模型的推理过程

**核心要点：**
- 让模型在内部进行推理
- 只输出最终答案，不展示思考过程
- 或者分阶段输出：先推理，后答案

**从概率角度理解：**
```
隐藏推理：
P(最终答案 | 问题，内部推理)

内部推理作为隐变量，提高最终答案的条件概率
```

**示例（隐藏推理）：**

```
请解答以下数学问题：

[数学题目]

要求：
- 在内部逐步推理
- 验证每一步的正确性
- 只输出最终答案和关键步骤
```

**示例（分阶段输出）：**

```
请按以下格式回答：

【分析过程】
[你的思考和推理]

【最终答案】
[简洁的答案]
```

**学习笔记：**
- 隐藏推理适合生产环境
- 展示推理适合教学场景
- 可以灵活选择输出模式

---

### 5.3 询问模型在之前的过程中是否遗漏了什么内容

**核心要点：**
- 让模型自我检查和反思
- 发现可能的遗漏和错误
- 提高答案的完整性

**从概率角度理解：**
```
P(完整答案 | 问题，自我检查) > P(完整答案 | 问题)

自我检查作为二次采样：
- 第一次采样：生成初始答案
- 第二次采样：检查并补充

降低遗漏重要信息的概率
```

**示例：**

```
请完成以下任务：

[任务描述]

完成之后，请自我检查：
1. 是否回答了所有问题？
2. 是否有遗漏的重要信息？
3. 是否有逻辑不一致的地方？
4. 是否需要补充说明？

如果有遗漏，请补充完整。
```

**学习笔记：**
- 自我检查提高答案质量
- 适合重要文档、代码审查等场景
- 可以多次迭代检查

## 6. 使用外部工具

### 6.1 使用基于嵌入的搜索来实现高效的知识检索

**核心要点：**
- 结合向量搜索和 LLM
- 从大量文档中快速找到相关信息
- 将检索结果作为上下文提供给模型

**工作流程：**
```
1. 将知识库文档转换为向量嵌入
2. 用户查询也转换为向量
3. 搜索最相关的文档片段
4. 将检索结果 + 用户查询一起发送给 LLM
5. LLM 基于检索结果生成答案
```

**从概率角度理解：**
```
P(准确答案 | 检索到的相关信息) >> P(准确答案 | 无检索)

RAG（检索增强生成）的核心：
- 检索提供相关上下文
- LLM 基于上下文生成
- 减少模型依赖参数记忆
```

**示例：**

```
检索到的相关信息：
"""
[从知识库中检索的 Top 3 文档片段]
"""

基于以上信息，请回答用户的问题：
[用户问题]

如果检索内容不足以回答问题，请说明需要更多信息。
```

**学习笔记：**
- RAG（检索增强生成）的核心技术
- 适合企业知识库、文档问答系统
- 需要平衡检索精度和召回率

### 6.2 利用代码执行进行精确计算或调用外部 API

**核心要点：**
- LLM 不擅长精确计算
- 通过代码执行弥补这一缺陷
- 可以调用 API 获取实时数据

**从概率角度理解：**
```
LLM 直接计算：
P(正确答案 | 计算题) —— 基于统计规律，容易出错

代码执行：
P(正确答案 | 代码执行) ≈ 100% —— 确定性计算
```

**示例（精确计算）：**

```
请编写并执行 Python 代码来解决这个问题：

问题：计算 12345 × 67890 + 98765 ÷ 5

要求：
1. 使用 Python 代码进行计算
2. 展示代码和计算结果
3. 验证结果的准确性
```

**为什么有效：**
- LLM 是概率模型，不擅长确定性计算
- 代码执行是确定性的
- 将概率问题转化为确定性问题

**学习笔记：**
- 代码解释器是强大的扩展工具
- 适合数据分析、可视化、自动化
- 注意代码安全性和沙箱环境

## 7. 系统地测试更改

### 7.1 以标准答案为基准评估模型输出

**核心要点：**
- 建立测试集和标准答案（Golden Set）
- 对比模型输出与标准答案
- 使用量化指标评估质量

**从概率角度理解：**
```
评估的本质：
估计 P(模型输出 ≈ 标准答案 | 提示词)

通过大量测试样本：
准确率 = (1/N) × Σ I(输出 i ≈ 标准答案 i)
```

**评估维度：**

1. **准确性**：答案是否正确
2. **完整性**：是否覆盖所有要点
3. **相关性**：是否切题
4. **简洁性**：是否冗余
5. **格式**：是否符合要求

**示例评估流程：**

```
测试用例：
问题：Python 中列表和元组有什么区别？

标准答案要点：
- 列表可变，元组不可变
- 列表用 []，元组用 ()
- 元组可用作字典键，列表不行
- 元组性能略优

模型输出评估：
□ 提到可变性差异（是/否）
□ 提到语法差异（是/否）
□ 提到使用场景（是/否）
□ 无错误信息（是/否）
```

**学习笔记：**
- 建立全面的测试集覆盖各种场景
- 可以自动化评估部分指标
- 定期回归测试确保质量稳定

### 7.2 A/B 测试不同的提示词版本

**核心要点：**
- 对同一任务设计多个提示词版本
- 随机分配请求到不同版本
- 收集用户反馈和性能指标

**从概率角度理解：**
```
比较不同提示词版本：
P(成功 | 版本 A) vs P(成功 | 版本 B) vs P(成功 | 版本 C)

通过大量样本估计：
P(成功 | 版本) = (成功次数) / (总次数)

选择概率最高的版本
```

**示例：**

```
版本 A：
"请总结这篇文章"

版本 B：
"请用 3 个要点总结这篇文章的核心内容，每个要点不超过 20 字"

版本 C：
"你是一位专业编辑，请提取这篇文章的关键信息，以要点形式呈现"

收集 100 个样本的反馈，比较三个版本的效果。
```

**测试指标：**
- 用户满意度评分
- 任务完成率
- 响应时间
- 后续追问率
- 人工评估质量

**学习笔记：**
- 小改动可能带来大影响
- 数据驱动优化提示词
- 持续迭代改进

## 8. 总结与进阶理解

### 8.1 提示词工程的核心原则（从概率角度）

| 原则 | 概率解释 |
|------|---------|
| **清晰具体** | 降低熵值，概率分布更集中 |
| **结构化** | 改善 Attention 模式，提高条件概率 |
| **示例驱动** | 提供 P(输出\|输入) 的条件分布样本 |
| **角色设定** | 调整先验分布，激活特定知识子空间 |
| **参考依据** | 提供强约束，减少幻觉概率 |
| **分而治之** | 将复杂采样分解为多个简单采样 |
| **深度思考** | 增加中间步骤，提高每步确定性 |
| **工具增强** | 扩展能力空间，提高任务完成概率 |
| **持续优化** | 通过测试估计 P(成功\|提示词) |

### 8.2 三个理解层次

```
Level 1: 表面技巧（死记硬背方法）
         ↓
Level 2: 认知心理学（如何引导模型"思考"）
         ↓
Level 3: 概率论 + 信息论（理解底层数学原理）
```

### 8.3 用概率框架统一理解所有技巧

**所有提示词技巧都在做一件事：**
```
最大化 P(理想输出 | 提示词，上下文)
```

**具体方法：**
1. **增加条件信息**（清晰指令、角色设定）
2. **降低熵值**（详细约束、长度限制）
3. **提供样本**（Few-Shot 示例）
4. **分解任务**（思维链、分步骤）
5. **外部增强**（参考文本、工具调用）
6. **多次采样**（自我检查、A/B 测试）

### 8.4 进阶思考题

基于概率框架理解：

1. **为什么 Self-Consistency 有效？**
   ```
   多次采样取多数：
   P(正确答案) = Σ P(答案 i) × I(答案 i = 众数)
   
   降低单次采样的方差
   ```

2. **为什么 Instruction Tuning 有效？**
   ```
   微调改变了模型参数 θ
   使得 P(next_token | instruction, θ_finetuned) 更准确
   ```

3. **温度（Temperature）的作用？**
   ```
   低温度：P(token) = softmax(logits / 0.2)  分布尖锐
   高温度：P(token) = softmax(logits / 1.0)  分布平滑
   
   提示词工程 + 温度调节 = 控制探索与利用的平衡
   ```

### 9.3 进阶理解类

**Q9: 从数学角度解释，为什么好的提示词能提高输出质量？**

**速答模板：**
```

- 概率分布更集中 → 采样质量更高

从贝叶斯角度：
P(理想输出 | 提示词) ∝ P(提示词 | 理想输出) × P(理想输出)

- 三者协同：后验概率更集中 → 输出质量更高

从分解角度：
复杂任务分解为多个简单步骤：
P(答案 | 问题) → P(答案 | 步骤 n) × ... × P(步骤 1 | 问题)

每一步的条件概率都很高，整体质量自然提升。
```

- 能否从多个角度解释

**Q12: 如何用 LLM 构建一个数据分析助手？**

### 9.5 速记口诀

**提示词工程核心要点（面试速记）：**

```
一清二角三分隔，
四步五例六长度。
参考文本不能少，
拆分任务降难度。

思考时间要给够，
工具调用扩能力。
系统测试持续改，
概率思维是核心。
```

**解释：**
- 一清：清晰指令
- 二角：角色设定
- 三分隔：分隔符
- 四步：分步骤
- 五例：举例说明
- 六长度：指定长度
- 参考文本：提供参考材料
- 拆分任务：复杂任务分解
- 思考时间：思维链
- 工具调用：外部工具
- 系统测试：评估优化
- 概率思维：理解本质

---

**面试加分项：**

✅ 能用概率论解释技巧为什么有效  
✅ 有系统化的方法论而非零散技巧  
✅ 能结合具体场景给出解决方案  
✅ 了解局限性并有应对策略  
✅ 有实际项目经验和量化结果  

❌ 只死记硬背技巧  
❌ 无法解释背后的原理  
❌ 没有实际应用场景  
❌ 过度神话 LLM 能力  

---

**典型面试对话示例：**

**面试官：** "你如何理解提示词工程？"

**优秀回答：**
```
我认为提示词工程的本质是概率工程。

LLM 的核心是 next token prediction，所有提示词技巧
都在影响这个条件概率分布。

比如角色设定，实际上是在调整先验分布，让模型更可能
采样到与角色相关的专业内容。

再比如思维链，是将一个复杂的边缘概率分解为多个
条件概率的乘积，每一步都很确定，整体准确率自然提高。

在实践中，我总结了一套方法论：首先明确任务目标，
然后根据任务特点选择合适的技巧，最后通过 A/B 测试
持续优化。我曾用这个方法将某个任务的准确率从 70% 
提升到了 90%。
```

**为什么优秀：**
- 有理论深度（概率角度）
- 有具体例子（角色设定、思维链）
- 有方法论（系统化流程）
- 有实际成果（70%→90%）
