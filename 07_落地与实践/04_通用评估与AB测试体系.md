# 通用评估与A/B测试体系

本章系统梳理大模型应用评估与A/B测试核心知识，内容组织分为4大模块：三层评估体系、核心评估方法与指标、全链路A/B测试框架、面试高频考点。
旨在解决以下核心问题：大模型应用上线需要做哪些评估？如何避免离线效果好但线上效果差？如何设计科学的A/B测试？
学习完本章，你将：
- 掌握三层评估体系和落地方法
- 熟悉核心评估指标与优化方案
- 掌握全链路A/B测试流程与统计方法
- 熟悉相关面试考点

## 1 三层评估体系
**开发岗要求**：
### 1.1 离线评估（上线前）
- **测试集要求**：覆盖常规场景(70%)、边界场景(20%)、异常场景(10%)，尽可能贴近真实线上数据分布
- **评估指标**：自动计算核心指标（准确率、BLEU/ROUGE/BERTScore等）
- **作用**：快速筛选出效果不达标的版本，避免低效的线上测试
### 1.2 在线小流量评估（灰度阶段）
- **分流方式**：用户维度分流，同一用户始终属于同一个实验组，避免体验不一致
- **评估指标**：核心业务指标（用户满意度、问题解决率、转化率）+ 技术指标（成本、延迟、错误率）
- **作用**：验证线上真实场景下的效果，避免离线效果好线上效果差的问题
### 1.3 人工抽检（上线前后）
- **抽检比例**：核心场景抽检10%~30%，普通场景抽检1%~5%
- **作用**：验证自动评估的准确性，发现自动评估无法识别的问题，比如语义通顺性、逻辑合理性等

**开发岗最佳实践**：
- 必须经过三层评估都达标才能全量上线
- 测试集需要定期更新，覆盖新的用户场景
- 自动评估指标和人工评估结果需要定期对齐，保证自动评估的可靠性

## 2 核心评估方法与指标
**算法岗要求**：
### 2.1 常用评估指标
| 指标类型 | 指标 | 适用场景 |
|----------|------|----------|
| 自动指标 | BLEU/ROUGE | 文本生成、翻译等有参考文本的场景 |
|  | BERTScore | 语义相似度评估，和人类判断相关性高 |
|  | 准确率/召回率/F1 | 分类、匹配类场景 |
| LLM-as-Judge | 通用评分 | 所有场景，和人工标注一致性达85%+ |
| 业务指标 | 用户满意度、问题解决率、转化率 | 最终业务效果评估 |

### 2.2 2026评估优化方案
- **小模型评估**：用微调后的小模型做LLM-as-Judge，成本比大模型低80%，准确率可达大模型的90%以上，是目前主流方案
- **多模型投票**：多个评估模型投票，减少单个模型的评估偏差
- **自动评估校准**：定期用人工标注数据校准自动评估模型，保证评估稳定性

### 2.3 离线线上效果对齐方法
1. **分布对齐**：保证离线测试集和线上数据分布一致，避免分布漂移导致效果不一致
2. **指标对齐**：离线评估指标和线上业务指标对齐，离线指标提升对应线上业务指标提升
3. **小流量验证**：必须经过小流量验证效果达标才能全量上线，避免盲目全量上线出问题

## 3 全链路A/B测试框架
**开发岗要求**：
### 3.1 核心流程
1. **实验设计**：明确实验假设、核心指标、样本量、实验周期
2. **流量分配**：用户维度随机分流，实验组和对照组用户特征一致，流量比例根据实验需要设置（通常10%~50%）
3. **数据收集**：收集两组的所有指标数据，排除异常数据（比如极端值、作弊数据）
4. **统计分析**：做显著性检验，判断实验组效果是否显著优于对照组
5. **决策**：效果显著优于对照组则全量发布，否则回滚迭代优化

### 3.2 关键注意事项
- 不要同时测试多个变量，每次实验只改一个因素，避免无法定位效果原因
- 实验周期至少覆盖完整的用户行为周期，比如7天，避免短期波动影响
- 不要中途调整流量比例，避免统计偏差

**算法岗要求**：
### 3.3 统计显著性检验
| 指标类型 | 检验方法 | 显著性阈值 |
|----------|----------|------------|
| 分类指标（准确率、解决率） | 卡方检验 | p < 0.05 |
| 连续指标（延迟、成本） | t检验 | p < 0.05 |
- p值<0.05表示实验组和对照组的差异由随机波动导致的概率<5%，认为效果显著
- 功效(Power)>0.8表示实验有足够的统计能力检测到真实的效果提升

### 3.4 常见统计偏差规避
1. **辛普森悖论**：由于混杂变量（用户组成，例如新老用户比例、提问场景差异、提问难度等）在对照组和实验组的分布不均匀，导致整体数据趋势和分组数据趋势相反，需要按用户分层分析
2. **多重比较偏差**：当同时测试多个实验组、或同时看多个指标时，即使所有实验组都没有真实效果，只要比较的次数足够多，总有概率出现看起来效果统计显著的假阳性结果。可以只明确少量核心指标、重复验证、或者统计校正（FDR错误发现率校正）来避免
3. **新奇效应**：新功能刚上线时用户好奇导致效果提升，需要观察长期效果，通常需要观察7~14天

## 4 面试高频考点
### 开发岗通用题
#### Q1：**大模型应用上线需要做哪些评估？为什么离线效果好但线上效果差？**
- 需要三层评估：离线评估→在线小流量评估→人工抽检
- 离线好线上差的常见原因：
  - 离线测试集和线上数据分布不一致
  - 离线评估指标和线上业务指标不匹配
  - 线上环境更复杂，有异常请求、高并发等离线没有覆盖的场景
  - 存在新奇效应，用户刚接触新功能的行为和长期行为不一致
- 解决方法：保证测试集贴近线上数据，必须做小流量验证，不要直接全量上线

#### Q2：**大模型A/B测试和传统互联网产品的A/B测试有什么区别？需要注意什么？**
- 最大的区别：大模型的输出有随机性，需要更大的样本量才能得到统计显著的结果；其次大模型实验的观测周期更长，需要覆盖完整的用户行为周期
- 注意事项：每次实验只改一个变量，用户维度分流，做统计显著性检验，不要只看均值就下结论，观察至少7天的长期效果

### 算法岗专项题
#### Q1：**LLM-as-Judge有什么优缺点？如何提升评估的准确性？**
- 优点：可以评估语义、逻辑等传统指标无法衡量的维度，和人工标注一致性达85%+，不需要大量人工标注
- 缺点：成本高、有评估偏差、速度慢
- 提升准确性的方法：
  - 用多个模型投票，减少单个模型的偏差
  - 给评估模型提供清晰的评估标准和Few-shot示例
  - 用微调后的小模型做评估，成本低效果好
  - 定期用人工标注数据校准评估模型

#### Q2：**如何判断A/B测试的效果提升是真实的还是随机波动？**
通过统计显著性检验：
- 首先保证样本量足够，统计功效>0.8
- 用合适的检验方法，分类指标用卡方检验，连续指标用t检验
- p值<0.05认为差异统计显著，由随机波动导致的概率<5%
- 同时看提升幅度，即使统计显著，如果提升幅度太小没有业务价值也不值得上线
- 还要观察至少7天的长期效果，避免短期新奇效应
