本章系统梳理Claude Code生产级Agent架构设计核心知识，内容组织分为4大模块：核心技术栈、核心Agent架构设计思路、可复用Agent设计亮点、Agent设计面试速答。所有内容均来自2026年Claude Code泄露源码分析，为生产级Agent落地的成熟参考方案。

## 1 核心技术栈
- 运行时：Bun，原生支持打包和异步高并发，构建时可通过feature flag死代码消除未发布功能
- 核心协议：MCP工具扩展协议，原生对接任意外部工具
- 底层依赖：Anthropic官方SDK，全链路支持流式响应
- 类型校验：Zod + TypeScript严格模式，保证输入输出可靠性

## 2 核心Agent架构设计思路
### 2.1 Agent Loop核心设计
- 基于经典ReAct模式工程化实现：用户输入→流式调用LLM→解析工具调用→执行工具→结果回填→循环直至无工具调用
- 状态机设计：每次迭代只读当前状态，迭代结束整体替换，保证状态一致性，避免跨迭代污染
- 长任务续跑机制：通过token预算跟踪自动续跑长任务，新增进度提示nudge，连续无进展时自动终止，解决长任务提前收尾问题
- 流式响应优先：全链路采用异步生成器实现，LLM输出、工具执行进度、最终结果均流式同步给用户

### 2.2 工具调用体系设计
- 权限优先设计：工具调用链路内置不可绕过的权限校验管道，支持细粒度规则配置（如允许所有git命令，禁止无确认的rm操作）
- 流式并发执行：LLM还在输出token时就已开始执行已解析完成的工具调用，只读工具可并行，写入工具串行，大幅降低感知延迟
- 错误熔断机制：单个工具执行失败时自动取消所有并行兄弟工具，避免无效资源消耗，工具结果按LLM调用顺序返回，保证上下文一致性
- 大结果降级处理：工具返回结果超过阈值时自动落地磁盘，上下文仅保留预览，模型可按需读取完整内容，避免撑爆上下文窗口

### 2.3 全链路内存/上下文管理（核心）
设计目标：解决LLM上下文窗口有限的痛点，覆盖从单会话工具结果处理、上下文压缩、到跨会话长期记忆、多Agent状态共享全链路内存需求
核心原则：分层递进式流水线，下层是上层的输入，低层成功处理则阻止高层触发，优先用最低成本解决问题；优先保证Prompt Cache稳定性，实际运行缓存命中率达92%，降低81%API成本
执行总流程：`工具结果预处理→单会话上下文管理→跨会话记忆生产→跨会话记忆治理→多Agent内存共享`

#### 2.3.1 Layer1：Tool Result Storage（工具结果存储）
- 执行时机：每轮工具调用返回结果后立即执行
- 核心输入：工具返回的原始结果
- 输出结果：大结果落地磁盘，上下文仅保留固定2KB预览；生成全局`ContentReplacementState`冻结替换规则，保证每轮Prompt前缀字节完全一致
- 作用：从源头减少大工具结果的上下文占用，同时最大化缓存命中率，是所有上层的基础

#### 2.3.2 Layer2：Microcompact（微压缩）
- 执行时机：每轮API请求发送前自动执行
- 核心输入：Layer1处理后的上下文、服务端缓存状态
- 输出结果：清理后的上下文，不修改本地消息历史
- 核心策略：三种零成本清理不破坏缓存：① 空闲60分钟搭缓存过期便车清理旧结果；② API`cache_edits`指令直接修改服务端缓存；③ API层自动清理旧工具结果/思考块，仅保留核心内容

#### 2.3.3 Layer3：Session Memory（会话内存）
- 执行时机：会话过程中异步后台执行，满足「token增长阈值 + 工具调用次数阈值/上轮无工具调用」时触发
- 核心输入：当前会话的所有历史消息
- 输出结果：结构化会话笔记（任务、文件、工作流、错误记录等），持久化到会话内存文件
- 作用：提前预生成压缩素材，需要压缩时直接复用，无需临时调用LLM，成本比全量压缩低一个数量级，是Layer4的优先输入

#### 2.3.4 Layer4：Full Compaction（全量压缩）
- 执行时机：上下文token量超过阈值（窗口大小 - 13K缓冲）时主动触发，优先用Layer3生成的会话笔记做摘要，没有可用笔记时才处理全量历史
- 核心输入：优先读Layer3的会话笔记，无可用笔记时读全量会话历史
- 输出结果：结构化压缩摘要，用户消息100%原文保留不压缩
- 熔断保护：连续3次压缩失败后本会话停止尝试，避免无效API浪费，压缩后自动恢复最近文件、技能、计划等核心上下文

#### 2.3.5 Layer5：Auto Memory Extraction（自动记忆提取）
- 执行时机：单会话完全结束后立即触发
- 核心输入：当前会话的全量历史、生成的会话笔记
- 输出结果：四类结构化长期记忆（用户偏好、反馈信息、项目信息、参考资源），持久化到项目`memory/`目录，和主Agent写操作互斥避免重复
- 作用：跨会话记忆的上游生产环节，输出是Layer6的唯一输入

#### 2.3.6 Layer6：Dreaming（做梦/记忆巩固）
- 执行时机：定期后台触发（积累足够多的记忆/会话后执行，10分钟节流限制）
- 核心输入：Layer5历史生成的所有记忆文件、全部历史会话日志、当前代码库状态
- 输出结果：整理后的记忆库，合并重复内容、更新过期信息、删除矛盾记忆，修剪`MEMORY.md`索引保持在200行/25KB以内，PID锁文件防止并发执行冲突
- 作用：跨会话记忆的下游治理环节，保证记忆库的准确性和精简性，不会凭空生成新记忆，所有内容都来自Layer5的输出

#### 2.3.7 Layer7：Cross-Agent Communication（跨Agent通信）
- 执行时机：多Agent协作场景下按需触发
- 核心输入：前6层的所有状态、记忆、上下文
- 输出结果：Forked Agent隔离运行环境、跨Agent消息传递、多Agent共享记忆状态
- 核心设计：子Agent隔离可变状态避免污染主会话，同时复用父Agent Prompt Cache前缀降低成本；支持点对点/广播通信，内置生命周期控制消息，Agent状态支持跨调用持久化

#### 2.3.8 七层架构速记流程
`每轮工具调用→Layer1存结果→每轮请求前→Layer2清垃圾→会话过程中→Layer3做笔记→上下文不够→Layer4做压缩→会话结束→Layer5提记忆→定期→Layer6理记忆→多Agent协作→Layer7做共享`

### 2.4 多Agent协作设计
- **Forked Agent模式**：所有后台任务（摘要、记忆提取等）均采用子Agent fork模式，子Agent隔离可变状态避免污染主会话，同时复用父Agent的Prompt Cache前缀，大幅降低成本
- **Task协作机制**：多Agent通过文件存储的Task对象协作，内置文件锁保证任务原子认领，支持依赖阻塞、状态变更通知，天然适配分布式多Agent场景
- **跨Agent通信**：支持点对点、广播两种通信模式，内置生命周期控制消息（如计划审批、关机通知），支持同进程、跨进程、跨会话三种路由

### 2.5 长期记忆体系设计
- **会话记忆**：单会话内自动结构化提取任务、文件、错误、工作流等信息，作为压缩时的低成本摘要
- **跨会话记忆提取**：每个会话结束后自动提取长期记忆（用户偏好、反馈、项目信息、参考资源），持久化到项目记忆目录
- **Dreaming（记忆巩固）机制**：后台异步进程，类生物睡眠记忆巩固逻辑，定期合并多会话记忆，去重、更新过期信息、修剪索引，保证跨会话记忆的一致性和有效性，运行时仅允许只读操作，不会破坏已有数据

## 3 可复用Agent设计亮点
- **缓存优先思想**：所有设计优先保证Prompt Cache前缀一致性，将缓存命中率作为核心指标，大幅降低运行成本
- **分层防御架构**：所有核心链路（上下文、错误、权限）均采用分层设计，低成本层先处理，失败再降级到高成本层，兼顾效率和可靠性
- **优雅降级能力**：所有子系统均内置降级逻辑，失败后静默切换到下一层方案，不影响用户使用
- **多Agent原生支持**：Task协作、fork模式、跨Agent通信均为原生设计，而非后期扩展，天然支持多Agent分工场景
- **可扩展性设计**：MCP协议支持任意工具扩展，Hook覆盖30+生命周期节点，Skill系统支持自定义工作流，无需修改核心代码即可扩展能力

## 4 Agent设计面试速答
### 通用题
#### Q1：**大语言模型Agent的上下文管理应该怎么设计？**
参考Claude Code的7层分层架构，优先保证Prompt Cache稳定性，从低到高依次为：
- 工具结果预处理
- 零成本清理
- 会话内存预生成摘要
- 全量LLM摘要
- 跨会话记忆提取
- Dreaming记忆巩固
- 多Agent内存共享
兼顾单会话效率和长期记忆能力。

#### Q2：**多Agent协作的核心设计思路是什么？**
核心是隔离+共享：
- 子Agent隔离状态避免污染，共享Prompt Cache降低成本
- 通过原子Task和标准化通信协议实现协作，不需要中心化调度，天然支持分布式部署

#### Q3：**Agent的长期记忆怎么实现？**
参考三级记忆体系：
- 单会话记忆自动提取
- 会话结束生成持久记忆
- 后台Dreaming进程异步合并巩固，避免记忆冗余和冲突，同时控制记忆检索成本
