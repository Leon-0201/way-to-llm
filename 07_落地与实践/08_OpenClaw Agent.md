# OpenClaw 核心知识梳理
## 一、核心机制1：智能体完整工作流程
智能体循环是将单条用户消息转化为动作和回复的完整端到端路径，全程保持会话状态一致性：
### 1. 入口层
- 对外暴露两种调用方式：Gateway RPC（`agent`/`agent.wait`）、CLI `agent` 命令
- 异步调用立即返回`runId`，`agent.wait`可等待执行结果

### 2. 调度层
- 按会话维度串行执行，避免工具调用/会话状态竞争
- 支持三种排队模式：`collect`/`steer`/`followup`

### 3. 准备层
- 工作区初始化：可选沙箱隔离，重定向到独立沙箱工作区
- 资源加载：Skill 快照加载、引导文件（AGENTS.md/SOUL.md等）解析注入
- 会话锁获取，保证串行执行

### 4. 核心执行循环
1. **提示词组装**：基础提示词 + Skill 提示词 + 引导上下文 + 运行时覆盖项，支持模型特定Token限制，自动注入最近2天的短期记忆和长期记忆
2. **模型推理**：支持流式输出推理过程和最终回复，若用户明确要求记忆某信息，智能体主动调用记忆工具更新对应记忆文件
3. **工具调用**：工具执行过程全程流输出，结果自动清理大小和图像负载
4. **记忆自动刷新触发**：当上下文接近窗口阈值即将触发压缩前，自动执行静默轮提醒智能体将本轮重要信息写入记忆文件，避免压缩丢失
5. **回复整形**：过滤静默令牌（`NO_REPLY`）、去重工具回复、错误兜底
6. **状态持久化**：会话历史/元数据落盘，若启用`session-memory` Hook，会话关闭/重置时自动将本次会话摘要写入每日记忆文件

### 5. 异常与生命周期
- 超时控制：`agent.wait`默认30秒，智能体运行默认48小时，LLM空闲超时默认120秒
- 自动重试：上下文溢出时自动触发压缩后重试
- 提前终止场景：超时、取消信号、RPC断开

## 二、核心机制2：Hook 扩展体系
Hook 是 OpenClaw 的事件驱动扩展点，可在生命周期特定节点执行自定义逻辑，分为两大类：

### 1. 内部 Hook（Gateway Hook）
- **触发时机**：命令和生命周期事件，例如 `/new`/`/reset` 命令、会话压缩、网关启动等
- **核心事件类型**：
  | 事件 | 触发时机 |
  |------|----------|
  | `command:new`/`command:reset` | 执行新建/重置会话命令时 |
  | `agent:bootstrap` | 注入工作区引导文件前，可修改注入文件列表 |
  | `session:compact:before`/`session:compact:after` | 压缩前后 |
  | `message:received`/`message:sent` | 消息收发节点 |
- **内置常用Hook**：
  - `session-memory`：会话关闭时自动保存上下文到每日记忆
  - `command-logger`：命令操作日志
  - `boot-md`：网关启动时执行BOOT.md检查清单

### 2. 插件 Hook
- **触发时机**：更底层的智能体/网关生命周期内部节点，适合深度集成
- **核心Hook点**：
  | Hook点 | 作用 |
  |--------|------|
  | `before_prompt_build` | 提示词提交前注入自定义上下文 |
  | `before_tool_call`/`after_tool_call` | 拦截工具参数/结果，支持阻断高危操作 |
  | `before_agent_reply` | LLM返回后接管回复，支持合成回复/静默本轮 |
  | `before_compaction`/`after_compaction` | 压缩流程前后干预 |
- **优先级规则**：阻断操作（`block: true`/`cancel: true`）是终局决定，低优先级Hook无法覆盖

### 3. Hook 开发与优先级
- 结构：每个Hook是独立目录，包含`HOOK.md`（元数据）和`handler.ts`（逻辑）
- 加载优先级（从低到高）：内置Hook < 插件Hook < 托管Hook < 工作区Hook

## 三、核心机制3：记忆与上下文管理
### 1. 三层记忆架构（纯文件存储，无隐藏状态）
| 记忆类型 | 存储位置 | 加载规则 | 作用 |
|----------|----------|----------|------|
| 长期记忆 | `MEMORY.md` | 每个私有会话启动自动加载 | 保存持久事实、偏好、决策 |
| 短期记忆 | `memory/YYYY-MM-DD.md` | 今天和昨天的笔记自动加载 | 每日上下文和观察记录 |
| 实验性记忆 | `DREAMS.md` | 按需加载 | Dreaming流程的审查摘要 |

### 2. 记忆核心能力
- **记忆工具**：`memory_search`（语义+关键词混合搜索）、`memory_get`（读取指定文件）
- **自动记忆刷新**：压缩前自动触发静默轮，提醒智能体保存重要信息到记忆文件，避免上下文丢失
- **Dreaming机制（可选）**：后台定期整合短期记忆，高价值内容自动升级到长期记忆，保持记忆高信噪比，结果可人工审查

### 3. 上下文窗口治理
#### （1）剪枝（Pruning）
- 作用：每次LLM调用前裁剪内存中的旧工具结果，不修改磁盘会话历史
- 适用场景：减少工具输出导致的上下文膨胀，降低缓存成本
- 默认策略：Anthropic模型自动启用，TTL 5分钟，保留最近3轮完整内容

#### （2）压缩（Compaction）
- 作用：上下文接近窗口限制时，将较早对话总结为单条摘要，写入会话转录
- 核心特性：
  - 自动触发：上下文溢出时自动压缩并重试
  - 支持指定压缩模型：可使用大模型生成更高质量摘要
  - 工具调用配对：压缩时保持工具调用和结果配对不拆分
- 手动触发：`/compact [摘要方向]` 强制压缩

| 特性 | 剪枝 | 压缩 |
|------|------|------|
| 操作对象 | 仅工具结果 | 整个对话历史 |
| 是否持久化 | 否（仅内存生效） | 是（写入会话转录） |
| 效果 | 移除冗余输出 | 总结历史内容 |

## 四、核心机制4：多智能体协作
OpenClaw 原生支持多智能体编排，通过会话工具实现跨会话协作和子智能体管理：


- **隔离性**：子会话拥有独立上下文、沙箱、工作区，互不干扰
- **编排能力**：支持层级编排，深度>=2的编排子智能体可继续 spawn 下一级子智能体，叶子节点无递归编排能力
- **结果投递**：子智能体完成后自动将结果推送给发起会话
- **安全可见性**：默认可见范围为当前会话+其启动的子树，沙箱会话强制限制为tree级别，可配置为agent/all级别

## 五、核心能力与架构
### 1. 工作区设计
- 标准文件结构：AGENTS.md（操作规则）、SOUL.md（人设语气）、TOOLS.md（本地工具约定）、USER.md（用户信息）、IDENTITY.md（智能体身份）
- 沙箱隔离：非主会话可启用沙箱，在独立目录运行，避免影响主机
- Git备份：工作区支持Git版本化，记忆和配置可私有仓库备份

### 2. Skill 系统
- 能力模块化：Skill 是独立功能包，按需加载
- 提示词轻量化：默认仅注入Skill元数据（名称+描述+位置），具体指令按需读取，避免上下文浪费

### 3. 提示词系统
- 三层结构：固定规则部分 + Skill列表 + 注入的工作区引导文件
- 子智能体优化：子会话自动使用`minimal`提示词模式，省略非必要部分，降低上下文开销
- 缓存友好：系统提示词尽量稳定，减少缓存失效成本

## 六、高频面试考点总结
1. **流程类**：描述OpenClaw智能体从接收到消息到返回回复的完整执行路径
2. **Hook类**：两种Hook类型的区别、适用场景，常用内置Hook的作用
3. **记忆类**：三层记忆架构的设计思路，自动记忆刷新和Dreaming机制的作用
4. **上下文治理**：剪枝和压缩的差异，各自解决的核心问题
5. **多智能体**：子智能体的隔离机制、编排方式、可见性控制逻辑
6. **生产级特性**：会话串行化、沙箱隔离、权限控制等企业级设计的考量
