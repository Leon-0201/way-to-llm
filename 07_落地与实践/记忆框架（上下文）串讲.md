# AI Agent记忆框架串讲

本章系统梳理AI Agent记忆框架核心知识，内容组织分为两大模块：跨框架核心维度总结、主流框架深度解析。

旨在解决以下核心问题：不同记忆框架的核心设计思路差异？记忆内容/更新/存储/使用各维度有哪些方案？如何根据场景选型合适的记忆框架？

学习完本章，你将：
- 掌握记忆内容、更新、存储、使用四大核心维度的设计思路
- 熟悉主流记忆框架的核心架构、特点与适用场景
- 能够根据业务需求进行记忆框架的选型与落地

---

## 1 跨框架核心维度总结

### 1.1 记忆内容

该维度汇总了不同框架会持久化存储的记忆内容，覆盖用户、任务、Agent等不同层级的信息。

#### 1.1.1 用户层记忆

- **用户个性化偏好与长期事实**：存储用户的习惯、喜好、个人信息等长期稳定的个性化数据，支持框架包括：Mem0、Memobase、Engram、OpenClaw、Hermes Agent、Letta、Zep、Graphiti、Cognee
- **用户的决策风格与思考模式建模**：对用户的思维方式、决策偏好进行深层建模，支持框架包括：Honcho、Hermes Agent
- **用户长期行为轨迹与行为模式**：记录用户的长期行为序列，归纳行为规律，支持框架包括：Hindsight

#### 1.1.2 任务/技能层记忆

- **任务执行经验与可复用工作流技能**：沉淀Agent完成任务的步骤、方法，形成可复用的技能，支持框架包括：Hermes Agent、LangMem、DeepAgent
- **任务高层摘要、决策rationale与当前子目标**：对任务过程进行抽象，记录核心决策与当前进度，支持框架包括：DeepAgent、GAM
- **工具调用记录、问题修复经验与错误模式**：沉淀工具使用的经验、问题解决的方法，支持框架包括：Engram、Hermes Agent

#### 1.1.3 知识/关系层记忆

- **实体间的结构化关系与知识图谱节点**：抽取交互中的实体与关联，构建结构化的知识网络，支持框架包括：Graphiti、Zep、Cognee、Mem0、Engram
- **带时间戳的演化型事实**：记录事实的时间属性，支持追溯不同时间点的状态，支持框架包括：Graphiti、Zep、Hindsight

#### 1.1.4 系统/Agent层记忆

- **原始会话交互日志与工作记录**：完整保留原始的对话、操作日志，用于审计与回溯，支持框架包括：OpenClaw、Claude Code、Letta、Memary
- **Agent自身的身份定义与行为规范**：存储Agent的身份设定、行为准则等核心配置，支持框架包括：OpenClaw、Claude Code、Letta
- **模型内部激活值与参数级记忆**：深入模型内部，存储激活状态与权重级的记忆，支持框架包括：MemOS

### 1.2 记忆更新

该维度汇总了记忆的写入、更新、演化的机制，覆盖从简单存储到复杂的自主迭代策略。

#### 1.2.1 写入策略

- **原始内容直接追加写入**：不对输入做预处理，直接将原始内容追加到日志中，支持框架包括：OpenClaw、Claude Code
- **LLM自动抽取关键事实与摘要**：通过大模型从原始交互中提取核心信息，压缩冗余内容，支持框架包括：Mem0、Memobase、GAM、DeepAgent
- **Agent自主编辑记忆内容**：Agent通过专用工具，自行修改、调整记忆内容，支持框架包括：Letta、DeepAgent

#### 1.2.2 异步/后台处理

- **后台异步语义富集与增量更新**：在后台异步任务中，对记忆进行语义补全、关联增强，不阻塞主交互，支持框架包括：Cognee、LangMem
- **Dreaming后台沉淀**：定期运行后台任务，扫描历史日志，自动将高价值内容从短期记忆升级沉淀为长期记忆，支持框架包括：OpenClaw、Claude Code
- **双模型管线更新**：通过两个模型协作，Deriver负责提取事实，Dialectic负责辩证推理完成记忆更新，支持框架包括：Honcho
- **4路并行更新与行为推理适配**：从向量、关键词、时间、行为四个维度并行处理记忆更新，适配行为推理需求，支持框架包括：Hindsight
- **技能自我迭代更新**：Agent在使用已有技能的过程中，自动优化、修改技能文档，让经验持续迭代，支持框架包括：Hermes Agent

#### 1.2.3 冲突与演化管理

- **时序式更新**：遇到冲突事实时，不覆盖旧内容，仅将旧事实标记为失效，新增新的事实记录，保留完整历史，支持框架包括：Graphiti、Zep
- **自动冲突检测与矛盾合并**：写入时自动检测重复、矛盾的记忆，进行去重或合并处理，支持框架包括：Mem0、LangMem

#### 1.2.4 生命周期管理

- **基于遗忘曲线的自动过期清理**：基于Ebbinghaus遗忘曲线，为记忆设置TTL，自动清理过期、低价值的记忆，支持框架包括：Engram、MemOS
- **触发式自动记忆Checkpoint**：根据消息数、时间等触发条件，自动保存会话的记忆快照，支持框架包括：Engram
- **读取时刷新记忆衰减**：当记忆被访问时，重置其衰减计时器，模拟生物记忆的间隔重复效应，让常用记忆更持久，支持框架包括：Engram

### 1.3 记忆存储

该维度汇总了不同框架采用的底层存储方案，适配不同的记忆访问模式。

#### 1.3.1 单层存储方案

- **纯文件系统**：以人类可读的Markdown文件存储记忆，支持Git版本控制，方便人工查看与编辑，支持框架包括：OpenClaw、Letta、DeepAgent
- **SQLite数据库**：轻量本地数据库，内置FTS5全文索引，支持快速的关键词检索，支持框架包括：Hermes Agent、Engram
- **向量数据库**：存储文本的嵌入向量，支持语义相似度检索，适配非结构化记忆的召回，支持框架包括：Mem0、LangMem、Memary、Letta、Zep、Graphiti、Cognee
- **图数据库/时序知识图谱**：存储实体、关系与时序信息，支持结构化关系查询与时序推理，支持框架包括：Graphiti、Zep、Cognee、Mem0、Engram
- **键值存储**：轻量KV存储，支持快速读写，可对接Redis实现分布式缓存，支持框架包括：LangMem、Memary
- **模型内部存储**：直接操作模型内部的KV-cache与LoRA权重，存储参数级的记忆，支持框架包括：MemOS

#### 1.3.2 混合存储方案

- **混合多存储架构**：组合多种存储介质，针对不同类型的记忆采用不同的存储方案，兼顾灵活性与性能，支持框架包括：Mem0、Cognee、Engram
- **Agent文件系统**：为Agent提供结构化的文件式存储抽象，统一管理记忆的组织与访问，支持框架包括：GAM

### 1.4 记忆使用

该维度汇总了记忆在Agent推理过程中的调用、注入、应用的机制。

#### 1.4.1 检索策略

- **混合多维度检索**：组合语义、关键词、时间、行为等多个维度的召回策略，提升记忆检索的准确性，支持框架包括：Hindsight、Engram、OpenClaw
- **Agent主动决策检索**：由Agent自主判断当前任务是否需要检索记忆，按需加载相关内容，而非自动注入，支持框架包括：Hermes Agent、Letta
- **路径编码的条件注入**：利用文件系统的目录结构，根据当前工作目录，自动加载对应范围的规则记忆，用路径编码相关性，支持框架包括：Claude Code
- **知识图谱推理**：通过图遍历分析实体间的关系，支持多跳的复杂知识推理，支持框架包括：Graphiti、Zep、Cognee
- **时序推理**：基于记忆的时间轴，追溯事实的演化过程，支持查询历史状态、分析变化趋势，支持框架包括：Graphiti、Zep
- **双Agent协作检索**：通过两个Agent协作，Memorizer负责突出关键历史信息，Researcher负责深度检索与集成，支持框架包括：GAM

#### 1.4.2 上下文构建

- **Token预算感知的上下文构建**：自动根据当前的Token预算，筛选最相关的记忆，组装成符合长度限制的上下文，支持框架包括：Engram
- **渐进式披露**：平时仅加载记忆的名称、描述等轻量索引，当判断当前任务与该记忆相关时，再加载完整的记忆内容，避免占用过多Token，支持框架包括：Hermes Agent
- **Prefix Cache优化**：将固定的记忆层做缓存复用，降低重复的推理成本，提升响应速度，支持框架包括：Claude Code、Hermes Agent

#### 1.4.3 协作与进化

- **跨Agent共享记忆**：支持多Agent共享同一记忆库，实现上下文同步与团队协作，支持框架包括：Zep、Engram
- **自动Prompt优化**：自动从交互历史中提取经验，更新Agent的核心指令，避免重复犯同样的错误，实现程序性记忆的迭代，支持框架包括：LangMem
- **记忆状态感知推理**：将记忆系统的状态（如剩余Token预算）注入Agent的推理上下文，让Agent可以自主调度记忆的使用，支持框架包括：Claude Code

## 2 主流框架深度解析

### 2.1 OpenClaw

> **方案性质**：完整的人机协作Agent框架，本身是独立的Agent产品，其文件化记忆系统是核心能力，记忆文件为通用Markdown格式，可被其他Agent直接读取使用。

OpenClaw是一个面向人机协作的Agent框架，它的核心设计哲学是**可见性压倒一切**，将所有记忆都以人类可读的文件形式存储，实现了完全透明的记忆管理。

#### 2.1.1 核心架构

OpenClaw采用**文件驱动的三层记忆架构**，所有的记忆都持久化到磁盘的Markdown文件中：

- **Soul层**：Agent的核心世界观、基础规则，是Agent永远不会忘记的常识，是Agent的底层认知，会话开始时直接注入系统提示词
- **User层**：用户的个人画像、工作习惯、偏好设置，存储在`USER.md`中
- **File-based Memory层**：所有的长期协作记忆，存储在`MEMORY.md`中，同时搭配`sqlite-vec`向量存储实现语义检索
  - 长期协作记忆 - 和用户协作过程中提炼的长期任务信息、关键决策
  - 语义向量记忆 - 用于语义检索的嵌入向量，支撑高效的记忆召回

#### 2.1.2 核心特点

- **完全透明的记忆**：所有记忆都是人类可读的Markdown文件，你可以直接用文本编辑器查看、修改、删除Agent的记忆
- **Git可回溯**：记忆文件支持Git版本管理，记忆的变更可完全追溯
- **高级检索能力**：内置语义搜索、混合搜索、时间衰减、MMR重排等高级检索能力
- **人机协作接口**：记忆文件本身就是人机协作的接口，你手动修改记忆文件，Agent的认知会立刻同步

#### 2.1.3 记忆管理细节

**存储方式**

- 核心的结构化记忆，全部以Markdown文件的形式存储在磁盘的`~/.openclaw/`目录下
- 向量记忆使用`sqlite-vec`进行存储，支持语义相似度检索
- 所有记忆文件都支持手动编辑，完全对用户透明

**更新方式**

采用文件驱动的自动更新机制：

- 每次交互完成后，Agent会自动提炼新的关键信息，更新`MEMORY.md`文件
- 同步将新的记忆生成嵌入向量，更新向量存储
- 自动合并重复记忆、清理过期信息，保证记忆文件的精简

**记忆的使用（加载）方式**

OpenClaw的记忆加载采用文件读取+语义检索的混合模式：

- 每次会话启动时，会自动读取所有的Markdown记忆文件，将核心记忆注入到系统提示词中
- 处理用户查询时，会自动对查询进行语义解析，从向量库中召回相关的长期记忆
- 将检索到的记忆按照相关性排序，精简后注入到LLM的prompt中，同时自动控制长度避免上下文溢出

### 2.2 Claude Code

> **方案性质**：完整的编程专用Agent，是Anthropic推出的面向编程场景的独立Agent，记忆系统内置且专门为编程场景优化，同时支持插件扩展，可接入外部记忆系统。

Claude Code是Anthropic推出的编程Agent，它的记忆系统是专门为编程场景设计的，模拟**人类的工作记忆-长期记忆-记忆巩固**的认知流程，实现了项目级的持久记忆。

#### 2.2.1 核心架构

Claude Code采用**四层认知架构**，同时搭配后台的`autoDream`记忆巩固进程：

- **全局规则层**：所有项目共享的全局规则，是Agent的基础配置
- **项目规则层**：当前项目的技术栈、架构、团队规范，跟随项目Git共享
- **会话工作记忆**：当前会话内的临时工作笔记、关键决策、踩过的坑
- **长期持久记忆**：跨会话的用户偏好、项目状态、用户反馈、外部引用

同时，后台的`autoDream`进程，会在Agent空闲时，自动完成记忆的巩固整理，类似人类的睡眠记忆整理。

#### 2.2.2 核心特点

- **项目级隔离**：不同项目的记忆完全隔离，不会出现项目间的记忆混淆
- **后台自动巩固**：空闲时自动整理记忆，合并重复、清理过期，保证记忆的有效性
- **可解释的记忆**：所有记忆都有明确的来源，你可以查看每个记忆的生成时间、来源
- **面向编程优化**：专门针对编程场景优化，支持项目状态、代码上下文的记忆

#### 2.2.3 记忆管理细节

**存储方式**

- 记忆以Markdown文件的形式，存储在项目的`~/.claude/projects/{project}/memory/`目录下
- 用`MEMORY.md`作为记忆的索引文件，最多保留200条记忆的指针
- 底层使用SQLite进行存储，支持全文检索

**更新方式**

采用触发式写入+后台巩固的更新机制：

- 记忆不是每次交互都写入，只有当用户纠正Agent、或者Agent发现有价值的新信息时，才会写入记忆
- 后台的`autoDream`进程，在空闲时自动处理记忆：合并重复记忆、将相对日期转换为绝对日期、删除过期的旧记忆、修剪`MEMORY.md`保证其精简
- 自动将新的记忆加入到索引中，保证检索的有效性

**记忆的使用（加载）方式**

Claude Code的记忆加载采用索引加载+主动召回的模式：

- 每次会话启动时，会自动加载项目的`MEMORY.md`索引，以及项目规则
- 处理用户查询时，Agent会主动从记忆库中检索相关的记忆，判断哪些信息对当前任务有用
- 将检索到的记忆整理为结构化的上下文，注入到LLM的prompt中
- 会话内的工作记忆，会在对话压缩时自动带入到新的上下文中，保证会话内的连续性

### 2.3 Hermes Agent

> **方案性质**：完整的全栈Agent框架，是独立的Agent产品，记忆系统支持插件化扩展，可接入Mem0、Honcho等外部记忆Provider，记忆模块也可独立对接其他Agent使用。

Hermes Agent是一个开源的全栈Agent框架，它的记忆系统采用**分层+插件化**的设计，支持记忆的自动进化，同时支持外部的记忆能力扩展。

#### 2.3.1 核心架构

Hermes Agent采用**五层进化记忆架构**，同时支持插件化的记忆Provider：

- **L1核心记忆**：常驻的核心记忆，存储在`MEMORY.md`和`USER.md`中，每次会话都会注入。
  - Agent的核心配置、关键规则
  - 用户的沟通风格、技术栈偏好、工作习惯
- **L2长时记忆**：所有过往交互的摘要，存储在SQLite中，支持全文检索，支持跨会话的检索
- **L3技能记忆**：从用户的任务中沉淀的可复用方法论，也就是这件事我之前是这么做的
- **L4血缘追溯**：每条记忆都记录来源，保证记忆的可解释性

同时，它支持插件化的记忆Provider，比如可以接入Honcho、Mem0等外部记忆系统。

#### 2.3.2 核心特点

- **插件化扩展**：支持接入外部的记忆Provider，灵活扩展记忆能力
- **技能记忆沉淀**：可以从用户的任务中自动沉淀可复用的技能，实现经验的复用
- **血缘追溯**：每条记忆都有来源记录，保证记忆的可解释、可审计
- **跨会话推理**：支持跨会话的辩证推理，更好的理解用户的长期需求

#### 2.3.3 记忆管理细节

**存储方式**

- 核心记忆存储在`MEMORY.md`文件中，人类可读
- 长时记忆使用SQLite+FTS5进行存储，支持全文检索
- 支持插件化的外部存储，比如接入Mem0、Honcho等外部记忆系统

**更新方式**

采用自动进化的更新机制：

- 每次交互完成后，自动更新用户画像，沉淀新的技能记忆
- 自动将新的交互生成摘要，更新到长时记忆的SQLite库中
- 外部的记忆Provider会自动同步更新，保证跨系统的记忆一致

**记忆的使用（加载）方式**

Hermes Agent的记忆加载采用分层加载+插件化检索的模式：

- 每次会话启动时，自动加载核心记忆、用户画像，注入到系统提示词中。中途不更新，主要目的是命中 KV Cache
- 处理用户查询时，自动从长时记忆、技能记忆中检索相关的信息
- 如果启用了外部记忆Provider，会自动调用Provider的检索接口，获取跨系统的记忆
- 将所有检索到的记忆按照相关性排序，精简后注入到LLM的prompt中

### 2.4 mem0

> **方案性质**：独立的通用记忆层，本身不是完整的Agent，可嵌入到任何LLM应用、Agent中，可与LangChain、LangGraph、各类Agent项目协同工作，快速优化其记忆能力。

mem0是目前最受欢迎的独立Agent Memory产品，定位为**自优化的LLM记忆层**，旨在为LLM应用提供可持续更新的个性化长期记忆，支持开源自托管与云服务两种部署模式，是当前落地最快的记忆方案之一。

#### 2.4.1 核心架构

mem0采用**向量+可选图的混合存储架构**，通过两阶段的记忆处理流程实现智能记忆管理：

**提取阶段**：当新的交互信息输入时，LLM自动从对话、文档中提取关键事实、用户偏好与上下文信息，而非直接存储原始文本。

**更新阶段**：系统自动进行冲突检测，识别重复或矛盾的记忆，通过LLM驱动的解析器完成记忆的新增、更新或删除，保证记忆库的一致性。

同时，mem0实现了**三级记忆分层**：

- **用户级记忆**：存储用户的长期偏好、个人信息、历史决策等个性化事实与通用事实
- **会话级记忆**：记录当前交互的临时上下文、短期对话信息
- **Agent级记忆**：跨用户共享的通用Agent知识，启用图存储后，会额外存储实体间的关联关系，用于复杂推理

#### 2.4.2 核心特点

- **低侵入性集成**：仅需几行代码即可为现有LLM应用添加记忆能力，兼容LangGraph等主流Agent框架
- **混合检索能力**：默认使用向量存储实现语义检索，可选启用图存储存储实体关系，支持混合查询
- **多模态记忆支持**：可从图像、文本等多模态输入中提取记忆信息
- **记忆质量管理**：内置去重、冲突检测、过期清理机制，保证记忆的有效性

#### 2.4.3 记忆管理细节

**存储方式**

- 默认模式下，将LLM提取后的关键事实生成嵌入向量，存储在向量数据库中，通过语义相似度实现召回
- 启用图存储后，会同步将抽取的实体作为节点、关系作为边存入图数据库，每个记忆条目附带时间戳、分类标签等元数据
- 支持多级隔离，不同用户、会话的记忆通过元数据进行逻辑隔离，避免数据混淆

**更新方式**

采用两阶段的自动更新机制：

- 新信息输入后，先通过LLM提取关键事实，同时识别是否存在重复或矛盾的已有记忆
- 对于重复记忆，自动合并去重；对于矛盾记忆，保留最新的有效事实，标记旧记忆为过期；对于全新记忆，直接新增入库
- 后台会定期清理低访问频率、过期的记忆，保证记忆库的精简与有效

**记忆的使用（加载）方式**

当处理用户查询时，mem0会自动完成记忆的检索与加载：

- **查询增强**：先对用户的输入进行语义解析，生成适配的检索查询，同时根据用户、会话的元数据进行过滤，保证只加载当前用户的有效记忆
- **混合检索**：默认先通过向量检索，召回语义最相关的记忆条目；如果启用了图存储，会以向量检索的结果为锚点，在图数据库中遍历关联的实体与关系，挖掘深层关联信息
- **上下文注入**：将检索到的记忆按照相关性排序，精简后注入到LLM的prompt中，同时自动控制注入的长度，避免超出上下文窗口限制
- **自动适配**：系统会自动判断当前查询需要的记忆类型，比如个性化偏好类查询会优先加载用户级记忆，临时交互类查询会优先加载会话级记忆

### 2.5 Graphiti

> **方案性质**：独立的时序记忆引擎，本身不是完整的Agent，可集成到任何Agent、LLM应用中，为其提供带历史追溯的时序记忆能力，可与其他项目协同工作。

Graphiti是Zep团队开源的**时序知识图谱记忆引擎**，核心定位是为AI Agent提供动态演化的结构化记忆，专门解决传统记忆方案无法追溯历史变更的痛点。

#### 2.5.1 核心架构

Graphiti的核心是**双时态知识图谱**，以事件（Episode）为核心驱动记忆的增量更新：

- **事件录入**：将所有输入信息（对话、文档、结构化JSON数据）统一封装为事件，作为图谱更新的基础单元。所有输入的对话、文档、结构化业务数据，作为原始事件完整留存
- **增量解析**：实时对新事件进行实体与关系抽取，同时识别信息中的时间属性，无需批量重算即可完成图谱更新
- **非覆盖式更新**：当新事实与旧事实冲突时，Graphiti不会直接删除旧数据，而是通过双时间戳标记旧关系的失效时间，保留完整的历史演化轨迹
- **混合索引检索**：整合语义搜索、关键词匹配与图遍历，实现高精度的混合检索

#### 2.5.2 核心特点

- **双时间轴能力**：每条关系边都包含`valid_at`/`invalid_at`（事实生效时间）与`created_at`/`expired_at`（数据录入时间），支持历史追溯查询
- **实时动态更新**：支持新数据的实时增量整合，图谱可随交互持续演化
- **低Token成本**：相比传统的LLM总结方案，大幅减少Token消耗，延迟更低
- **企业级可解释性**：记忆以结构化图谱呈现，关系可溯源，满足企业合规需求

#### 2.5.3 记忆管理细节

**存储方式**

全部存储在双时态知识图谱中：

- 实体作为图谱的节点，关系作为带属性的边
- 每条关系边都附带双时间戳：`valid_at`/`invalid_at`标记事实的生效时间段，`created_at`/`expired_at`标记数据的录入时间
- 同时为实体和事件构建语义索引，支持混合检索

**更新方式**

采用非覆盖的增量更新：

- 新事件输入后，实时抽取实体与关系，无需批量重算整个图谱即可完成增量更新
- 当新事实与旧事实冲突时，不会删除旧的关系数据，而是将旧关系的`invalid_at`标记为当前时间，同时新增新的关系边
- 这种更新方式保证了所有历史事实都可追溯，支持过去某个时间点，实体的状态是什么这类时序查询

**记忆的使用（加载）方式**

Graphiti针对时序记忆设计了专门的加载流程：

- **查询解析**：识别用户查询中的实体与时间属性，比如自动识别查询是否是历史追溯类的请求
- **混合检索**：先通过语义检索找到相关的事件与实体，然后在时序知识图谱中，根据查询的时间窗口，过滤出对应时间段有效的关系边，同时支持多跳的图遍历，挖掘关联信息
- **结构化上下文构建**：将检索到的实体、时序关系、事件信息整理为结构化的上下文，清晰呈现事实的时间线与演化过程，方便LLM进行时序推理
- **低开销注入**：由于记忆本身是结构化的，无需额外的LLM总结即可直接注入，大幅减少了Token消耗，保证加载的效率

### 2.6 memary

> **方案性质**：独立的类人记忆层，本身不是完整的Agent，可集成到任何Agent、LLM应用中，为其提供类人的记忆管理能力，可与其他项目协同工作。

memary是一个**模拟人类记忆机制的开源记忆层**，旨在让AI Agent能够像人类一样组织、检索和演化记忆，核心是将记忆的广度与深度进行分离管理。

#### 2.6.1 核心架构

memary借鉴认知科学的记忆模型，构建了**类人双层记忆体系**：

- **路由Agent**：作为系统的入口，负责接收用户查询，根据任务类型自动调度搜索、视觉等工具，实现多工具的协同处理
- **记忆流记忆（Memory Stream）**：记录所有接触过的实体及对应的时间戳，用于捕捉用户知识的广度，保留交互的完整时间线
- **实体知识存储记忆（Entity Knowledge Store）**：针对每个实体，记录其引用频率、最近访问时间等信息，用于刻画用户知识的深度，实现高频信息的优先召回
- **知识图谱记忆**：以图结构存储实体与关系，支持递归检索与多跳推理，实现复杂查询的信息定位

#### 2.6.2 核心特点

- **自动记忆构建**：无需人工干预，Agent在运行过程中自动捕获、整理和更新记忆
- **递归检索推理**：支持多跳的递归检索，能够从关联实体中挖掘深层信息，应对复杂任务
- **动态记忆演化**：记忆会随交互持续更新，根据访问频率调整信息的优先级，模拟人类的记忆遗忘与强化
- **可视化监控**：提供性能仪表盘，支持开发者查看记忆的演化过程，方便调试与优化

#### 2.6.3 记忆管理细节

**存储方式**

全部基于知识图谱进行存储：

- 知识图谱中存储实体节点与关系边，同时为每个节点附加访问频率、最近访问时间等元数据
- 记忆流以实体的时间戳序列的形式，记录在图谱的元数据中
- 支持多图谱隔离，不同用户的记忆存储在独立的图谱中

**更新方式**

动态的自适应更新：

- 每次交互完成后，自动将新的实体加入记忆流，更新对应实体的访问频率与最近访问时间
- 知识图谱自动新增抽取到的实体与关系，对于已有的实体，更新其关联信息
- 记忆会根据访问频率自动调整优先级，高频访问的信息会被优先保留，低频信息会逐步降低权重，模拟人类的记忆遗忘机制

**记忆的使用（加载）方式**

memary模拟人类的记忆召回机制，实现自适应的记忆加载：

- **实体识别**：路由Agent先识别用户查询中的目标实体，定位需要召回的记忆范围
- **优先级召回**：首先根据实体的访问频率、最近访问时间，优先召回高频、近期的实体记忆，保证核心信息优先加载
- **递归检索**：针对目标实体，在知识图谱中进行多跳的递归检索，挖掘深层的关联信息，应对复杂的推理查询
- **动态上下文压缩**：将召回的记忆按照相关性排序，自动过滤低相关的信息，保证注入到上下文的内容都是最相关的，避免上下文溢出
- **时间线辅助**：同步加载记忆流的时间线信息，帮助LLM理解交互的先后顺序，更好地还原历史交互的上下文

### 2.7 Letta

> **方案性质**：完整的自主Agent框架，是独立的Agent产品，分层记忆系统是其核心能力，记忆模块也可独立对接其他项目，为其提供自主的内存管理能力。

Letta是MemGPT团队推出的完整Agent框架，其核心记忆架构源自MemGPT的开创性研究，通过上下文工程突破LLM的上下文窗口限制，将LLM改造为具备自主内存管理的**操作系统**。

#### 2.7.1 核心架构

Letta采用**分层记忆架构**，将记忆划分为上下文内与上下文外两个层级：

- **核心（In-Context）内存记忆**：常驻在LLM的上下文窗口中，包含系统指令、可读写的内存块、近期的对话历史，Agent可通过工具直接编辑这部分内容
- **归档（Out-of-Context）内存记忆**：存储被挤出上下文的旧数据，包括历史对话摘要、外部知识存储，通过向量检索实现召回

同时，Agent内置了内存管理工具，支持：

- 核心内存的追加与替换编辑
- 历史对话的搜索检索
- 归档内存的插入与查询

当上下文窗口满时，系统会自动将旧的对话历史递归压缩为摘要，存入归档内存，保证长期会话的连续性。

#### 2.7.2 核心特点

- **自主内存管理**：Agent可自主编辑自己的内存，实现类似计算机的内存分页机制
- **睡眠态后台处理**：v2版本新增睡眠Agent能力，可在空闲时自动进行记忆的整理与优化
- **多工具协同**：将记忆操作封装为工具，支持Agent自主调用，实现真正的自主代理
- **长会话支持**：通过递归摘要与归档存储，支持无限长度的长期交互

#### 2.7.3 记忆管理细节

**存储方式**

- 核心内存以可读写的文本块形式存在，常驻在LLM的上下文窗口中，Agent可直接读取和编辑
- 归档内存的内容会生成嵌入向量，存储在向量数据库中，同时保留原始的摘要文本
- 所有的记忆操作都被封装为工具，Agent可自主调用完成记忆的读写

**更新方式**

自主的分层更新机制：

- 当上下文窗口即将满时，系统自动将旧的对话历史递归压缩为摘要，从核心内存移动到归档内存
- Agent可自主调用内存编辑工具，对核心内存进行追加、替换操作，自主更新自己的记忆
- 在v2版本中，新增了睡眠态后台更新，Agent在空闲时会自动整理归档内存，合并重复信息、清理过期内容，优化记忆的存储结构

**记忆的使用（加载）方式**

Letta的记忆加载由Agent自主控制，实现类操作系统的内存分页：

- **核心内存常驻**：核心内存的内容会一直常驻在LLM的上下文窗口中，作为当前交互的基础上下文，Agent可直接读取
- **自主检索触发**：当Agent判断当前查询需要历史记忆时，会自主调用归档记忆的搜索工具，输入检索query，从向量库中召回相关的历史摘要
- **内存换入换出**：Agent可自主决定将检索到的归档记忆加载到核心内存中，同时将核心内存中不再需要的旧内容压缩后移到归档内存，实现内存的动态分页
- **上下文整理**：加载完成后，Agent会自动整理上下文的结构，保证核心信息在前，检索到的记忆在后，方便LLM进行推理

### 2.8 Cognee

> **方案性质**：独立的知识记忆引擎，本身不是完整的Agent，可集成到任何Agent、LLM应用中，为其提供多源数据融合的知识记忆能力，可与其他项目协同工作。

Cognee是一个端到端的记忆引擎，定位为**AI的知识引擎**，通过整合关系、向量、图三种存储，实现从原始数据到可推理记忆的全流程处理，支持复杂的企业级知识融合场景。

#### 2.8.1 核心架构

Cognee采用**三存储混合架构**，统一了三种存储能力：

- **关系存储**：默认使用SQLite，用于跟踪文档、数据块的来源与元数据，实现数据的溯源与权限管理
    - 原始数据记忆 - 导入的原始文档、对话、多模态数据的归一化版本，用于溯源
    - 元数据记忆 - 数据的来源、权限、访问记录等元信息
- **向量存储**：默认使用LanceDB，用于存储嵌入向量，支持语义相似度检索
- **图存储**：默认使用Kuzu，用于存储实体与关系，支持结构化的图推理
    - 结构化知识记忆 - 抽取的实体、关系，以及基于本体定义的领域知识
    - 衍生事实记忆 - 通过Memify阶段生成的衍生事实、强化后的关联关系

同时，Cognee提供了标准化的处理Pipeline：

- **Add**：支持38+格式的多源数据导入，自动完成数据归一化与去重
- **Cognify**：核心的知识构建阶段，通过六阶段流程，从数据中提取实体、关系，构建知识图谱并生成嵌入
- **Memify**：记忆优化阶段，自动修剪过期节点、强化高频连接、生成衍生事实，实现记忆的自优化
- **Search**：提供14种检索模式，支持从纯向量检索到多跳图推理的全场景查询

#### 2.8.2 核心特点

- **多源数据融合**：支持对话、文档、图片、音频等30+种数据类型的统一记忆构建
- **本体驱动建模**：支持RDF/OWL本体定义，可针对特定行业定制知识结构
- **丰富的检索模式**：提供14种查询模式，可适配从简单问答到复杂多跳推理的不同需求
- **零配置启动**：默认的嵌入式存储无需额外部署基础设施，pip安装即可使用

#### 2.8.3 记忆管理细节

**存储方式**

采用三存储协同的方式：

- **关系存储**：存储原始数据块、元数据，用于数据溯源与权限管理
- **向量存储**：存储所有数据点的嵌入向量，用于语义相似度检索
- **图存储**：存储实体节点与关系边，用于结构化的图推理
- 所有的记忆单元（DataPoint）都会同时在三个存储中建立关联，保证检索时可以跨层融合

**更新方式**

Pipeline驱动的增量更新：

- 新数据导入时，系统自动完成去重，仅处理新增或变更的数据，无需重算已有数据
- Cognify阶段，针对新数据抽取实体与关系，增量更新知识图谱与向量存储
- Memify阶段，自动对整个记忆库进行优化：修剪过期的节点、强化高频访问的连接、生成新的衍生事实，实现记忆的自优化
- 支持动态的本体更新，可随时调整领域知识的定义，适配不同的业务场景

**记忆的使用（加载）方式**

Cognee针对不同的场景，提供了自适应的记忆加载方式：

- **检索模式自动选择**：系统会根据用户的查询，自动选择最合适的14种检索模式中的一种，比如简单问答用向量检索，多跳推理用Graph COT检索
- **跨层融合检索**：先通过向量检索找到相关的DataPoint，然后在图存储中遍历关联的实体与关系，同时从关系存储中拉取对应的元数据与原始数据，实现跨存储的信息融合
- **本体驱动过滤**：根据本体的定义，过滤掉不符合领域规则的信息，保证加载的记忆符合业务的约束
- **结构化知识注入**：将检索到的实体、关系、衍生事实整理为结构化的知识，同时附带溯源信息，注入到LLM的prompt中，支持复杂的逻辑推理
- **动态上下文适配**：自动调整加载的记忆数量，保证注入的内容不会超出上下文窗口的限制

## 3 面试高频考点

### 3.1 算法岗通用题

#### Q1：**记忆框架的核心设计维度有哪些？**

记忆框架的核心设计维度包括四个方面：

- **记忆内容**：涵盖用户层记忆（个性化偏好、决策风格、行为轨迹）、任务/技能层记忆（执行经验、任务摘要、工具调用记录）、知识/关系层记忆（实体关系、演化型事实）、系统/Agent层记忆（会话日志、身份定义、模型内部记忆）
- **记忆更新**：包括写入策略（直接追加、LLM抽取、自主编辑）、异步/后台处理（语义富集、Dreaming沉淀、技能迭代）、冲突与演化管理（时序式更新、冲突检测合并）、生命周期管理（遗忘曲线、Checkpoint、读取刷新）
- **记忆存储**：分为单层存储方案（文件系统、SQLite、向量数据库、图数据库、键值存储、模型内部存储）和混合存储方案（多存储架构、Agent文件系统）
- **记忆使用**：包括检索策略（混合检索、主动决策、知识图谱推理、时序推理）、上下文构建（Token预算感知、渐进式披露、Prefix Cache）、协作与进化（跨Agent共享、自动Prompt优化、状态感知推理）

#### Q2：**向量数据库与图数据库在记忆存储中各有什么优劣？**

向量数据库与图数据库在记忆存储中各有优势：

**向量数据库优势**：
- 支持语义相似度检索，适配非结构化记忆的召回
- 对于自然语言查询的理解能力更强
- 实现相对简单，生态成熟，集成成本低

**向量数据库劣势**：
- 难以处理结构化关系推理
- 不擅长多跳的复杂逻辑查询
- 对于实体间关系的存储和检索能力有限

**图数据库优势**：
- 存储实体与关系，支持结构化关系查询
- 支持多跳的复杂知识推理
- 能够很好地刻画实体间的关联关系
- 支持时序知识图谱，可以追溯事实演化

**图数据库劣势**：
- 实现复杂度高，需要设计合理的Schema
- 纯图结构对于模糊语义查询的支持有限
- 集成和运维成本较高

**最佳实践**：通常采用混合架构，向量数据库用于语义检索，图数据库用于关系推理，两者协同工作效果最佳。

#### Q3：**如何根据业务场景选型合适的记忆框架？**

记忆框架选型需要根据业务需求综合考虑：

- **快速落地、通用对话场景**：优先选择mem0，易用性最高，集成成本最低
- **需要追溯历史变更、企业时序数据**：优先选择Graphiti，双时态能力可完美支撑历史追溯
- **个性化助手、多工具复杂任务**：优先选择memary，类人记忆模型更适配个性化交互
- **自主Agent、超长会话任务**：优先选择Letta，分层内存管理可支撑自主代理的长期运行
- **复杂企业知识融合、深度推理**：优先选择Cognee，三存储架构可处理多源异构的复杂数据
- **人机协作、透明可控的记忆**：优先选择OpenClaw，文件化的记忆完全透明，支持手动编辑
- **编程开发、项目级协作**：优先选择Claude Code，专门针对编程场景优化，项目级记忆隔离
- **插件化扩展、技能沉淀**：优先选择Hermes Agent，插件化架构支持灵活扩展，自动沉淀技能

选型时还需要考虑数据规模、标注成本、团队技术栈等因素。

#### Q4：**记忆更新中"时序式更新"的设计思路是什么？**

时序式更新的核心设计思路是**保留完整历史而非覆盖**：

- 当新事实与旧事实冲突时，不直接删除旧数据，而是通过时间戳标记旧关系的失效时间
- 新增新的事实记录，同时保留旧记录作为历史版本
- 这种设计可以支持追溯过去某个时间点的实体状态
- 典型实现是双时间戳：`valid_at`/`invalid_at`标记事实的生效时间段，`created_at`/`expired_at`标记数据的录入时间
- 适用场景：企业合规审计、历史状态回溯、时序推理等需要保留完整变更记录的场景

#### Q5：**如何解决记忆框架中的"灾难性遗忘"问题？**

灾难性遗忘是指模型在学习新知识时忘记旧知识的问题，记忆框架中可以通过以下方式解决：

- **参考模型约束**：以SFT模型作为参考点，损失函数约束当前模型不要偏离参考模型太远
- **分层记忆架构**：区分核心记忆和归档记忆，核心记忆常驻不被遗忘，归档记忆用于长期存储
- **混合训练**：在记忆更新时混合通用能力数据和新任务数据，避免对齐冲刷通用能力
- **记忆巩固**：类似人类的睡眠机制，在后台定期整理记忆，合并重复、强化重要记忆
- **知识图谱固化**：将关键知识固化在图结构中，不易被遗忘
- **技能独立存储**：将沉淀的技能独立存储，定期复用和强化

#### Q6：**记忆检索中的"渐进式披露"设计有什么优势？**

渐进式披露的设计优势：

- **节省Token消耗**：平时仅加载记忆的名称、描述等轻量索引，避免占用过多Token
- **按需加载完整内容**：当判断当前任务与该记忆相关时，再加载完整的记忆内容
- **降低推理延迟**：减少上下文长度，提升推理速度
- **提高检索精准度**：先通过轻量索引筛选相关记忆，再加载完整内容，避免无关记忆干扰

典型实现：Hermes Agent的Skills系统，Claude Code的记忆索引等。

#### Q7：**跨Agent共享记忆需要解决哪些核心问题？**

跨Agent共享记忆需要解决的核心问题：

- **记忆隔离**：需要支持按用户、会话、项目等维度进行逻辑隔离，避免数据混淆
- **权限管理**：不同Agent可能有不同的访问权限，需要细粒度的权限控制
- **一致性保证**：多个Agent同时更新记忆时，需要保证数据一致性
- **冲突解决**：不同Agent可能生成冲突的记忆，需要有冲突检测和解决机制
- **同步延迟**：需要平衡实时性和性能，选择合适的同步策略

典型实现：Zep的Group Graphs，Engram的跨namespace搜索等。

#### Q8：**"记忆税"是什么意思？如何降低记忆税？**

记忆税是指**为了实现记忆功能而付出的代价**，包括：

- **计算资源消耗**：记忆的存储、检索、更新都需要额外的计算资源
- **Token消耗**：将记忆注入到上下文需要消耗额外的Token
- **推理延迟**：记忆检索和上下文构建会增加推理延迟
- **对齐税**：为了记忆功能可能牺牲部分模型能力

降低记忆税的方法：

- **渐进式披露**：先加载轻量索引，按需加载完整内容
- **Token预算感知**：根据当前预算动态调整加载的记忆数量
- **Prefix Cache**：将固定记忆层缓存复用
- **混合检索**：组合多种检索策略，提高召回准确率，减少无效记忆
- **轻量化存储**：选择合适的存储方案，降低存储和检索成本

#### Q9：**Dreaming后台沉淀机制的工作原理是什么？**

Dreaming后台沉淀机制的工作原理：

- 定期运行后台任务，扫描历史日志
- 自动将高价值内容从短期记忆升级沉淀为长期记忆
- 类似人类的睡眠记忆整理过程
- 可以在后台完成记忆合并、过期清理、关系强化等工作
- 不阻塞主交互流程，用户体验更好

典型实现：OpenClaw的Dreaming功能，Claude Code的autoDream进程，Letta v2的睡眠态处理等。

#### Q10：**记忆框架中的"技能沉淀"是如何实现的？**

记忆框架中的技能沉淀通常通过以下方式实现：

- **自动提取**：Agent在执行任务过程中，自动将成功的工作流提取为可复用的技能
- **结构化存储**：将技能以结构化的方式存储，包含步骤、参数、注意事项等
- **版本管理**：支持技能的版本迭代，不断优化改进
- **检索复用**：在新任务中，检索相似的历史技能，指导当前任务执行
- **自我迭代**：Agent在使用已有技能的过程中，自动优化、修改技能文档

典型实现：Hermes Agent的L4技能记忆，LangMem的自动Prompt优化等。
