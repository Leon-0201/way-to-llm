# 2. 推理性能指标
> 核心考点：各指标定义、计算方式、行业基准值、优化方向，工程岗高频考点，是性能优化的基础
---
## 一、核心性能指标定义与计算
### 🔹 延迟类指标（用户感知最明显）
| 指标 | 全称 | 定义 | 计算方式 |
|------|------|------|----------|
| TTFT | Time To First Token | 从用户发请求到收到第一个Token的时间 | 第一个Token返回时间 - 请求发起时间 |
| TPOT | Time Per Output Token | 平均每个Token生成的时间 | 总生成时间 / 总生成Token数 |
| 端到端延迟 |  | 从用户发请求到收到完整回答的时间 | 最后一个Token返回时间 - 请求发起时间 |
| P99延迟 |  | 99%的请求可以在该时间内完成 | 按请求延迟排序取99百分位值 |
### 🔹 吞吐类指标（成本相关核心指标）
| 指标 | 定义 | 计算方式 |
|------|------|----------|
| Tokens/s | 每秒生成Token总数 | 总生成Token数 / 总时间 |
| RPS | Requests Per Second，每秒处理请求数 | 总请求数 / 总时间 | 
| 并发数 | 同时支持的最大请求数 | 按延迟阈值倒推 |
### 🔹 资源效率指标（成本优化核心）
| 指标 | 定义 | 计算方式 |
|------|------|----------|
| 显存利用率 | 显存实际使用占总显存比例 | 已用显存 / 总显存 |
| GPU利用率 | GPU计算单元使用率 | 平均SM使用率 |
| Token成本 | 每生成1000Token的成本 | 总GPU成本 / 总生成Token数 |
---
## 二、性能瓶颈定位方法论
1. **延迟高但吞吐低**：调度问题，比如batch太小、GPU利用率低，优先优化批处理策略
2. **延迟高且吞吐高**：模型本身计算量大，优先优化算子、量化、升级硬件
3. **TTFT高但TPOT正常**：预填充阶段瓶颈，优先优化输入处理、KV Cache初始化
4. **TPOT高但TTFT正常**：生成阶段瓶颈，优先优化解码算子、KV Cache读写

## 三、面试高频考点（分岗位）
### ✅ 开发岗通用题
1. **Q：大模型推理的核心性能指标有哪些？对话场景最关注什么指标？**
   答：核心指标分为三类：① 延迟类：TTFT首Token延迟、TPOT每Token生成时间、端到端延迟；② 吞吐类：Tokens/s每秒生成Token数、RPS每秒请求数；③ 资源类：显存利用率、GPU利用率。对话场景最关注TTFT和TPOT，直接影响用户体验。
2. **Q：TTFT高的常见原因有哪些？怎么优化？**
   答：TTFT高的核心是预填充阶段瓶颈，常见原因：① 输入太长，预填充计算量大；② KV Cache初始化慢；③ batch太大，预填充时间长；④ 调度策略不合理，长请求卡住短请求。优化手段：① Chunked Prefill分块预填充，避免长请求阻塞；② 输入长度限制，超长输入截断；③ 优化预填充算子，用Flash Attention加速；④ 按输入长度分类调度，短请求优先级高。
3. **Q：Tokens/s和RPS的关系是什么？为什么有时候Tokens/s高但RPS低？**
   答：Tokens/s是每秒生成的总Token数，RPS是每秒处理的请求数，两者关系：RPS = Tokens/s / 平均每请求生成Token数。Tokens/s高但RPS低的原因是平均每请求生成的Token太长，比如长文本生成场景，每个请求生成几千Token，虽然总Tokens/s高，但处理的请求数少。优化手段可以用动态batch，混合长短请求，提升GPU利用率同时兼顾RPS。

### 🔬 算法岗专项题
1. **Q：怎么估算大模型推理的FLOPs？7B模型生成一个Token需要多少FLOPs？**
   答：大模型推理的FLOPs估算公式：`推理1个Token的FLOPs = 2 * 参数量`（因为每个Transformer层都要做一次前向传播，包括注意力和FFN，总计算量约为2倍参数量）。7B模型生成1个Token的FLOPs约为2*7B=14GFLOPs。如果是prefill阶段（处理输入），输入长度为L的话，FLOPs约为`2*参数量*L`。
2. **Q：7B模型在A10G上理论最高Tokens/s是多少？怎么估算？**
   答：A10G的FP16算力是31TFLOPS，7B模型生成1个Token需要14GFLOPs，理论最高Tokens/s = 31TFLOPS / 14GFLOPs ≈ 2200tokens/s。考虑到实际有调度开销、显存带宽瓶颈、算子利用率等因素，实际能做到1500tokens/s就是优秀水平。
3. **Q：为什么大模型推理是内存带宽 bound，而不是计算 bound？**
   答：因为大模型推理时，每个Token生成都需要加载整个模型的参数，计算量是2倍参数量，访存量是参数量*2字节（FP16），计算访存比约为1（每访存1字节做1次计算）。而GPU的计算能力提升比显存带宽快很多，所以大部分场景下显存带宽是瓶颈，而不是计算能力。这也是为什么量化可以大幅提升性能的原因，量化可以直接降低访存量。
