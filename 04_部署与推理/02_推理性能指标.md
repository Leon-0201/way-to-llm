# 推理性能指标

本章系统梳理大模型推理性能核心知识，内容组织分为5大模块：推理两阶段原理、核心性能指标、性能估算方法、优化方向、面试高频考点。
旨在解决以下核心问题：大模型推理的性能指标有哪些？如何估算推理性能？如何定位和优化性能瓶颈？
学习完本章，你将：

- 能够快速估算不同场景下的推理性能
- 熟练掌握推理性能优化和瓶颈定位方法论
- 熟悉推理性能相关的面试考点

## 1 大模型推理两阶段核心原理
大模型推理采用KV Cache（以空间换时间）技术，分为两个计算特征完全不同的阶段，所有性能指标都基于这两个阶段的特性设计：
### 1.1 Prefill（预填充阶段）
并行处理用户输入的所有Prompt Token，一次性计算并生成初始KV Cache，完成一次完整的前向传播后输出第一个Token。这个阶段的耗时主要由输入Token长度决定，是首Token延迟（TTFT）的核心来源。
### 1.2 Decoding（解码阶段）
逐个生成后续的Output Token，每一步只处理当前最新生成的1个Token，复用之前已计算好的KV Cache完成前向传播。这个阶段需要执行N次前向传播（N为输出Token数量），耗时主要由输出Token长度决定。

## 2 核心性能指标定义与计算
### 2.1 延迟类指标（用户感知最明显）
| 指标 | 全称 | 定义 | 计算方式 |
|------|------|------|----------|
| TTFT | Time To First Token | 从用户发请求到收到第一个Token的时间 | 第一个Token返回时间 - 请求发起时间，等于Prefill阶段耗时 |
| TPOT | Time Per Output Token | 解码阶段平均每个输出Token的时间（不含首Token） | (端到端延迟 - TTFT) / 输出Token数量 |
| 端到端延迟 |  | 从用户发请求到收到完整回答的时间 | 标准公式：`端到端延迟 = TTFT + TPOT * 输出Token数量` |
| P99延迟 |  | 99%的请求可以在该时间内完成 | 按请求延迟排序取99百分位值 |
### 2.2 吞吐类指标（成本相关核心指标）
| 指标 | 定义 | 计算方式 |
|------|------|----------|
| TPS | Tokens Per Second | 单个请求每秒生成的Token数 | 输出Token数量 / 端到端延迟 |
| Throughput（吞吐量） | 并发场景下GPU每秒处理的总Token数 | 所有请求总生成Token数 / 总时间 |
| RPS | Requests Per Second，每秒处理请求数 | 总请求数 / 总时间 | 
| 并发数 | 同时支持的最大请求数 | 按延迟阈值倒推 |
### 2.3 资源效率指标（成本优化核心）
| 指标 | 定义 | 计算方式 |
|------|------|----------|
| 显存利用率 | 显存实际使用占总显存比例 | 已用显存 / 总显存 |
| GPU利用率 | GPU计算单元使用率 | 平均SM使用率 |
| Token成本 | 每生成1000Token的成本 | 总GPU成本 / 总生成Token数 |

## 3 性能估算方法论
### 3.1 两阶段计算量快速估算公式
大模型推理的计算量可以通过以下简化公式快速估算（FLOPs，浮点运算次数）：
- **Prefill阶段总计算量**：`Prefill_FLOPs = 2 * BatchSize * 输入Token长度 * 模型参数量`
- **Decoding阶段单Token计算量**：`Decoding_FLOPs_PerToken = 2 * BatchSize * 模型参数量`
> 注：对于MoE架构模型，公式中的参数量取激活参数量，而非总参数量。
### 3.2 性能指标估算方法
基于GPU算力和利用率可以快速估算理论性能极限：
- **TTFT估算**：`TTFT ≈ Prefill_FLOPs / (GPU标称算力 * 实际GPU利用率)`
- **解码吞吐量估算**：`Throughput ≈ (GPU标称算力 * 实际GPU利用率) / Decoding_FLOPs_PerToken`
> 典型GPU利用率参考：工业界优化较好的推理框架一般能达到30%-60%的GPU算力利用率。
### 3.3 Output Token比Input Token成本更高的核心原因
1. **计算次数差异**：Input Token只需要一次Prefill前向传播即可完成处理，而N个Output Token需要N次Decoding前向传播
2. **通信瓶颈影响**：虽然单步Decoding计算量小于Prefill，但Decoding阶段需要频繁读写KV Cache，受限于GPU显存I/O带宽瓶颈，单位计算的实际开销更高
3. **资源占用差异**：Output Token生成过程中KV Cache持续增长，显存占用随生成长度线性增加，进一步拉高资源成本

## 4 性能优化核心方向
### 4.1 批处理优化：Naive Batching vs Continuous Batching
批处理是提升GPU利用率的核心手段，主流实现有两种：
1. **Naive Batching（静态/同步批处理）**
   - 原理：将多个请求合并成一个固定Batch，一次性输入模型处理，全部完成后统一返回
   - 缺点：存在短板效应，Batch中单个长请求会拖慢整个Batch的处理速度，GPU资源浪费严重
2. **Continuous Batching（动态/连续批处理）**
   - 原理：迭代级调度，每步前向传播后动态移除已完成的请求、加入新请求，无需等待整个Batch处理完成
   - 优势：大幅提升GPU利用率（可提升2-3倍），降低平均延迟，是当前主流推理框架（vLLM、TensorRT-LLM等）的标准配置
---
## 5 性能瓶颈定位方法论
1. **延迟高但吞吐低**：调度问题，比如batch太小、GPU利用率低，优先优化批处理策略，采用Continuous Batching
2. **延迟高且吞吐高**：模型本身计算量大，优先优化算子、量化、升级硬件
3. **TTFT高但TPOT正常**：预填充阶段瓶颈，优先优化输入处理、KV Cache初始化、采用Chunked Prefill分块预填充
4. **TPOT高但TTFT正常**：生成阶段瓶颈，优先优化解码算子、KV Cache读写、降低KV Cache显存开销

## 6 面试高频考点
### 开发岗通用题
#### Q1：**大模型推理的核心性能指标有哪些？对话场景最关注什么指标？**
核心指标分为三类：
- 延迟类：TTFT首Token延迟、TPOT每Token生成时间、端到端延迟
- 吞吐类：TPS单请求每秒Token数、Throughput总吞吐量、RPS每秒请求数
- 资源类：显存利用率、GPU利用率

对话场景最关注TTFT和TPOT，直接影响用户体验。

#### Q2：**TTFT高的常见原因有哪些？怎么优化？**
TTFT高的核心是预填充阶段瓶颈，常见原因：
- 输入太长，预填充计算量大
- KV Cache初始化慢
- batch太大，预填充时间长
- 调度策略不合理，长请求卡住短请求

优化手段：
- Chunked Prefill分块预填充，避免长请求阻塞
- 输入长度限制，超长输入截断
- 优化预填充算子，用Flash Attention加速
- 按输入长度分类调度，短请求优先级高

#### Q3：**TPS和Throughput的区别是什么？为什么有时候总Throughput高但RPS低？**
TPS是单个请求每秒生成的Token数，衡量单请求生成速度；Throughput是并发场景下GPU每秒处理的总Token数，衡量集群总处理能力。两者关系：`RPS = Throughput / 平均每请求生成Token数`。

Throughput高但RPS低的原因是平均每请求生成的Token太长，比如长文本生成场景，每个请求生成几千Token，虽然总Throughput高，但处理的请求数少。优化手段可以用动态batch，混合长短请求，提升GPU利用率同时兼顾RPS。

#### Q4：**为什么商业大模型的Output Token定价普遍比Input Token高？**
核心原因是Output Token的计算和资源开销更高：
- 计算次数差异：N个Input Token只需要1次Prefill前向传播，而N个Output Token需要N次Decoding前向传播
- 通信瓶颈：Decoding阶段需要频繁读写KV Cache，受限于GPU显存I/O带宽瓶颈，单位开销更高
- 显存占用：Output Token生成过程中KV Cache持续增长，显存占用随长度线性增加，进一步拉高成本

#### Q5：**Naive Batching和Continuous Batching的核心区别是什么？后者有什么优势？**
核心区别是调度粒度不同：
- Naive Batching是Batch级调度，必须等整个Batch所有请求处理完成才能替换新请求，存在短板效应，GPU利用率低
- Continuous Batching是迭代级调度，每步前向传播后即可动态移除完成的请求、加入新请求，无需等待整个Batch

优势是GPU利用率可提升2-3倍，平均延迟更低，是当前主流推理框架的标准配置。

### 算法岗专项题
#### Q1：**怎么估算大模型推理的FLOPs？7B模型生成一个Token需要多少FLOPs？**
大模型推理的FLOPs估算分阶段：
- Decoding阶段：`推理1个Token的FLOPs = 2 * 参数量`（每个Transformer层做一次前向传播，总计算量约为2倍参数量），7B模型生成1个Token的FLOPs约为2*7B=14GFLOPs
- Prefill阶段：输入长度为L时，FLOPs约为`2*参数量*L`

#### Q2：**7B模型在A10G上理论最高Tokens/s是多少？怎么估算？**
A10G的FP16算力是31TFLOPS，7B模型生成1个Token需要14GFLOPs，理论最高Throughput = 31TFLOPS / 14GFLOPs ≈ 2200tokens/s。考虑到实际有调度开销、显存带宽瓶颈、算子利用率等因素，实际能做到1500tokens/s就是优秀水平。

#### Q3：**为什么大模型推理是内存带宽 bound，而不是计算 bound？**
因为大模型推理时，每个Token生成都需要加载整个模型的参数，计算量是2倍参数量，访存量是参数量*2字节（FP16），计算访存比约为1（每访存1字节做1次计算）。而GPU的计算能力提升比显存带宽快很多，所以大部分场景下显存带宽是瓶颈，而不是计算能力。这也是为什么量化可以大幅提升性能的原因，量化可以直接降低访存量。

#### Q4：**Prefill和Decoding阶段的计算量有什么差异？分别对应什么性能指标？**
计算量差异：
- Prefill阶段总计算量与输入长度成正比：`Prefill_FLOPs = 2 * BatchSize * 输入长度 * 参数量`，对应TTFT指标
- Decoding阶段单Token计算量固定，与输入长度无关：`Decoding_FLOPs_PerToken = 2 * BatchSize * 参数量`，总解码计算量与输出长度成正比，对应TPOT指标

#### Q5：**KV Cache为什么只缓存K和V，不缓存Q？**
因为注意力计算中，Query是当前最新生成的单个Token的特征，每次都需要重新计算；而Key和Value是历史所有生成Token的特征，可以复用，不需要重复计算。因此只需要缓存K和V即可大幅降低计算量，缓存Q没有意义。
