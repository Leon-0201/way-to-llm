# 显存与并行

本章系统梳理大模型显存管理与并行部署核心知识，内容组织分为4大模块：显存组成与计算、多卡并行策略选型、通信硬件对比、面试高频考点。
旨在解决以下核心问题：大模型推理显存占用如何计算？不同并行策略的技术差异和适用场景是什么？生产级并行部署如何选型？
学习完本章，你将：
- 掌握大模型推理显存的组成和计算方法
- 熟悉5种主流并行策略的原理和适用场景
- 能够根据业务需求选择合适的并行部署方案

## 1 显存组成与计算
大模型推理显存占用分为固定占用和动态占用两部分，总显存 = 模型权重 + KV Cache + 激活值 + 运行时开销（约10%冗余）
### 1.1 模型权重（固定占用，占比最大）
- 计算方式：`显存 = 参数量 * 量化位宽 / 8`
- 例子：
  - 7B模型FP16：7B * 2字节 = 14GB
  - 7B模型INT4量化：7B * 0.5字节 = 3.5GB
- 优化方向：量化、模型压缩、稀疏化
### 1.2 KV Cache（动态占用，和并发数正相关）
KV Cache是注意力层的Key和Value缓存，用于加速多轮对话和生成，不需要每次都重新计算
- 计算方式：`单请求显存 = 2 * 层数 * 头维度 * 头数 * 上下文长度 * 量化位宽 / 8`
  其中2表示Key和Value两个张量
- 简化公式：7B模型FP16下，每并发每1024Token上下文约占16MB显存
- 例子：100并发，2048上下文的7B FP16模型，KV Cache占比 = 100 * 2 * 16MB = 3.2GB
- 优化方向：KV Cache量化、动态释放、分页管理（PagedAttention）、上下文窗口限制
### 1.3 激活值（动态占用，和batch size正相关）
- 计算方式：`显存 ≈ batch size * 上下文长度 * 隐藏层维度 * 2字节 * 系数`
  系数一般2-4，取决于模型结构
- 例子：batch size=32，上下文长度2048，7B模型（隐藏层维度4096），激活值约占`32*2048*4096*2*2 = 1GB`
- 优化方向：算子优化、激活值复用、Offloading到CPU
### 1.4 显存占比参考（7B FP16模型，2048上下文，100并发）
| 组成 | 显存 | 占比 |
|------|------|------|
| 模型权重 | 14GB | 77% |
| KV Cache | 3.2GB | 17% |
| 激活值 | 0.8GB | 4% |
| 总占用 | 18GB | 100% |

## 2 多卡并行策略选型
当单卡放不下模型或者需要更高吞吐时，用多卡并行，五种主流策略对比：
### 2.1 数据并行（Data Parallel，DP/FSDP）
- 原理：每个卡放完整模型，不同卡处理不同batch的数据，梯度同步
- 显存降低：无，每个卡都要放完整模型
- 通信开销：梯度同步，全量通信，开销和参数量成正比
- 适用场景：模型能放单卡，需要提升吞吐的场景
- 优势：实现简单，扩展性好，适合多机多卡横向扩容
### 2.2 张量并行（Tensor Parallel，TP）
- 原理：把模型的每层参数按张量维度拆分到不同卡，每层计算都需要卡之间通信
- 显存降低：和卡数成正比，N卡并行显存降低N倍
- 通信开销：极高，每层计算都要通信，要求卡之间高带宽低延迟
- 适用场景：模型太大放不下单卡，且卡之间带宽高（同机NVLink互联）
- 限制：最多并行到模型的隐藏层维度数，一般最大8卡
### 2.3 流水线并行（Pipeline Parallel，PP）
- 原理：把模型按层拆分到不同卡，每个卡负责几层的计算，流水线接力处理
- 显存降低：和卡数成正比，N卡并行显存降低N倍
- 通信开销：中等，只有相邻层之间需要通信，跨卡通信量小
- 适用场景：模型层数多，适合大模型（比如70B以上）多机并行
- 缺陷：有气泡开销，batch越大气泡占比越低
### 2.4 序列并行（Sequence Parallel，SP）
- 原理：按输入序列长度拆分，每个卡处理部分序列，适合超长上下文场景
- 显存降低：和卡数成正比，上下文部分显存降低N倍
- 通信开销：中等，注意力层需要全局通信
- 适用场景：超长上下文场景（100K以上）
### 2.5 专家并行（Expert Parallel，EP，MoE模型特有）
- 原理：MoE（混合专家）架构专属并行策略，把不同的完整FFN专家分配到不同GPU上，仅路由到对应专家时才需要跨卡通信
- 显存降低：和卡数/专家数成正比，N卡并行显存降低N倍
- 通信开销：极低，仅激活专家时通信，对带宽要求远低于张量并行
- 适用场景：所有MoE架构模型（Mixtral 8x7B、Qwen MoE、GPT-4 MoE等）部署
- 优势：效率极高，无论PCIe/NVLink/跨机InfiniBand部署效率都>85%，是MoE模型生产部署的标配
- 限制：仅支持MoE架构模型，对普通稠密模型无效

### 2.6 并行策略选型表
| 场景 | 首选策略 | 次选策略 |
|------|----------|----------|
| 7B/13B模型，单卡能放下，需要高吞吐 | 数据并行（FSDP） | - |
| 34B/70B模型，单卡放不下，同机有8卡NVLink | 张量并行 | 张量+流水线并行 |
| 70B以上大模型，多机部署 | 张量+流水线并行 | 张量+数据并行 |
| 超长上下文场景（100K+） | 序列并行+数据并行 | - |
| MoE架构模型部署（Mixtral/Qwen MoE等） | 专家并行+张量并行 | 专家并行+数据并行 |

### 2.7 主流混合并行组合方案（生产级标配）
实际生产环境很少用单一并行策略，通常会根据场景组合使用，叠加显存降低收益：
1. **TP+SP（张量并行+序列并行）：超长上下文场景标配**
   - 适用场景：70B+模型 + 64K+上下文，是当前各大厂商大模型服务的主流架构
   - 收益：TP降低模型权重显存N倍，SP降低KV Cache显存M倍，总显存降低N*M倍
   - 实例：70B FP16模型 + 128K上下文，用4卡TP+2卡SP，总共8张24GB A10G就能跑，单张A10G仅需存17GB模型+5GB KV缓存，而纯TP需要16卡，纯SP需要8卡80GB A100，成本降低60%以上
   - 支持度：vLLM 0.4+、TensorRT-LLM均原生完美支持
2. **EP+TP+SP（专家并行+张量并行+序列并行）：MoE超长上下文场景标配**
   - 适用场景：MoE大模型+超长上下文，比如Mixtral 8x22B 128K
   - 收益：三层并行叠加，显存降低倍数是三者乘积，普通消费级显卡也能跑超大规模MoE模型
3. **TP+PP（张量并行+流水线并行）：超大模型多机部署**
   - 适用场景：175B以上超大规模稠密模型多机部署，一般仅头部厂商会用到

## 3 通信硬件对比
不同互联硬件的带宽和延迟差异巨大，直接决定并行效率：
| 硬件 | 带宽 | 延迟 | 适用场景 |
|------|------|------|----------|
| PCIe 4.0 x16 | 32GB/s双向 | ~1us | 同机普通卡互联，适合数据并行、专家并行，不适合张量并行 |
| NVLink 3.0 | 600GB/s双向 | ~0.5us | 同机GPU互联，适合张量并行、专家并行，是当前主流 |
| NVLink 4.0 | 900GB/s双向 | ~0.3us | H100/H200专用，性能提升50% |
| InfiniBand HDR | 200Gbps（25GB/s） | ~1.5us | 多机互联，适合大集群数据并行/流水线并行/专家并行 |
| InfiniBand NDR | 400Gbps（50GB/s） | ~1us | 最新多机互联硬件，适合超大规模集群 |
### 3.1 并行效率参考
- 同机NVLink张量并行：效率>90%
- 同机PCIe张量并行：效率<60%
- 多机InfiniBand数据并行：效率>80%
- 多机普通TCP数据并行：效率<50%
- 同机/跨机专家并行：效率>85%，对硬件要求低，普通PCIe即可跑满

## 4 面试高频考点
### 开发岗通用题
#### Q1：**大模型推理显存占用由哪几部分组成？7B FP16模型100并发2048上下文需要多少显存？**
显存占用=模型权重+KV Cache+激活值+冗余。7B FP16权重占14GB，100并发2048上下文的KV Cache约占3.2GB，激活值约占0.8GB，加10%冗余总占用约18GB，所以需要24GB显存的卡（比如A10G）才能跑。

#### Q2：**张量并行和数据并行的区别是什么？分别适合什么场景？**
核心区别：
- 拆分方式不同：数据并行是按数据batch拆分，每个卡有完整模型；张量并行是按模型参数拆分，每个卡只有部分模型
- 显存变化：数据并行不降低单卡显存占用，张量并行单卡显存随卡数线性降低
- 通信要求：数据并行通信要求低，普通PCIe/InfiniBand就行；张量并行通信要求极高，必须要NVLink互联

适用场景：数据并行适合模型能放单卡、需要提升吞吐的场景；张量并行适合模型太大放不下单卡、同机有NVLink的场景。

#### Q3：**为什么KV Cache会随并发数线性增长？怎么优化KV Cache占用？**
因为每个并发请求都有自己独立的KV Cache，需要缓存每个请求的历史Key和Value，所以并发越多KV Cache占用越大。
优化手段：
- KV Cache量化，用INT4/INT8存KV，降低75%以上占用
- PagedAttention分页管理，减少碎片
- 动态释放长时间空闲的会话KV
- 限制最大上下文长度，限制最大并发数

#### Q4：**专家并行是什么？适合什么场景？**
专家并行是MoE（混合专家）架构模型特有的并行策略，核心是把不同的完整FFN专家分配到不同GPU上，每张卡仅存1/N的专家参数，显存降低N倍，仅激活对应专家时才需要通信，通信开销极低，部署效率>85%。是所有MoE架构模型（比如Mixtral 8x7B、Qwen MoE系列）生产部署的标配，普通24GB A10G就能跑8x7B级别的MoE模型，成本降低70%以上。

### 算法岗专项题
#### Q1：**KV Cache的访问模式有什么特点？为什么是显存带宽瓶颈？**
KV Cache的访问模式是：每生成一个Token，需要遍历整个历史KV做注意力计算，属于顺序访问+随机访问混合，而且每次生成都要读取整个KV Cache，访存量和上下文长度成正比。因为每次生成的计算量很小，但访存量很大，所以KV Cache的读写经常成为显存带宽瓶颈，尤其是长上下文场景。

#### Q2：**张量并行的通信开销为什么比数据并行高？**
因为数据并行只需要在每个batch结束后同步一次梯度，通信频率低；而张量并行每层计算都需要和其他卡通信，通信频率是每层一次，通信量和batch size无关，不管batch多大都要通信。而且张量并行的通信是all-reduce或者all-gather，比数据并行的梯度同步通信更频繁，所以要求更高的带宽。

#### Q3：**流水线并行的气泡开销是什么？怎么降低？**
气泡开销是指流水线启动和结束阶段，部分卡处于空闲状态的时间占比，比如4卡流水线并行，第1卡先算第一层，其他卡空闲；第1卡算完给第2卡，第1卡又空闲，直到所有卡都跑起来，结束阶段也是同理。
降低方法：
- 增大batch size，让流水线填充更满，气泡占比降低
- 微批次拆分，把大batch拆成小的微批次，重叠计算和通信
- 用 interleaved 流水线调度，减少气泡

#### Q4：**专家并行和张量并行有什么核心区别？为什么MoE模型优先用专家并行？**
核心区别：
- 拆分对象不同：张量并行是把单个参数张量按维度拆分，每层计算都需要通信；专家并行是把不同的完整专家分配到不同卡，只有路由到对应专家时才通信
- 通信要求不同：张量并行要求极高的带宽（必须NVLink），通信开销大；专家并行通信量极小，普通PCIe甚至跨机InfiniBand都能做到>85%的效率
- 适用范围不同：张量并行适合所有稠密模型，专家并行仅适合MoE架构模型

MoE优先用专家并行的原因是：MoE每层有多个独立专家，天然适合按专家拆分，效率远高于张量并行，对硬件要求低，成本大幅降低，是MoE部署的首选方案。

#### Q5：**为什么张量并行和序列并行可以同时使用？叠加收益是什么？**
两者拆分维度完全正交，互不冲突：张量并行是沿**隐藏层维度**拆分模型参数，序列并行是沿**序列长度维度**拆分输入/KV缓存，底层数学计算逻辑是等价的，结果和单卡完全一致，没有精度损失。
叠加收益是显存降低倍数是两者卡数的乘积：比如4卡TP+2卡SP，模型权重降低4倍，KV缓存降低2倍，总显存降低8倍，是70B+模型64K以上超长上下文场景的标准部署方案，成本可以降低60%以上，现在主流推理框架都已原生支持。
