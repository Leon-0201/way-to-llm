## 1. 一句话定义：梯度累积是什么

**梯度累积（Gradient Accumulation）**：在多次前向/反向传播中累积梯度，然后才执行一次优化器更新，从而在不增加显存占用的前提下，增大**等效 batch size**。

直觉理解：

- 正常训练：每处理 1 个 batch → 计算梯度 → 更新参数
- 梯度累积：每处理 $N$ 个 micro-batch → 累积 $N$ 次梯度 → 更新 1 次参数
- 效果等价于：把 $N$ 个 micro-batch 拼成 1 个大 batch 一起训练

---

## 2. 为什么需要梯度累积：三个核心动机

### 2.1 显存受限：GPU 装不下大 batch

这是 LLM 训练中最常见的场景。假设：

- 模型：7B 参数（FP16 约 14GB）
- GPU：单卡 24GB（如 RTX 3090/4090）
- 目标 batch size：512

如果直接 batch=512 一起前向/反向：

- 激活值（activation）占用随 batch 线性增长
- 梯度占用随 batch 线性增长
- 优化器状态（Adam 为 2×参数量）固定
- 很容易 OOM（Out Of Memory）

**梯度累积的解法**：

- 把 batch=512 拆成 8 个 micro-batch（每个 64）
- 每次只处理 64 个样本，显存占用大幅降低
- 累积 8 次梯度后再更新，等效 batch 仍为 512

### 2.2 训练稳定性：大 batch 带来更稳定的梯度

从优化角度看：

- 小 batch：梯度估计的**方差大**，训练曲线抖动明显
- 大 batch：梯度估计的**方差小**，训练更稳定

**重要澄清**：

- batch size **不影响梯度的期望**（均值）：无论 batch 大小，单个 step 的梯度都是总体梯度的无偏估计
- batch size **影响梯度的方差**：batch 越大，梯度估计越准确，波动越小

数学表述：

设单个样本的梯度为 $g_i$，batch size 为 $B$，则 batch 梯度为：

$
g_{\text{batch}} = \frac{1}{B}\sum_{i=1}^{B}g_i
$

- 期望：$\mathbb{E}[g_{\text{batch}}] = \mathbb{E}[g_i]$（与 $B$ 无关）
- 方差：$\text{Var}(g_{\text{batch}}) = \frac{1}{B}\text{Var}(g_i]$（与 $B$ 成反比）

但直接增大 batch 会 OOM，因此用梯度累积"曲线救国"。

### 2.3 学习率缩放：Batch Size 与 LR 的关系

很多研究（如 GPT 系列训练）表明：

- 增大 batch size 时，通常可以**同步增大学习率**
- 常见缩放规则：`lr ∝ batch_size`（线性缩放）或 `lr ∝ sqrt(batch_size)`

梯度累积让你能灵活调整等效 batch size，从而配合学习率日程。

## 3. 核心原理：梯度累积的数学推导

### 3.1 标准 SGD 更新（无累积）

设 batch 大小为 $B$，损失函数为 $\mathcal{L}$，参数为 $\theta$。

单步更新：

$$
g = \frac{1}{B}\sum_{i=1}^{B}\nabla_\theta \ell(x_i, y_i; \theta)
$$

$$
\theta \leftarrow \theta - \eta \cdot g
$$

其中 $\eta$ 是学习率。

### 3.2 梯度累积：拆成 $N$ 个 micro-batch

把 batch $B$ 拆成 $N$ 个 micro-batch，每个大小为 $M = B/N$。

**第 $k$ 个 micro-batch 的梯度**：

$$
g_k = \frac{1}{M}\sum_{i=(k-1)M+1}^{kM}\nabla_\theta \ell(x_i, y_i; \theta)
$$

**累积 $N$ 步后的总梯度**：

$$
g_{\text{accum}} = \frac{1}{N}\sum_{k=1}^{N} g_k
$$

代入 $g_k$ 的表达式：

$$
g_{\text{accum}} = \frac{1}{N}\sum_{k=1}^{N}\left(\frac{1}{M}\sum_{i=(k-1)M+1}^{kM}\nabla_\theta \ell(x_i, y_i; \theta)\right)
$$

$$
g_{\text{accum}} = \frac{1}{NM}\sum_{i=1}^{NM}\nabla_\theta \ell(x_i, y_i; \theta) = \frac{1}{B}\sum_{i=1}^{B}\nabla_\theta \ell(x_i, y_i; \theta)
$$

**结论**：梯度累积后的梯度 $g_{\text{accum}}$ 与直接处理整个 batch 的梯度 $g$ **完全相同**。

### 3.3 关键细节：梯度是"累加"还是"平均"

PyTorch 的默认行为：

- `loss.backward()` 会**累加**梯度到 `.grad` 字段
- 不会自动除以累积步数

因此有两种实现方式：

**方式 1：手动平均（推荐）**

```python
for i, (inputs, labels) in enumerate(micro_batches):
    outputs = model(inputs)
    loss = criterion(outputs, labels) / accumulation_steps  # 关键：先缩放 loss
    loss.backward()

if (i + 1) % accumulation_steps == 0:
    optimizer.step()
    optimizer.zero_grad()
```

**方式 2：不缩放 loss，最后手动除梯度**

```python
for i, (inputs, labels) in enumerate(micro_batches):
    outputs = model(inputs)
    loss = criterion(outputs, labels)
    loss.backward()

if (i + 1) % accumulation_steps == 0:
    # 手动对梯度求平均
    for param in model.parameters():
        param.grad /= accumulation_steps
    optimizer.step()
    optimizer.zero_grad()
```

**推荐方式 1**：代码更简洁，且与混合精度训练（AMP）兼容更好。

## 4. 工程实现：PyTorch 代码模板

### 4.1 基础模板（单卡、无混合精度）

```python
from torch.utils.data import DataLoader

# 配置
batch_size = 512
micro_batch_size = 64
accumulation_steps = batch_size // micro_batch_size  # = 8

dataloader = DataLoader(dataset, batch_size=micro_batch_size, shuffle=True)
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)

for epoch in range(num_epochs):
    for step, (inputs, labels) in enumerate(dataloader):
        # 前向传播
        outputs = model(inputs)
        loss = criterion(outputs, labels) / accumulation_steps
        
        # 反向传播（梯度累加）
        loss.backward()
        
        # 每 accumulation_steps 步更新一次
        if (step + 1) % accumulation_steps == 0:
            # 可选：梯度裁剪
            torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
            
            optimizer.step()
            optimizer.zero_grad()
        
        # 可选：记录日志（注意 loss 要还原）
        if step % 10 == 0:
            print(f"Step {step}, Loss: {loss.item() * accumulation_steps:.4f}")
```

### 4.2 与混合精度训练（AMP）结合

```python
from torch.cuda.amp import autocast, GradScaler

scaler = GradScaler()

for step, (inputs, labels) in enumerate(dataloader):
    inputs, labels = inputs.to('cuda'), labels.to('cuda')
    
    with autocast():
        outputs = model(inputs)
        loss = criterion(outputs, labels) / accumulation_steps
    
    # 缩放 loss 后反向传播
    scaler.scale(loss).backward()
    
    if (step + 1) % accumulation_steps == 0:
        # 可选：梯度裁剪（在 unscale 之后）
        scaler.unscale_(optimizer)
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
        
        scaler.step(optimizer)
        scaler.update()
        optimizer.zero_grad()
```

**关键点**：

- `loss / accumulation_steps` 的缩放要在 `autocast()` 内部
- 梯度裁剪需要在 `scaler.unscale_()` 之后进行
- `scaler.update()` 必须在每 $N$ 步更新时调用（不是每步）

### 4.3 处理最后不足一个 accumulation cycle 的 batch

实际训练中，最后一个 epoch 可能剩不到 $N$ 个 micro-batch。有两种策略：

**策略 1：立即更新（推荐）**

```python
for step, (inputs, labels) in enumerate(dataloader):
    outputs = model(inputs)
    loss = criterion(outputs, labels) / accumulation_steps
    loss.backward()
    
    # 两种情况触发更新：
    # 1. 累积满 N 步
    # 2. 到达最后一个 batch
    is_last_step = (step + 1) == len(dataloader)
    if (step + 1) % accumulation_steps == 0 or is_last_step:
        optimizer.step()
        optimizer.zero_grad()
```

**策略 2：丢弃剩余 batch（不推荐）**

- 简单但浪费数据，可能导致训练不稳定

## 5. 等效 Batch Size 的计算与调参

### 5.1 核心公式

$$
\text{等效 batch size} = \text{micro-batch size} \times \text{accumulation steps}
$$

示例：

- micro-batch = 32, accumulation_steps = 16 → 等效 batch = 512
- micro-batch = 16, accumulation_steps = 32 → 等效 batch = 512

### 5.2 如何选择 micro-batch size

**原则**：在显存允许范围内，尽量选大的 micro-batch。

考虑因素：

1. **显存上限**：
   - 模型参数 + 梯度 + 优化器状态 + 激活值 ≤ GPU 显存
   - 激活值随 micro-batch 线性增长
   - 用 `torch.cuda.max_memory_allocated()` 监控

2. **计算效率**：
   - micro-batch 太小 → GPU 利用率低（kernel 启动开销占比大）
   - micro-batch 太大 → OOM

3. **经验值**（针对 LLM）：
   - 7B 模型（FP16）：micro-batch ≈ 16-64（取决于序列长度）
   - 13B 模型（FP16）：micro-batch ≈ 8-32
   - 70B 模型（多卡）：每卡 micro-batch ≈ 2-8

### 5.3 梯度累积步数的选择

**原则**：在显存允许下，accumulation_steps 越小越好（更新频率高）。

**经验配置**：

| 模型规模 | GPU 显存 | micro-batch | accumulation_steps | 等效 batch |
| --- | --- | --- | --- | --- |
| 7B | 24GB | 32 | 16 | 512 |
| 7B | 40GB (A100) | 64 | 8 | 512 |
| 13B | 40GB | 32 | 16 | 512 |
| 70B | 8×40GB | 4 (每卡) | 16 | 512 |

## 6. 梯度累积的副作用与注意事项

### 6.1 训练速度：更新频率降低

- 累积 $N$ 步才更新 1 次 → 参数更新次数减少为原来的 $1/N$
- 但每步的计算量不变
- **收敛速度可能变慢**（同样 step 数下，更新次数少）

**对策**：

- 按 epoch 而非 step 评估训练进度
- 适当增加训练总 step 数

### 6.2 梯度噪声：累积后的梯度方差更低

- 单次 micro-batch 的梯度**方差大**（不是梯度本身大）
- 累积 $N$ 步后，梯度方差降低为原来的 $1/N$
- 效果等价于大 batch 训练：梯度期望不变，方差减小

**影响**：

- 训练曲线更平滑
- 可能需要更大的学习率（参考大 batch 训练的学习率缩放规则）

**重要澄清**：

- 梯度累积**不会放大梯度的期望值**（因为每步 loss 都除以了 accumulation_steps）
- 梯度累积**降低的是梯度的方差**，让更新方向更稳定

### 6.3 与 BatchNorm 的冲突（如果使用 BN）

**问题**：

- BatchNorm 的统计量（running mean/var）是按 micro-batch 计算的
- 梯度累积不会改变 BN 的统计量计算方式
- 导致 BN 的统计量与等效 batch 不匹配

**对策**：

- LLM 通常用 LayerNorm/RMSNorm，不受影响
- 如果用 CNN + BN，考虑：
  - 使用 SyncBatchNorm（跨卡同步统计量）
  - 或者不用梯度累积

### 6.4 与梯度裁剪的配合

**正确做法**：在累积完成后裁剪

**错误做法**：每步 micro-batch 都裁剪 → 梯度被多次裁剪，累积效果不对

### 6.5 与学习率日程的配合

**关键问题**：学习率应该按"实际 step"还是"更新 step"调整？

**推荐做法**：按**更新 step**（即考虑累积后的 step）调整学习率。

## 7. 分布式训练中的梯度累积

### 7.1 DDP（DistributedDataParallel）+ 梯度累积

在 DDP 中：

- 每卡的 gradient accumulation 独立进行
- `all_reduce` 在每次 `backward()` 时同步梯度
- 更新时机仍由每卡独立控制

**等效 batch size 计算**：

$$
\text{等效 batch} = \text{micro-batch} \times \text{accumulationsteps} \times \text{GPU 数量}
$$

示例：

- 8 卡 A100，每卡 micro-batch=8，accumulation_steps=8
- 等效 batch = 8 × 8 × 8 = 512

### 7.2 FSDP（FullyShardedDataParallel）+ 梯度累积

FSDP 中：

- 参数/梯度/优化器状态被分片到各卡
- 梯度累积的逻辑与 DDP 类似
- 但显存占用更低（适合大模型）

## 8. 面试高频问题清单

### 8.1 基础概念

**Q1: 梯度累积的核心思想是什么？**

A: 在多次前向/反向传播中累加梯度，然后才执行一次优化器更新。效果等价于增大 batch size，但不增加显存占用。

**Q2: 梯度累积为什么能节省显存？**

A: 因为每次只处理 micro-batch，激活值和梯度的峰值占用由 micro-batch 决定，而非等效 batch。

**Q3: 梯度累积的等效 batch size 怎么算？**

A: `等效 batch = micro-batch × accumulation_steps × GPU 数（分布式）`

### 8.2 工程实现

**Q4: PyTorch 中如何实现梯度累积？**

A: 关键点：
1. `loss / accumulation_steps` 后调用 `backward()`
2. 每 $N$ 步调用一次 `optimizer.step()` 和 `optimizer.zero_grad()`
3. 梯度会自动累加到 `.grad` 字段

**Q5: 梯度累积时，loss 为什么要除以 accumulation_steps？**

A: 因为 PyTorch 的 `backward()` 会累加梯度。如果不缩放 loss，累积后的梯度会是原始梯度的 $N$ 倍，导致更新步长过大。

**Q6: 梯度累积与混合精度训练如何配合？**

A: 
- `loss / accumulation_steps` 在 `autocast()` 内部
- `scaler.scale(loss).backward()`
- 每 $N$ 步调用一次 `scaler.step()` 和 `scaler.update()`

### 8.3 进阶问题

**Q7: 梯度累积对训练速度有什么影响？**

A: 
- 参数更新频率降低为 $1/N$
- 同样 step 数下，收敛可能变慢（因为更新次数少）
- 但梯度估计的方差更低，训练曲线更平滑
- 可以通过增大学习率补偿（参考大 batch 训练的学习率缩放规则）

**注意**：梯度累积不改变梯度的期望值，只影响梯度的方差。

**Q8: 梯度累积与 BatchNorm 有冲突吗？**

A: 
- 有冲突。BN 的 running stats 按 micro-batch 计算，与等效 batch 不匹配。
- LLM 用 LayerNorm/RMSNorm，不受影响。

**Q9: 梯度裁剪应该在什么时候进行？**

A: 在梯度累积完成后、`optimizer.step()` 之前进行。不能在每步 micro-batch 都裁剪。

## 9. 实战建议与常见坑点

### 9.1 调参顺序

1. 先确定**目标等效 batch size**（如 512）
2. 测试单卡能容纳的**最大 micro-batch**（逐步增大直到 OOM）
3. 计算需要的 **accumulation_steps**
4. 根据等效 batch 调整**学习率**（线性或 sqrt 缩放）
5. 监控训练曲线，必要时微调

### 9.2 常见坑点

- **忘记除以 accumulation_steps**：梯度爆炸或训练发散
- **梯度清零时机错误**：每步都 `zero_grad()` → 累积失效
- **学习率未调整**：等效 batch 增大后，学习率仍用原值 → 收敛慢
- **日志记录混乱**：记录的 loss 是缩放后的，忘记还原
- **最后一个 batch 处理不当**：剩余样本不足 accumulation_steps 时直接丢弃

### 9.3 监控指标

建议记录：

- 实际梯度范数（验证累积是否正确）
- 有效学习率（考虑 warmup 和累积）
- 显存占用峰值（验证 micro-batch 选择是否合理）
- 更新频率（每 epoch 实际更新次数）

## 10. 总结

梯度累积是 LLM 训练中**必备的基础技术**，核心价值：

1. **突破显存限制**：用小显存 GPU 训练大模型
2. **灵活调整 batch size**：配合学习率缩放规则
3. **训练稳定**：梯度噪声更低，收敛更平滑

**关键公式**：

- 等效 batch = micro-batch × accumulation_steps
- Loss 缩放：`loss / accumulation_steps`
- 更新频率：每 $N$ 步更新 1 次

**工程要点**：

- 梯度自动累加，需手动缩放 loss
- 梯度裁剪在累积完成后进行
- 学习率按"更新 step"而非"micro-batch step"调整
- 与混合精度、分布式训练兼容良好

掌握梯度累积后，可以进一步学习：

- 混合精度训练（进一步降低显存）
- 梯度检查点（用计算换显存）
- 分布式训练（多卡/多机扩展）
