# 检查点机制（Checkpoint）：训练中断与恢复

本节面向 LLM 训练/微调工程的“训练可靠性”与“可恢复性”。你需要能回答：

- 为什么训练一定要做 checkpoint？（尤其是长训练 + 抢占式集群/spot）
- checkpoint 到底要存什么？只存 `model.state_dict()` 够不够？
- 分布式（DDP/FSDP/ZeRO）怎么存、怎么恢复？
- 保存频率/保留策略怎么定？怎么避免写坏/写半截？

> 常见混淆：本节的 checkpoint 指“**训练状态快照**”。
> 另一个同名概念是 **gradient checkpointing（激活重计算）**，它是显存优化手段，不是保存训练进度；两者完全不同。

和其它章节的关系：
- 学习率日程与恢复的 `global_step` 对齐见《梯度问题》《优化器》
- 混合精度的 AMP scaler 状态见《混合精度训练》
- 分布式并行与 ZeRO/FSDP 见第三章并行技术

---

## 1. 一句话定义：Checkpoint 是“可恢复的训练快照”

你可以把一次训练看成一个状态机：

- 输入：数据流（DataLoader）、随机性（seed）、超参、代码版本
- 状态：参数、优化器统计量、scheduler 计数器、混合精度 scaler、RNG 状态……
- 输出：模型权重 + 训练日志

**checkpoint 的目标**：在任意时刻把“足以继续训练”的状态落盘；一旦进程中断，可以从 checkpoint 恢复到近似同一条训练轨迹（至少数值上稳定且可继续收敛）。

---

## 2. 要存什么：按“能否无损恢复训练”分层

### 2.1 必须存（想要“继续训练”就必须）

1) **模型参数**：`model.state_dict()`
2) **优化器状态**：`optimizer.state_dict()`
	- 对 Adam/AdamW 包含一阶矩/二阶矩（`exp_avg/exp_avg_sq`），不存会导致恢复后训练动态突变
3) **学习率调度器状态**：`lr_scheduler.state_dict()`（或至少存 `global_step`）
	- 否则会出现：恢复后 lr 位置不对（warmup/decay 重新开始），训练曲线会断崖
4) **全局步数/epoch**：`global_step`、`epoch`、`micro_step`（如果你需要精确恢复梯度累积）

### 2.2 强烈建议存（想要“更可复现/更稳”）

5) **随机数状态（RNG states）**：
	- Python `random`、NumPy、PyTorch CPU/CUDA RNG
	- 分布式下每个 rank 的 RNG 状态可能不同
6) **混合精度 scaler**（AMP）：`GradScaler.state_dict()`
	- 不存会导致恢复后 scaler 重新爬升/下降，前期更容易溢出或过度保守
7) **数据采样器/数据游标**：
	- 例如 `DistributedSampler` 的 `epoch`，或你自定义的数据游标/offset
	- 否则恢复后会重复/跳过样本，影响训练统计与可复现

### 2.3 工程可追溯信息（面试加分项）

8) **配置与版本**：
    - 训练超参数（学习率 learning rate、权重衰减 weight decay、学习率预热步数 warmup steps、梯度累积步数 gradient accumulation steps、混合精度数据类型 bfloat16/float16、梯度裁剪 gradient clipping 等）
	- 代码版本（git commit hash）、数据版本（数据集 snapshot id）、tokenizer 版本
9) **评估指标与最优模型信息**：`best_metric`、`best_ckpt_path`（方便自动选最佳）

> 关键对比：
> - “只存模型权重”适用于 **推理/部署**。
> - “能无缝继续训练”需要 **模型 + 优化器 + scheduler + step +（最好还有 自动混合精度/RNG/数据状态）**。

## 3. 保存策略：保存频率、保留策略、以及“写坏”防护

### 3.1 什么时候保存

常见三种策略可以组合：

- **按 step 保存**：每 N step（最常见；适合长训练）
- **按时间保存**：每 M 分钟（适合 spot/preempt 环境，避免刚好错过保存点）
- **按指标保存**：验证集指标更好就保存 `best`（用于最终选模）

工程上通常同时保留：

- `last`：最近一次（可恢复训练）
- `best`：验证最好的一次（用于推理/发布）
- `periodic`：每隔一段保存的历史（用于回溯/排查）

### 3.2 保留策略（retention）

- 只保留最近 K 个（rolling）+ 一个 best
- 也可用“指数间隔”保留：越久远越稀疏（排查回归/不稳定时很有用）

### 3.3 防止写半截/写坏（非常重要）

推荐的落盘方式：

1) 先写到临时目录：`ckpt_stepXXXX.tmp/`
2) 写完并 `fsync`/关闭句柄
3) 再原子重命名：`mv ckpt_stepXXXX.tmp ckpt_stepXXXX`

这样即使机器中断，也不会得到“看似存在但不完整”的 checkpoint。

## 4. 分布式训练下怎么做（DDP / FSDP / ZeRO）

大原则：**谁持有“完整状态”，谁就负责保存；谁持有“分片状态”，就要用框架提供的 sharded checkpoint 机制。**

### 4.1 DDP（Distributed Data Parallel）

常见做法：

- **只让 rank 0 写盘**（避免多进程同时写同一路径）
- 保存 `model.module.state_dict()`（DDP 包装下真实模型在 `module`）
- 恢复时各 rank 都从同一 checkpoint 读，然后各自 `load_state_dict`

要点：

- 优化器状态每个 rank 通常一致（数据并行下），但仍建议各 rank 都 `load_state_dict` 保持一致。
- DataLoader 的 `DistributedSampler` 要设置同样的 `epoch`，否则各 rank 的 shuffle 不一致。

### 4.2 FSDP（Fully Sharded Data Parallel）

FSDP 会把参数/优化器状态分片到各个 rank，checkpoint 也分两类：

- **Full checkpoint**：聚合成完整权重再保存（更通用，但保存时可能需要额外 CPU/GPU 内存）
- **Sharded checkpoint**：各 rank 保存自己的 shard（更省内存/更快，但恢复时需要同等并行布局或兼容逻辑）

实践建议：

- 训练恢复优先用 sharded checkpoint（成本低）
- 要发布/推理时，再额外导出一个“全量权重”（例如 HF 格式 / safetensors）

### 4.3 ZeRO（DeepSpeed）

ZeRO（尤其 stage 2/3）同样会切分优化器状态/参数：

- 需要使用 DeepSpeed 提供的 `engine.save_checkpoint()` / `engine.load_checkpoint()`
- 不要自己直接 `torch.save(model.state_dict())` 试图恢复训练（多半不够/不一致）

工程建议：

- checkpoint 路径用“目录”而不是单文件（DeepSpeed 会生成多文件 shard）
- 同步保存训练配置与 deepspeed config，便于恢复时校验

## 5. 大模型训练的现实问题：checkpoint 很大怎么办

### 5.1 体量来源

- **优化器状态**通常比模型权重更大（Adam 的一阶矩 + 二阶矩，理论上约 2× 参数量级，外加一些开销）。
- 如果你还保存 指数滑动平均 / 随机权重平均 权重、副本，会更大。

### 5.2 常见优化手段（根据需求选）

- 只在“可恢复训练” checkpoint 中保存优化器；在“发布/推理” checkpoint 中只保存模型权重。
- 降低保存频率 + 只保留最近 K 个。
- 使用 sharded checkpoint（FSDP/ZeRO）。
- 如果训练环境允许，做异步保存（后台线程/进程写盘），避免阻塞训练。

## 6. 恢复训练的流程：要点是对齐 `global_step`

一个正确的恢复流程通常是：

1) 构建 model/optimizer/scheduler/scaler（结构与超参必须匹配）
2) `load_checkpoint` 恢复它们的 state_dict
3) 恢复 `global_step/epoch`，并用它们驱动：
	- scheduler 当前步（lr 位置）
	- 训练日志的 step 计数
	- 数据采样器的 epoch/seed
4) 继续训练

最常见的“恢复后不收敛/曲线断裂”原因就是：

- scheduler 没恢复（warmup 又从头开始、或 decay 位置错了）
- optimizer 没恢复（Adam 矩估计清零导致更新动态突变）
- 自动混合精度 scaler 没恢复（溢出策略突变）

---

## 7. 常见坑点清单（面试常问）

- 只存 `model.state_dict()`：能推理，但**不能无缝继续训练**。
- 忘了存/恢复 `global_step`：scheduler 步数错位。
- 自动混合精度：没 `unscale_` 不影响保存，但恢复时不带 scaler 会导致早期不稳定。
- DDP：保存时忘了用 `model.module`。
- FSDP/ZeRO：自己手写保存逻辑，结果保存的不是完整/一致的状态。
- checkpoint 写盘被抢占中断，留下半截文件：需要临时目录 + 原子重命名。
- 恢复后 DataLoader shuffle 不一致：sampler 的 epoch/seed 不一致。

## 8. 面试高频问答（速答模板）

### Q1：checkpoint 里最少要存哪些才能继续训练？
答：至少模型参数 + 优化器状态 + scheduler 状态/全局步数；混合精度还要存 scaler，想更可复现再加 RNG 和数据游标。

### Q2：为什么只存模型参数不够？
答：因为 AdamW 的动量/二阶矩、以及学习率日程的当前位置都会丢；恢复后更新动态突变，曲线可能断裂甚至不收敛。

### Q3：分布式（FSDP/ZeRO）怎么 checkpoint？
答：参数/优化器是分片的，必须用框架的 sharded checkpoint API（或显式聚合 full state dict）；DDP 通常 rank0 保存即可。

### Q4：checkpoint 保存频率怎么定？
答：平衡训练中断风险与 IO 成本；常用每 N step 或每 M 分钟保存 `last`，并保留 best + 最近 K 个；大模型下优先 sharded checkpoint。
