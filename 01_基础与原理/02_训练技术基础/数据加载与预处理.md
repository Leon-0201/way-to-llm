## 1 数据加载的核心组件：Dataset 与 DataLoader

### 1.1 统一表述：PyTorch 数据加载架构

PyTorch 的数据加载流程可以概括为：

```
数据文件 → Dataset（封装样本） → DataLoader（批量加载） → 训练循环
```

**Dataset 的职责**：定义"如何读取单个样本"
- 实现 `__len__()` 方法：返回数据集大小
- 实现 `__getitem__()` 方法：根据索引读取并预处理单个样本
- 对于 LLM：通常包括读取文本、分词、返回 input_ids 和 attention_mask

**DataLoader 的职责**：定义"如何批量加载样本"
- 自动批处理（将多个样本合并为 batch）
- 多进程加载（通过 num_workers 参数）
- 数据打乱（shuffle）
- 预取机制（prefetch）

### 1.2 DataLoader 的关键参数详解

**核心参数（面试高频）**：

| 参数 | 作用 | 推荐值 | 注意事项 |
| --- | --- | --- | --- |
| `batch_size` | 每个 batch 的样本数 | 根据显存调整 | 分布式下是**每卡**的 batch size |
| `shuffle` | 是否打乱数据 | 训练集=True | 分布式下由 Sampler 控制 |
| `num_workers` | 数据加载进程数 | 4-8 | 太大反而慢（进程切换开销） |
| `pin_memory` | 锁定内存 | True（有 GPU 时） | 加速 CPU→GPU 传输 |
| `drop_last` | 丢弃不完整 batch | 训练=True | 避免 BatchNorm 问题 |
| `prefetch_factor` | 预取 batch 数 | 2-4 | 越大越流畅，但内存占用高 |
| `persistent_workers` | 持久化 worker | True | 减少 epoch 间重启开销 |

**面试要点**：
- `num_workers` 不是越大越好，通常是 CPU 核心数的 50-80%
- `pin_memory=True` 可以加速 PCIe 传输，但会占用更多系统内存
- `drop_last=True` 避免最后一个 batch 太小导致 BatchNorm 统计量计算异常

### 1.3 多进程加载的工作原理

**架构示意**：

```
主进程（训练循环）
    ↓ 从 queue 获取 batch
    ↓
DataLoader（启动 num_workers 个子进程）
    ↓
Worker 进程 1 → 读取样本 → 放入 queue
Worker 进程 2 → 读取样本 → 放入 queue
Worker 进程 3 → 读取样本 → 放入 queue
Worker 进程 4 → 读取样本 → 放入 queue
```

**关键机制**：

1. **并行加载**：每个 worker 进程独立执行 Dataset 的 `__getitem__()` 方法
2. **进程间通信（IPC）**：通过 Queue 将处理好的 batch 传递给主进程
3. **预取（Prefetching）**：worker 提前准备下一个 batch，避免 GPU 等待数据

**性能瓶颈分析**：

- **理想情况**：GPU 计算时间 > 数据加载时间 → GPU 利用率接近 100%
- **瓶颈情况**：GPU 计算时间 < 数据加载时间 → GPU 等待数据，利用率低
- **解决方案**：增加 num_workers、优化预处理、使用更快的存储

## 2 分布式训练中的数据加载

### 2.1 核心问题：如何避免数据重复？

在分布式训练（DDP/FSDP）中的挑战：

- N 张 GPU，每张卡有一个训练进程（称为 rank）
- **目标**：每个样本在每个 epoch 中**只被一个 rank 使用**
- **错误做法**：每个 rank 都独立 shuffle 自己的数据 → 导致数据重复 N 倍

### 2.2 解决方案：DistributedSampler

**使用方式**：

```python
from torch.utils.data.distributed import DistributedSampler

# 创建 DistributedSampler
sampler = DistributedSampler(
    dataset,
    num_replicas=world_size,  # 总 GPU 数
    rank=local_rank,          # 当前 GPU 的 rank（0 到 world_size-1）
    shuffle=True,
    seed=42                   # 随机种子（所有 rank 相同）
)

# 创建 DataLoader（使用 sampler，不再设置 shuffle=True）
dataloader = DataLoader(
    dataset,
    batch_size=32,
    sampler=sampler,
    num_workers=4
)

# 训练循环中，每个 epoch 调用
for epoch in range(num_epochs):
    sampler.set_epoch(epoch)  # 关键：确保每个 epoch 不同的 shuffle
    for batch in dataloader:
        # 训练代码
```

**面试关键点**：
- 必须使用 `DistributedSampler`，不能简单地在每个 rank 上 shuffle
- 每个 epoch 必须调用 `sampler.set_epoch(epoch)`，否则每个 epoch 的数据顺序相同
- `seed` 参数在所有 rank 上必须相同，但通过 `set_epoch()` 实现每个 epoch 不同的 shuffle

### 2.3 DistributedSampler 的工作原理

**数据划分策略**：

假设有 1000 个样本，4 张 GPU：
- rank 0：样本索引 [0, 4, 8, 12, ...] ← 每隔 4 个取一个
- rank 1：样本索引 [1, 5, 9, 13, ...]
- rank 2：样本索引 [2, 6, 10, 14, ...]
- rank 3：样本索引 [3, 7, 11, 15, ...]

**数学表述**：

设总样本数为 $N$，GPU 数为 $K$，rank 为 $r$（$0 \leq r < K$）：
- rank $r$ 的样本索引：$\{i \mid i \equiv r \pmod K\}$
- 每个 rank 的样本数：$\lfloor N/K \rfloor$

**关键理解**：
- 每个 rank 只看到部分样本（1000/4 = 250 个）
- 但所有 rank 合起来覆盖全部 1000 个样本
- 通过取模运算确保**无重叠**

## 3 LLM 数据预处理的完整流水线

### 3.1 从原始文本到训练数据

LLM 训练数据的完整处理流程：

```
原始文本 → 文本清洗 → 文本规范化 → 分词 → 截断/填充 → 打包/拼接 → Token IDs
```

每个步骤的目标和内容：

1. **文本清洗**：去除噪声、不可见字符、格式问题
2. **文本规范化**：统一书写格式，减少"同义不同形"
3. **分词**：将文本转换为 Token IDs
4. **截断/填充**：处理序列长度，满足模型输入要求
5. **数据打包**：减少 padding 浪费，提高训练效率

### 3.2 文本清洗（Text Cleaning）

**目标**：去除数据中的噪声和不必要的字符

**常见清洗操作**：

1. **移除控制字符**：
   - 删除 Unicode 控制字符（category 以'C'开头）
   - 保留常用字符如换行符（\n）、制表符（\t）

2. **移除 HTML 标签**（网页数据）：
   - 删除 `<script>`、`<style>` 等标签及其内容
   - 删除所有 HTML 标签，只保留纯文本

3. **规范化空白字符**：
   - 多个连续空格合并为一个
   - 移除行尾空格
   - 统一换行符（\r\n → \n）

4. **处理编码问题**：
   - 移除或替换乱码字符
   - 统一使用 UTF-8 编码

**不同数据类型的清洗重点**：

| 数据类型 | 清洗重点 | 示例 |
| --- | --- | --- |
| 网页爬取 | HTML 标签、JavaScript、广告 | `<script>...</script>` |
| PDF 转换 | 多余换行、连字符、页眉页脚 | `train-\ning → training` |
| 社交媒体 | @提及、#标签、URL | `@user http://...` |
| 代码数据 | 注释、docstring（可选） | `# comment` |

### 3.3 文本规范化（Text Normalization）

**目标**：统一书写格式，减少"同义不同形"的情况

**常见规范化操作**：

1. **Unicode 规范化**：
   - 使用 NFKC 或 NFKD 形式
   - 合并兼容字符（如上标数字、连字）

2. **全角/半角转换**（中文场景）：
   - 全角字符转半角（如 A → A，1 → 1）
   - 统一标点符号（如"，" → ","）

3. **统一引号和标点**：
   - 中文引号转英文引号（或反之，根据需求）
   - 统一破折号（— → --）
   - 统一省略号（… → ...）

4. **空白字符统一**：
   - \r\n → \n（Windows 换行转 Unix）
   - \t → 空格（制表符转空格）

**面试高频**：为什么 LLM 需要文本规范化？

**答案要点**：
1. **减少词表压力**：避免"同义词不同写法"占用宝贵的词表空间
2. **提升泛化能力**：模型学到的是语义，而非表面格式差异
3. **训练/推理一致性**：确保线上输入和训练数据的处理方式一致
4. **提升训练效率**：减少重复模式，加速收敛

### 3.4 分词（Tokenization）

**目标**：将文本转换为模型可以理解的 Token IDs

**关键概念**：

| 术语 | 含义 | 示例 |
| --- | --- | --- |
| Token | 分词的基本单元 | `"hello"` → `["hello"]` |
| Vocabulary | 词表（所有 token 的集合） | LLaMA-2: 32,000 tokens |
| BOS/EOS | 开始/结束标记 | `<s>`, `</s>` |
| PAD | 填充标记 | `[PAD]`（用于补齐长度） |
| Attention Mask | 注意力掩码 | `[1,1,1,0,0]`（1=真实 token，0=填充） |

**LLM 常用的分词算法**：
- **BPE（Byte Pair Encoding）**：GPT 系列、LLaMA 采用
- **WordPiece**：BERT 采用
- **Unigram**：SentencePiece 默认算法
- **Word**：按单词分词（英文常用）

**分词过程的注意事项**：
- 添加特殊 token（BOS、EOS、PAD）
- 处理未知词（UNK token）
- 大小写处理（是否保留大小写信息）

### 3.5 截断与填充（Truncation & Padding）

**问题背景**：LLM 有最大序列长度限制（如 LLaMA-2: 4096 tokens）

**截断策略**：

1. **从右侧截断**（默认）：
   - 保留文本开头部分
   - 适合大多数场景

2. **从左侧截断**：
   - 保留文本末尾部分
   - 适合对话场景（保留最近的对话内容）

3. **按句子截断**（更智能）：
   - 以句子为单位进行截断
   - 避免在句子中间切断

**填充策略**：

1. **Batch 内填充**（推荐）：
   - padding 到 batch 内最长样本的长度
   - 减少不必要的填充
   - 需要自定义 collate_fn

2. **固定长度填充**：
   - 所有样本 padding 到固定长度（如 512）
   - 实现简单，但可能浪费计算

3. **不填充**：
   - 每个样本保持原始长度
   - 需要自定义 batch 合并逻辑

**面试要点**：
- 填充会导致计算浪费（模型需要处理 PAD token）
- 应该尽量减少 padding 的比例
- 可以使用动态批处理（按长度分组）来优化

### 3.6 数据打包与拼接（Packing & Concatenation）

**问题**：如果样本长度差异大，padding 会浪费大量计算

**示例**：
```
样本 1: [token, token, token, PAD, PAD]  ← 40% 是填充
样本 2: [token, token, token, token, token]
样本 3: [token, PAD, PAD, PAD, PAD]  ← 80% 是填充
```

**解决方案 1：样本拼接（Concatenation）**

**思路**：将多个短样本拼接成一个长样本

**实现要点**：
- 在样本之间添加 EOS token 作为分隔
- 拼接后的总长度不超过最大序列长度
- 可以显著提高 GPU 利用率

**解决方案 2：动态批处理（Dynamic Batching）**

**思路**：根据样本长度动态分组，减少 padding 浪费

**实现要点**：
- 将长度相近的样本分到同一 batch
- 可以设置最大 token 数而非固定 batch size
- 显著减少 padding 比例

**面试要点**：
- 数据打包可以将训练效率提升 20-50%
- 是大规模 LLM 训练的标配技术
- 需要在数据预处理阶段实现

## 4 性能优化：解决数据加载瓶颈

### 4.1 诊断数据加载瓶颈

**判断标准**：

1. **GPU 利用率监控**：
   - 使用 `nvidia-smi` 或 `nvtop` 命令
   - 如果 GPU 利用率 < 80%，很可能数据加载是瓶颈

2. **时间分析**：
   - 分别记录数据加载时间和训练时间
   - 如果 `data_time > 0.3 × train_time`，说明数据加载太慢

3. **PyTorch Profiler**：
   - 使用 PyTorch 自带的性能分析工具
   - 可以精确看到每个操作的时间占比

### 4.2 优化策略（面试高频）

**策略 1：增加 num_workers**

- **原理**：使用多进程并行加载数据
- **推荐值**：CPU 核心数的 50-80%
- **注意事项**：
  - 不是越大越好，进程切换有开销
  - 如果预处理很简单，增加 num_workers 效果有限
  - 需要测试找到最优值

**策略 2：优化 Dataset 的 `__getitem__()`**

- **慢的实现**：每次都读取文件、每次都分词
- **快的实现**：
  - 启动时一次性加载所有数据到内存
  - 启动时预分词（如果内存允许）
  - 避免在 `__getitem__()` 中进行复杂计算

**策略 3：使用更快的存储**

存储类型对比（读取速度）：
- HDD：100-200 MB/s ← **不推荐**
- SATA SSD：500 MB/s ← 可接受
- NVMe SSD：3-7 GB/s ← **推荐**
- 内存文件系统（tmpfs）：10+ GB/s ← 最快

**实用技巧**：在 Linux 系统上，可以将数据复制到 `/dev/shm`（内存文件系统）

**策略 4：使用高效数据格式**

- **HDF5 格式**：适合大规模数据，支持随机访问
- **Arrow/Parquet 格式**：适合表格数据，读取速度快
- **TFRecord 格式**：TensorFlow 的数据格式，也可用于 PyTorch

**策略 5：数据预加载**

- **原理**：提前准备下一个 batch，避免 GPU 等待
- **实现**：通过 `prefetch_factor` 参数控制
- **权衡**：预取越多越流畅，但内存占用越高

### 4.3 优化效果评估

**优化前后的对比指标**：

1. **GPU 利用率**：从 50% 提升到 90%+
2. **每秒处理的 batch 数**：提升 2-5 倍
3. **训练总时间**：显著缩短

**面试要点**：
- 好的数据流水线可以让训练速度提升 2-5 倍
- 优化是一个迭代过程，需要不断测试和调整
- 优先考虑收益最大的优化（如存储升级、预加载）

## 5 常见陷阱与解决方案

### 5.1 数据泄漏（Data Leakage）

**问题**：训练集和测试集/验证集有重叠，导致评估结果过于乐观

**常见错误**：
- 先 shuffle 再 split，如果数据有重复会导致泄漏
- 文档级去重不彻底，同一篇文章出现在 train 和 test 中
- 数据预处理（如标准化）使用了全量数据的统计量

**正确做法**：
1. **严格按样本 ID 划分**：
   - 先获取所有样本 ID
   - 使用 `train_test_split()` 划分 ID
   - 根据 ID 读取数据

2. **确保无重叠**：
   - 使用 `assert` 检查 train 和 test 的交集为空
   - 文档级去重要在划分之前完成

3. **预处理顺序**：
   - 只用训练集拟合预处理参数（如标准化的 mean/std）
   - 验证集和测试集使用训练集的参数

**面试要点**：
- 数据泄漏是面试中常见的问题
- 会导致模型在实际应用中表现远低于评估结果
- 必须在数据划分和预处理阶段就避免

### 5.2 随机性控制

**问题**：训练不可复现，每次运行结果不同

**需要控制的随机源**：

1. **Python 随机**：`random.seed()`
2. **NumPy 随机**：`np.random.seed()`
3. **PyTorch CPU 随机**：`torch.manual_seed()`
4. **PyTorch GPU 随机**：`torch.cuda.manual_seed_all()`
5. **cuDNN 行为**：设置 `torch.backends.cudnn.deterministic=True`

**分布式场景的特殊处理**：

- 每个 rank 需要**不同的数据 shuffle 种子**
- 实现方式：`rank_seed = base_seed + rank`
- 确保每个 rank 的 shuffle 不同但可复现

**面试要点**：
- 可复现性是科研和工程的基本要求
- 需要设置所有可能的随机源
- 分布式训练时每个 rank 的种子要不同

### 5.3 数据不平衡

**问题**：某些类别/领域的样本过多，导致模型偏向这些类别

**检测方法**：
- 统计每个类别的样本数
- 计算类别分布的比例

**解决方案**：

1. **加权采样**：
   - 给少数类更高的采样权重
   - 使用 `WeightedRandomSampler`

2. **过采样/欠采样**：
   - 过采样：复制少数类样本
   - 欠采样：随机删除多数类样本

3. **在 Loss 中使用 class_weight**：
   - 给少数类的 loss 更高的权重
   - 很多 loss 函数支持 `weight` 参数

**面试要点**：
- 数据不平衡是实际项目中的常见问题
- 需要先检测再处理
- 不同方案有不同的适用场景

### 5.4 内存泄漏

**问题**：训练过程中内存持续增长，最终 OOM

**常见原因**：

1. **在循环中累积 tensor**：
   - 错误：忘记 `detach()` 或 `.cpu()`
   - 正确：`output.detach().cpu()`

2. **梯度未清零**：
   - 错误：忘记 `optimizer.zero_grad()`
   - 正确：每步训练后清零梯度

3. **Dataset 中持有大对象**：
   - 错误：使用无限增长的缓存
   - 正确：使用 LRU 缓存或限制缓存大小

**调试方法**：
- 使用 `torch.cuda.max_memory_allocated()` 监控显存
- 使用 memory profiler 工具分析内存使用

**面试要点**：
- 内存泄漏是训练中的常见问题
- 需要理解 PyTorch 的内存管理机制
- 及时释放不需要的 tensor

## 6 实战建议与最佳实践

### 6.1 性能检查清单

训练前检查项目：

- [ ] 数据已预处理并缓存（避免训练时重复计算）
- [ ] 使用 DistributedSampler（分布式场景）
- [ ] num_workers 设置为 CPU 核心数的 50-80%
- [ ] pin_memory=True（有 GPU 时）
- [ ] 数据存储在 SSD（而非 HDD 或网络存储）
- [ ] 测试一个 epoch，确认 GPU 利用率 > 80%
- [ ] 设置了随机种子（保证可复现性）
- [ ] 检查了数据分布（避免严重不平衡）

### 6.2 调试技巧

**数据验证**：

在训练开始前，检查数据是否正确：

1. **检查形状**：batch 的维度是否符合预期
2. **检查特殊 token**：BOS/EOS 是否正确添加
3. **检查 attention mask**：是否正确标识了 padding 位置
4. **检查 token 范围**：token IDs 是否在词表范围内

**性能基准测试**：

- 运行几个 epoch，记录平均每个 batch 的处理时间
- 对比不同配置下的吞吐量（batches/sec）
- 找到最优的 num_workers 和 prefetch_factor 组合

---

## 7 总结

数据加载与预处理是 LLM 训练中**容易被忽视但至关重要**的环节。

**核心要点**：

1. **DataLoader 是多进程架构**：
   - 主进程负责训练，worker 进程负责加载
   - 通过 Queue 通信，prefetch 机制避免 GPU 等待
   - 合理配置 num_workers 可以显著提升性能

2. **分布式训练需要特殊处理**：
   - DistributedSampler 避免数据重复
   - 每个 epoch 调用 `set_epoch()` 确保 shuffle
   - 等效 batch = 每卡 batch × GPU 数 × 梯度累积

3. **LLM 数据预处理流水线**：
   - 清洗 → 规范化 → 分词 → 截断/填充 → 打包
   - 确保训练/推理处理一致
   - 文本规范化对 LLM 尤其重要

4. **性能优化是关键**：
   - 诊断瓶颈（GPU 利用率、计时分析）
   - 优化策略（num_workers、存储、数据格式）
   - 目标：GPU 利用率 > 80%

5. **避免常见陷阱**：
   - 数据泄漏（严格划分 train/test）
   - 随机性失控（设置所有随机源）
   - 内存泄漏（及时释放、避免累积）

**工程价值**：

- 好的数据流水线可以让训练速度提升 **2-5 倍**
- 减少调试时间（数据问题是最常见的 bug 来源）
- 提升模型效果（数据质量决定模型上限）

**面试准备建议**：

- 理解 DataLoader 的工作原理（多进程、预取）
- 掌握分布式数据加载的核心问题（DistributedSampler）
- 熟悉 LLM 数据预处理的每个步骤及其目的
- 了解性能优化的常见策略和权衡
- 能够识别和解决常见的数据陷阱

掌握数据加载与预处理，是成为合格 LLM 工程师的**必备技能**！

## 面试高频考点
### 算法岗通用题

#### Q1：**DataLoader 的 num_workers 是如何工作的？**
- num_workers 指定子进程数，用于并行加载数据
- 每个 worker 进程独立执行 Dataset 的 `__getitem__()` 方法
- 通过 Queue 将处理好的 batch 传递给主进程
- 推荐值是 CPU 核心数的 50-80%，不是越大越好
- 如果预处理很简单，增加 num_workers 效果有限

#### Q2：**分布式训练中如何避免数据重复？**
- 使用 DistributedSampler
- 每个 rank 只读取部分数据（按 rank 取模划分）
- 每个 epoch 调用 `sampler.set_epoch(epoch)` 确保 shuffle
- 这样可以保证每个样本在每个 epoch 中只被一个 rank 使用

#### Q3：**pin_memory=True 的作用是什么？**
- 将 CPU 内存锁定为 page-locked memory
- GPU 可以直接通过 DMA 读取，无需经过 CPU 中转
- 加速 CPU → GPU 数据传输（PCIe 带宽利用率更高）
- 缺点是会占用更多系统内存
- 只在有 GPU 训练时使用

#### Q4：**如何诊断数据加载是否成为瓶颈？**
- 监控 GPU 利用率（< 80% 说明可能有瓶颈）
- 分别计时数据加载和训练时间
- 使用 PyTorch Profiler 分析各阶段耗时
- 如果 `data_time > 0.3 × train_time`，需要优化
- 观察训练过程中 GPU 利用率是否稳定

#### Q5：**数据加载慢的优化策略有哪些？**
- 增加 num_workers（但不超过 CPU 核心数的 80%）
- 优化 `__getitem__()`（减少 I/O 和计算）
- 使用更快的存储（NVMe SSD、内存文件系统）
- 数据预加载（prefetch_factor）
- 使用高效数据格式（HDF5、Arrow）
- 启动时预加载数据到内存
- 优先级建议：
  - 首先检查存储速度（HDD → SSD 提升最大）
  - 然后优化 num_workers
  - 最后考虑数据格式和预加载

#### Q6：**如何保证训练的可复现性？**
- 设置所有随机源种子（Python、NumPy、PyTorch）
- 分布式下每个 rank 使用不同的种子（base_seed + rank）
- 使用 `torch.backends.cudnn.deterministic=True`
- 记录数据版本、代码版本、超参数
- 使用固定版本的依赖库

#### Q7：**LLM 训练数据预处理的关键步骤有哪些？**
- **文本清洗**：去 HTML、控制字符、乱码
- **文本规范化**：Unicode、全角半角、标点统一
- **分词**：BPE/Unigram，添加特殊 token
- **截断/填充**：处理序列长度
- **数据打包**：减少 padding 浪费

#### Q8：**为什么 LLM 训练需要文本规范化？**
- 减少"同义不同形"的情况（如全角/半角数字）
- 降低词表压力，提升词表利用率
- 确保训练/推理数据处理一致
- 提升模型泛化能力（学到语义而非格式）
- 例如："2024"和"2024"应该被视为相同的数字

#### Q9：**如何处理 LLM 训练中的长序列？**
- **截断**：保留重要部分（对话场景保留最近的）
- **拼接**：将多个短样本合并成长样本
- **动态批处理**：按长度分组，减少 padding
- **使用支持长上下文的模型架构**（如 RoPE 扩展）
- **梯度累积**：在显存受限时处理长序列
